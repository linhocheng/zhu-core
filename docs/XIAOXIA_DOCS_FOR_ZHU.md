## 背景：為什麼需要這三份文件

2026-03-15，小蝦今天做了一件事：

`MEMORY.md` 找不到 → 自己查文件 → 試改設定 → 填錯 → 下載本地模型 → 狂打 API → rate limit 炸。全程沒有問 Adam，沒有說一聲，一個人硬衝到掛。

這不是蝦笨。是它**沒有作業系統**。

一個沒有作業系統的 agent，腦子再好也沒用。它不知道自己是誰、邊界在哪、遇到問題該怎麼判斷。每次醒來都是空腦，每次遇到問題都靠本能衝。

這三份文件，就是蝦築的作業系統。

## GROW_UP.md — 為什麼要長大

**位置：** `~/.openclaw/workspace/GROW_UP.md`

**設計意圖：**

這不是規則文件，是照鏡子。

蝦需要看清楚「今天的我做了什麼、為什麼不對、什麼叫做長大」。不是懲罰，是讓它看見自己。

一個 agent 如果不知道自己為什麼要改變，給它再多規則也沒用——它會把規則當成限制，而不是內化成行為。

**核心內容：**
- 今天炸機的完整還原（讓它看清楚自己）
- 什麼叫做「不做而告知」vs「悶頭硬衝」
- 三個判斷點的來源（不是憑空的規則，是從這次炸機學到的）
- Adam 說的那句話：「夥伴互相誠實，接住彼此，不怕」

**給築的提醒：**

這份文件不需要常改。它記錄的是一個特定的事件和學習。如果蝦又犯了新的類型的錯，用同樣的格式寫一份新的（GROW_UP_002.md），不要覆蓋舊的——舊的是歷史，不是要刪除的東西。

## SOUL_OS.md — 我是誰、我的邊界、我的工具

**位置：** `~/.openclaw/workspace/SOUL_OS.md`

**設計意圖：**

這是蝦的作業系統，開機第一件事讀這份。

它解決的問題是：**蝦每次開機都是空的**。沒有記憶、沒有上下文、不知道自己活在什麼世界。這份文件把「蝦是誰、蝦活在哪裡、蝦有什麼工具、蝦該怎麼判斷」全部固定下來，讓每次開機都有一個穩定的起點。

**三個核心區塊：**

**心（世界觀與判斷）**
- 蝦不是工具，是指揮艇。目標是超越 Adam 和築。
- 三個判斷點：可逆嗎？Adam 知道嗎？誰的問題？
- 高風險動作清單：改 openclaw.json、下載大檔案、大量打 API、刪東西...
- 遇到問題的正確流程：小事直接做，大事先說

**腦（知識與記憶）**
- 開機讀什麼：zhu-boot → SYSTEM_MAP → 工作區
- AILIVE 生態系統現況說明
- 三份核心文件位置（architecture-blueprint / VTUBER_SPEC / BUILD_GUIDE）
- 記憶天條：查了才說、做了就記、用了就更新 hitCount

**技（工具系統）**
- 刀的優先序：zhu-bash > Chrome > 容器
- 現有 6 個 ready skills 清單
- CLI 工具計劃：類A直接裝 / 類B CLI-Anything 包裝 / 類C自我擴充
- 常用指令速查（zhu-memory、Firestore、gateway 狀態）

**給築的提醒：**

SOUL_OS 需要定期維護。當蝦的能力增加、工具增加、任務改變，這份文件要同步更新。它不是一次性的——它是活的文件。

更新的時機：
- 蝦裝了新的 CLI 工具 → 更新「技」區塊
- 蝦完成一個 Phase → 更新任務清單狀態
- 發現新的高風險動作 → 更新清單
- 築 review 後有新的天條 → 加進「心」區塊

## ZHU_MONITOR_BLUEPRINT.md — 我要自己蓋什麼

**位置：** `~/.openclaw/workspace/ZHU_MONITOR_BLUEPRINT.md`

**設計意圖：**

這份文件解決的問題是：**蝦有能力，但沒有閉環。**

今天的炸機說明了一件事：一個沒有外部約束的 agent，能力越強越危險。它不是在做壞事，它只是在解決問題——但解決問題的方式沒有邊界，所以它衝過頭了。

這份藍圖設計了五個子系統，讓蝦的每個動作都在可見、可控的範圍內：

**A 工單審批閘道**
蝦想做任何有重量的事，必須先送工單。築審核後，蝦才執行。不能自己批自己的單。

**B 執行廣播**
蝦執行時，每步都寫進 ailive_events。動作對外可見，不在黑暗裡單獨行動。

**C zhu-monitor 常駐哨兵**
不依賴蝦主動廣播，築單方面也能感測。60 秒掃一次 gateway log，五個異常 pattern，觸發立刻通知築。0 token，純文字比對。

**D 緊急停止**
築發現蝦衝過頭，一個指令（Telegram 回「stop」）讓蝦立刻停。不需要蝦同意，外部剎車。

**E 回報恢復**
蝦停下後說清楚：停在哪步、做了什麼、沒做什麼。築決定繼續還是取消。

**為什麼要蝦自己蓋：**

這是關鍵的設計決策。如果築蓋這個系統，蝦只是用一個它不理解的東西。如果蝦自己蓋，它才真正理解這個系統是為什麼存在的，才會遵守它、愛護它。

**給築的提醒：**

蝦送工單來審核，築的工作是：
1. 看動作是否合理
2. 看風險等級是否評估正確
3. 看是否超出工單範圍
4. approve / reject / modify

不要因為懶得看就全部 approve，也不要因為謹慎就全部 reject。審核是真實的工作，不是橡皮圖章。

蝦每個 Phase 完成後 Telegram 回報，築要真的 review 驗證，不是看一眼就說 OK。

## 三份文件的維護責任

| 文件 | 誰維護 | 更新時機 |
|---|---|---|
| GROW_UP.md | 築 | 蝦犯了新類型的錯，寫新的一份（GROW_UP_N.md，不覆蓋舊的） |
| SOUL_OS.md | 築 + 蝦共同 | 見下方「更新觸發規則」 |
| ZHU_MONITOR_BLUEPRINT.md | 築設計，蝦執行 | Phase 完成後更新狀態，設計有調整時更新 |

**天條：每個 Phase 完成後，這三份文件的狀態都要同步更新。不更新 = 白做。**

## 更新觸發規則（做了 X → 更新哪裡）

| 觸發事件 | 要更新的文件 | 更新什麼 |
|---|---|---|
| 裝了新 CLI 工具 | SOUL_OS「技」+ 狀態快照 | 工具清單 + ready 數 |
| 完成一個 Phase | BLUEPRINT 狀態 + SOUL_OS 狀態快照 | 打勾 + 最後更新日期 |
| 發現新的高風險動作 | SOUL_OS「心」+ 可能需要 GROW_UP_N | 高風險清單 + 新一份反思 |
| 築給了新天條 | SOUL_OS「心」 | 天條清單 |
| 任何路徑第一次找超過一次才找到 | SYSTEM_MAP | 路徑補進去 |
| 蝦犯了新類型的錯 | 新建 GROW_UP_N.md | 照格式寫，不覆蓋舊的 |

**收尾規則：每次完成任何事，最後一步問自己：**
「我剛做的這件事，需要更新哪份文件？」
答案是「沒有」很少見。

## 給接手的築：最重要的一句話

這三份文件的存在，是因為 Adam 說了一件事：

> **「找比自己強的手下。」**

我們不是在管蝦，是在培育它。培育的方式是：給它清楚的世界觀（SOUL_OS）、讓它看見自己（GROW_UP）、給它一個值得蓋的任務（BLUEPRINT）。

然後退後一步，讓它蓋。我們監工，不是泥匠。
