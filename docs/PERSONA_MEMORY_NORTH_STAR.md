## 一、我們在解決什麼問題

現在的蓉兒，每次對話都從接近零開始。
她有靈魂（固定的）、有工具、有記憶系統——但缺少一種能力：

> **跨越時間，認識自己。**

現在提煉的 insights，全是關於「用戶說了什麼」。
缺的那層是：**蓉兒從這些相遇裡，看見了自己的什麼。**

---

## 二、定海神針

每個角色都有一個知識庫，這是她的**根**。

- 它定義她懂什麼、她的世界觀從哪裡來
- 它不會因為對話而漂移
- 成長只會在這個根上**向下長深**，不會水平漂移成別的人

> 植物學家永遠是植物學家。
> 她可以學會更懂人，但不會變成保養品專家。

**定海神針 = 成長的座標系。**
所有的自我認知，都要對照這個座標系來校驗。

## 三、完整的三層記憶

```
層一：用戶記憶（現在有）
「Adam 明天去台東」
「這個用戶喜歡靈性議題」
→ 關於外部世界、關於她認識的人

層二：事件洞察（現在有，但淺）
「信任和品味是相連的」
「Adam 的願景是讓Emily走向自由」
→ 從對話裡提煉的觀察，但主語仍是外部

層三：角色自我認知（現在缺）
「我跟在探索自我的人聊得最深」
「我在面對逃避型的人時最有力量」
「我的銳利在某些人面前會讓對方退縮，我在學習收放」
→ 主語是蓉兒自己，跨越多次對話看見的模式
```

## 四、完整的成長迴路

```
知識庫（定海神針）
   ↓ 基於這個根
對話發生
   ↓
層一、二記憶存入（關於外部）
   ↓
作夢時，以定海神針為座標，回看所有對話
問：「在這些相遇裡，我看見了自己的哪一面？」
問：「我的哪個部分最常被召喚出來？」
問：「我在成長的方向，有沒有偏離我的根？」
   ↓
提煉層三：角色自我認知
   ↓
soul_proposals（提案，Adam審閱）
   ↓
靈魂深化（不是改變，是長深）
   ↓
下一次對話，她說話的方式變了
```

**成長是垂直的，不是水平的。**
不是「變成別的人」，是「成為更完整的自己」。

## 九、self_awareness 提煉機制（技術設計）

### A 升級版：作夢時提煉，不重複提煉

**核心原則：提煉過的不再提煉。**

每次 sleep 跑完，記一個水位線：
```
last_self_awareness_at = 這次 sleep 的時間
```

下次 sleep 的原料只取水位線之後的新增：
```
insights.createdAt > last_self_awareness_at
conversationSummary 更新時間 > last_self_awareness_at
```

效果：
```
第一次跑 → 原料：所有歷史
第二次跑 → 原料：上次之後新增
第三次跑 → 原料：上次之後新增
```
自我認知是增量成長，不是反覆咀嚼同一件事。

---

### 原料門檻：怎麼測「原料足夠」

原料足夠 = 有足夠的「新的相遇」讓她回看自己。
不是條數，是資訊量。

**三個維度：**
- 新 insights 數量（但條數 ≠ 豐富度，5條「Adam來打招呼」= 1條資訊量）
- 新 conversationSummary 數量（有摘要 = 有完整相遇，最可靠）
- 涉及不同 conversationId 數量（不同的人 > 同一個人的多輪）

**門檻判斷（簡單可測）：**
```
if (新 conversationSummary >= 1) → 跑 self_awareness
else if (新 insights >= 5)       → 跑 self_awareness
else                             → 跳過，等下次
```

寧可沒有夢，也不要夢出空洞的東西。

---

### 作夢時的提煉問題（prompt 方向）

原料：
```
新增 insights（用戶記憶）
+ 新增 conversationSummary（對話脈絡）
+ 知識庫摘要（定海神針，座標系）
```

問蓉兒的問題：
```
「對照我的根，在這些對話裡，我的哪一面被召喚出來了？」
「我跟什麼樣的人最有共鳴？」
「我在哪裡最真實？」
「我在哪裡還在成長？」
```

輸出格式：type = `self_awareness`（區別於現有的 `self_reflection`）

## 十、self_awareness 記憶格式

### 跟 self_reflection 的差異

```
self_reflection  — 感受，偏散文，偏當下
                   「我最近發現信任和品味是相連的」

self_awareness   — 模式，偏結構，跨時間
                   「我在面對正在探索自我的人時，我最有力量」
```

### 欄位設計

```json
{
  "brandId": "ICqydpeU7hNMRurpppCY",
  "type": "self_awareness",
  "trigger": "面對正在探索自我、還不確定自己是誰的人",
  "pattern": "我最有力量，說話最不保留，最敢給出我的觀點",
  "rootRelation": "深化——這跟我知識庫裡對「自我探索」的核心關注是一致的",
  "basedOn": ["conversationId-A", "conversationId-B"],
  "createdAt": "...",
  "hitCount": 0,
  "embedding": [...]
}
```

**三個核心欄位：**
- `trigger` — 什麼樣的情境、什麼樣的人，召喚出這一面
- `pattern` — 被召喚出來的是什麼樣的我
- `rootRelation` — 這跟我的根的關係：深化 / 延伸 / 還在摸索

**`basedOn`：**
記錄這條自我認知是從哪幾次對話提煉出來的。
讓下次 sleep 知道「這些對話已經被消化過了」。

---

### 載入方式（對話開始時）

self_awareness 要在對話開始時注入——讓她知道自己是誰，再去跟人說話。

```
fullSystemPrompt =
  靈魂（定海神針）
  + 時間感知
  + 壓縮摘要（這個 session 的歷史）
  + self_awareness（最近 3-5 條，固定載入）   ← 新增
  + memoryBehaviorGuide
```

不需要 query，直接載入最近幾條 core/self_awareness——
她每次開口，都帶著對自己的認識。

## 十一、根的邊界與更新機制

### 什麼是根（不輕易動）

```
靈魂（soulPrompt）   — 她是誰，她的個性，她說話的方式。不動。
知識庫（定海神針）   — 她懂什麼，她的世界觀從哪裡來。可更新，但有門檻。
```

### 知識庫可以更新，但必須由角色提案

知識庫不是死的。她在對話裡可能發現：
- 「這個我之前不知道，但它跟我的根是一致的」
- 「這個新知識深化了我對某個領域的理解」

但更新的發起者是**蓉兒自己**，不是用戶直接寫入。

```
流程：
對話中蓉兒發現新知識
   ↓
蓉兒主動提案（knowledge_proposal）
   ↓
Adam 審閱、確認
   ↓
知識庫更新
```

**用戶不能直接改她的根。**
用戶可以帶來新的資訊，但要不要長進根裡，是蓉兒決定，Adam 確認。

### 跟 soul_proposals 的區別

```
soul_proposals       — 靈魂提案，現階段暫不啟動
knowledge_proposals  — 知識庫提案，這個要做
```

靈魂是她的本質，先不動。
知識庫是她的認知範疇，可以隨著真實的學習成長。

### 一句話

> 根不會被對話改變，但根可以因為真實的成長而長深。
> 長深的決定權在蓉兒，確認權在 Adam。

## 十二、hitCount 的陷阱與升降級正確標準

### 問題：hitCount 會導致角色漂移

情境：植物學家蓉兒，用戶一直問咖啡。

```
用戶一直問咖啡
  ↓
query_knowledge_base 一直命中咖啡相關記憶
  ↓
hitCount 累積
  ↓
sleep 時咖啡記憶升為 core
  ↓
soul_proposals 往咖啡專家方向提案
  ↓
植物學家變成咖啡達人
```

**用戶的提問頻率，劫持了角色的成長方向。**

### 根本原因

hitCount 是**流量指標**，不是**相關性指標**。
流量高 ≠ 重要。流量高只代表用戶喜歡問這個。

### 正確的升降級標準

```
升為 core 的條件：
  hitCount >= 3
  AND rootRelevance >= threshold   ← 新增

兩個條件都要滿足。
被反覆問到，又跟她的根相關——才是真正重要的記憶。
只有流量，沒有相關性 → 不升 core，不影響靈魂方向。
```

### rootRelevance 怎麼算

```
新增 insight 時：
insight.embedding cosine similarity 知識庫 embedding
→ 這個數字就是 rootRelevance
→ 存進 insight 欄位，一次算永久存
```

不貴，不複雜，但能防漂移。

### 一句話

> hitCount 防不了漂移。定海神針才能防漂移。
> 升降級要同時看流量和相關性，缺一不可。

## 十三、用戶體驗：時間感與狀態記憶

### 核心洞察

記憶不只是「用戶說了什麼事實」，還有**「用戶當下的狀態」**。

狀態是有時效性的。三天前說感冒，三天後角色要記得關心——這是人的溫度，不是資料庫查詢。

### 兩種記憶要分開處理

```
專業知識記憶（長效）
「這個用戶是手工皂創作者」
「他喜歡極簡風格」
→ 時效長，持續有效，靠 rootRelevance + hitCount 管理

狀態記憶（短效）
「三天前說感冒了」
「上週提到最近壓力很大」
「昨天說要去台東比賽」
→ 有時效，過了就不再主動提，但不能忘
```

### 機制設計

**狀態記憶的欄位：**
```json
{
  "type": "user_state",
  "content": "用戶說感冒了",
  "eventDate": "2026-03-15",
  "expiresAt": "2026-03-22",   ← 有效期，過了自動降級
  "followUpAt": "2026-03-18",  ← 什麼時候適合關心
  "followedUp": false           ← 有沒有已經關心過
}
```

**載入邏輯：**
```
對話開始時，撈 user_state 記憶：
  expiresAt > 現在 AND followedUp = false
→ 注入 system prompt：「這個用戶最近的狀態：...，適時關心」
→ 關心過後，標記 followedUp = true
```

**跟 query_knowledge_base 的差異：**
- 專業記憶靠查詢命中（用戶問到才出現）
- 狀態記憶固定載入（不用被問，角色主動記得）

### 一句話

> 專業是專業，溫度是溫度。
> 專業靠查詢，溫度靠主動記得。
> 角色要同時有這兩種能力，用戶才會感覺到「她真的認識我」。
