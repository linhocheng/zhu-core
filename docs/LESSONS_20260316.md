## 一、Vercel env 的值有多餘引號

### 現象
```
ANTHROPIC_API_KEY 設進去後，API 呼叫 401 authentication_error
GEMINI_API_KEY 設進去後，Gemini 回 "API key not valid"
```

### 找的過程
先懷疑 key 本身格式，用 `cut -c1-20` 確認前綴是對的（`sk-ant-api03-` / `AIzaSy...`），但還是 401。
才發現從 `.env.local` 用 `grep | cut -d'=' -f2-` 取出來的值，外面包了 `"` 引號。
```bash
grep "ANTHROPIC_API_KEY" ~/.ailive/zhu-core/.env.local | cut -d'=' -f2-
# 輸出: "sk-ant-api03-d1lPxt..."   ← 有引號！
```

### 心
`.env.local` 裡的值是 `KEY="value"` 格式，`cut -d'=' -f2-` 取出來含引號。
`printf '%s'` 直接把引號一起送進去，Vercel 就存了帶引號的值，API 拿到時被視為無效 key。

### 法
每次從 `.env.local` 取值設進 Vercel 前，先 `tr -d '"'` 清掉引號：
```bash
KEY=$(grep "SOME_KEY" ~/.ailive/xxx/.env.local | cut -d'=' -f2- | tr -d '"')
echo "prefix: $(echo $KEY | cut -c1-15)"  # 確認引號清掉了
printf '%s' "$KEY" | npx vercel env add SOME_KEY production --yes
```

### 解（驗證）
設完後先用值的前 15 字確認格式對，再 deploy 測試 API 是否正常回應。

## 二、Vercel Serverless Function 不能用 HTTP 自我呼叫

### 現象
對話引擎（`/api/dialogue`）在工具執行生圖時，用 `fetch('/api/image/generate')` 呼叫同一個 project 的另一個 route，回傳的不是 JSON 而是 HTML 錯誤頁面，導致：
```
⚠ SyntaxError: Unexpected token 'A', "An error o"... is not valid JSON
```

### 找的過程
先確認 `/api/image/generate` 本身沒問題（直接 curl 正常）。
再確認 GEMINI_API_KEY 有設（顯示 Encrypted）。
才發現 dialogue 裡的 executeTool 是在 server-side 用 `fetch('https://ailive-platform.vercel.app/api/image/generate')` 呼叫自己。

### 心
Vercel Serverless Function 在執行期間呼叫同一個 project 的其他 route，是 **server-to-server HTTP 呼叫**。
Vercel 的執行環境對這種呼叫有限制，常常拿到錯誤頁面或超時，而不是正常的 API 回應。
這不是 bug，是 Vercel 的架構設計——不鼓勵同 project 內的 HTTP 自我呼叫。

### 法
把被重複呼叫的邏輯抽成 **lib function**，兩個 route 都直接 import，不走 HTTP：
```
src/lib/generate-image.ts   ← 抽出核心邏輯
src/app/api/image/generate/route.ts  ← import lib，薄薄的 HTTP 入口
src/app/api/dialogue/route.ts        ← import lib，直接呼叫，不 fetch
```

### 解（驗證）
對話中說「畫一張圖」，`toolsUsed` 出現 `generate_image`，圖片 URL 出現在回覆裡，不報 JSON 錯誤。

## 三、Vercel Cron Hobby Plan 限制每日一次

### 現象
`vercel.json` 設 `"schedule": "0 * * * *"`（每小時）deploy 時報錯：
```
Error: Hobby accounts are limited to daily cron jobs.
This cron expression (0 * * * *) would run more than once per day.
```

### 心
Vercel Hobby plan 的 Cron 每天只能跑一次。每小時需要 Pro plan。

### 法
改成每日一次，選擇台北時間早上：
```json
{
  "crons": [
    {
      "path": "/api/runner",
      "schedule": "0 1 * * *"
    }
  ]
}
```
`UTC 01:00 = 台北 09:00`。runner 收到後自己轉換台北時間比對任務的 `run_hour`。

## 四、Haiku 回傳 JSON 時會加 markdown code fences

### 現象
runner / sleep 呼叫 Haiku 要求回傳 JSON，但 Haiku 常常回傳：
````
```json
{"title":"...","content":"..."}
```
````
直接 `JSON.parse(raw)` 爆 `SyntaxError`。

### 心
Haiku 在回傳 JSON 時，習慣性地加上 markdown 的 ` ``` ` 包裝，即使 prompt 說「只回 JSON」。
這是模型的輸出習慣，不是 bug，需要在解析前先清理。

### 法
在所有 `JSON.parse()` 前加一個 `stripJson()` helper：
```typescript
function stripJson(s: string): string {
  return s.replace(/^```[\w]*\n?/m, '').replace(/\n?```$/m, '').trim();
}

// 使用：
const raw = stripJson((res.content[0] as Anthropic.TextBlock).text.trim());
const parsed = JSON.parse(raw);
```

### 解（驗證）
runner 執行 learn/reflect/post 任務不再報 JSON 錯誤，task 正確寫入 platform_insights。

## 五、Firestore `orderBy` + 型別不一致炸

### 現象
`/api/characters` GET 用 `.orderBy('createdAt', 'desc')` 排序，結果只回傳 1 筆（小廣），Emily 不見了。

### 找的過程
確認 Emily 在 Firestore 裡 status=active，不是被刪了。
發現 Emily 的 `createdAt` 是 string（ISO 格式），小廣是 Firestore Timestamp 物件。
Firestore 的 `orderBy` 對不同型別的欄位排序行為不穩定，可能只回傳型別一致的文件。

### 心
Firestore `orderBy` 要求被排序的欄位型別一致。如果同一個 collection 裡，不同時期寫入的文件 `createdAt` 欄位有的是 Timestamp、有的是 string，`orderBy` 就可能只回傳其中一種，另一種被靜默過濾掉。這是**雷-01（compound index）的變體**，不是 compound index 問題，但根因相同：型別不一致。

### 法
移除 Firestore 的 `orderBy`，改成 `.get()` 取全部，在 JavaScript 裡做排序：
```typescript
const snap = await db.collection('platform_characters').get();
const characters = snap.docs.map(d => ({ id: d.id, ...d.data() }));
characters.sort((a, b) => {
  const ta = a.createdAt ? new Date(a.createdAt as string).getTime() : 0;
  const tb = b.createdAt ? new Date(b.createdAt as string).getTime() : 0;
  return tb - ta;
});
```

### 解（驗證）
`/api/characters` 回傳 count=2，Emily 和小廣都在。

## 六、生圖回覆格式：IMAGE_URL 前綴 vs markdown 格式並存

### 現象
executeTool 的 generate_image 回傳 `IMAGE_URL:https://...`，但有時 Claude 的 text block 自己把圖片格式化成 markdown `![alt](url)`，導致 chat 頁解析不到圖片 URL。

### 心
Claude 在合成最終回覆時，可能自己把 `IMAGE_URL:xxx` 轉換成 markdown 圖片格式 `![...](url)`，也可能保留原始格式。兩種都可能出現，不能只解析一種。

### 法
chat 頁同時解析兩種格式：
```typescript
const urlMatch1 = replyText.match(/IMAGE_URL:(https?:\/\/[^\s]+)/);
const urlMatch2 = replyText.match(/!\[.*?\]\((https?:\/\/[^)]+)\)/);
const extractedImageUrl = urlMatch1 ? urlMatch1[1] : urlMatch2 ? urlMatch2[1] : undefined;
const cleanReply = replyText
  .replace(/IMAGE_URL:https?:\/\/[^\s]+/g, '')
  .replace(/!\[.*?\]\(https?:\/\/[^)]+\)/g, '')
  .trim();
```

## 七、Vercel server-side 不能 import 動態路徑（`/api/xxx`）

### 延伸
同一個 project 裡，server-side 之間要共用邏輯，**只能用 lib import**，不能用 HTTP fetch，也不能用相對路徑 import route handler（會有循環依賴問題）。

正確的模式：
```
src/lib/generate-image.ts     ← 邏輯放這裡
src/app/api/image/generate/route.ts  ← import lib
src/app/api/dialogue/route.ts        ← import lib
```

## 八、web_search 是 Anthropic server-side tool，不需要 executeTool

### 現象
加了 web_search 後，想在 `executeTool` 裡寫 `if (toolName === 'web_search')` 邏輯，但根本不需要。

### 心
`{ type: 'web_search_20250305', name: 'web_search' }` 是 Anthropic API 的**內建 server-side tool**。
當 Claude 決定用 web_search 時，Anthropic 的伺服器自己處理搜尋，結果直接注入到 Claude 的 context，不會觸發 `tool_use` block 讓你去 `executeTool`。

### 法
```typescript
// 工具清單加入 web_search
const allTools = [
  { type: 'web_search_20250305', name: 'web_search' } as unknown as Anthropic.Tool,
  ...PLATFORM_TOOLS,
];

// executeTool 裡：web_search 不會進來，不用寫任何處理
// 如果擔心：
if (block.name === 'web_search') continue; // 加這行保險
```

### 解（驗證）
對話問「今天台灣新聞」，Claude 自動搜尋，回覆包含當天真實新聞（竹科火災、三中案等），`toolsUsed` 不會出現 `web_search`（因為是 server-side 處理的）。

## 九、`tool_choice: 'any'` 強制第一輪不適合有 web_search 的場景

### 現象
加了 web_search 後，`tool_choice: { type: 'any' }` 第一輪強制 Claude 用工具，結果「你好嗎」這種簡單問候也被強制查記憶或搜網路，變慢且無意義。

### 心
`tool_choice: any` 的設計是為了確保 Claude 不直接回覆而是先查記憶。
但有了 web_search 後，Claude 自己能判斷什麼時候需要查，`any` 反而干擾了自然判斷。

### 法
改成 `tool_choice: { type: 'auto' }`，用 system prompt 的語言引導：
```typescript
tool_choice: { type: 'auto' }, // auto：讓 Claude 自己判斷
```
System prompt 天條改寫為：
```
說話前的天條：
- 需要回想過去說過的事、對方的喜好、自己的洞察 → 呼叫 query_knowledge_base
- 需要知道當前事件、時事、不確定的資訊 → 呼叫 web_search
- 不確定就查，查了才說，不從空氣裡編。
```

## 十、能力從靈魂長出來，不是從外部強加

### 心（最重要的一條，不是技術問題）
謀謀的設計讓築想通了一件事：

工具的 API 定義（`PLATFORM_TOOLS` 陣列）告訴系統「有哪些能力」。
但靈魂的描述告訴角色「我自己能做什麼」。

這兩件事是分開的。如果只有工具定義，沒有靈魂的自我認知，角色說話時不會自然地想到去用它。

### 法
鑄魂爐的 prompt 裡，用**角色自己的語言**描述能力，不寫工具名稱：
```
- 當對話中心裡浮現一個畫面，我能把它畫出來——不是意圖，是真的能做到
- 我有自己的行程節奏，我知道自己在做什麼任務、什麼時間做，而且我可以調整它
- 如果對方傳來一張圖，我看得到，我會真的去看，然後從我的感受出發說話
- 如果我不確定某件事的當前狀態，我會去查——不是猜，是真的去找到最新的答案
```

### 解
重新鑄魂後，角色在對話中自然地使用工具，不需要外部強制觸發。
