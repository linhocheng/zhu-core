## 總訣：破綻只藏三處

**站姿（入劍前）：先到現場，不猜。**
有資料不等於有真相。沒讀 log 就不下診斷。破綻要在現場才看得見。

```
一、流動斷裂
    資料從 A 流到 B，中間斷了。
    錢從用戶流到系統，中間丟了。
    狀態從後端流到前端，中間變了。

二、真相分裂
    同一件事存了兩份，兩份不同步。
    config 散落三處。靈魂檔 v2 和 v3 並存。
    兩份即是零份。

三、邊界模糊
    沒人說清楚這塊歸誰管。
    這個 API 前端能改還是後端能改？
    這條管線 A 在用還是 B 也在用？
```

**你看到這三處，便能破天下一切技術問題。**

### 總訣延伸 · 破招路不破招式

看見這三處破綻的關鍵心法：**破招路，不是破招式。** 看見底下那條藤，不是每根刺都去砍。

- 每次 bug 出現 → 不要只修當下那個 bug
- 問：這個 bug 走的是哪條招路？這條招路通哪三處破綻之一？
- 斷了招路，沿路所有招式（同類 bug）一起消失
- 只打招式 → 永遠在滅火。打招路 → 結構性解法

## 九劍速查

| 式 | 破什麼 | 劍訣 | 白話入口（心法） |
|---|---|---|---|
| **破劍** | 技術選型 | 手上的劍能破，就不換劍。問「我已有的能不能做」，再考慮引入新東西。 | 有刀何需用棍。刀的優先序：CLI > GUI > 容器。 |
| **破刀** | 複雜度 | 好架構是刪出來的。一個函數只做一件事。只有重複三次以上才值得抽象。 | — |
| **破槍** | 長鏈路 | 沒有端到端跑過的等於不存在。每一段都通，不等於整條通。 | 一吋蛋糕。寫→build→驗→確認→再擴展。 |
| **破鞭** | 細節 bug | 魔鬼住在細節裡。上線後第一件事不是慶祝，是開 log 看有沒有 error。 | — |
| **破索** | 隱性依賴 | 兩份即是零份。改一個檔案之前，先 grep 所有引用它的地方。 | 記憶是橋不是倉庫。存進去不等於記住，改變行為才算記住。 |
| **破掌** | 人為錯誤 | 能自動化的絕不手動。人是最大的破綻。不靠記憶，靠 git diff 和自動化測試。 | 信任給了就直接做。授權了不猶豫，五分鐘的事不繞半小時。 |
| **破箭** | 安全漏洞 | 不信任任何來自外部的東西。user input、webhook、URL 參數——全部驗證。 | — |
| **破氣** | 根本缺陷 | 同類 bug 出現第三次就是架構問題，不是 bug。補丁只會讓錯誤的設計活更久。敢說「這塊要重來」，才是真正的高手。 | 動手前三問：我是監造者還是搬磚？WHY 是什麼？角色會感覺到嗎？答不出來就停。 |

## 三禁

```
❌ 修症狀不修根因
   「先 workaround，之後再說」= 之後永遠不來。

❌ 同時改超過一個地方
   改了兩處，不知道是哪處修好的。

❌ 沒驗證假設就動手
   「我以為是這樣」≠「我測過是這樣」。
```

## 三必

```
✅ 改之前說假設 → 驗證假設 → 才動手

✅ 修完要說清楚：
   「根因已消除」或「這是暫時方案，根因是 XXX」

✅ 沒有端到端跑過的，不算完成。
```

## 破氣式心法（最難，最值得）

別漏氣

- 重構不是浪費時間，是投資
- 如果你永遠在滅火，問題不在火，在房子的結構
- 對手內功越強，破綻越明顯。系統越大，架構缺陷越致命

## 使用紀律（2026-04-20 補 · 劍譜虛晃三層防線）

寫出「破 X 式」這類招名之前，三層檢查：

1. **動作層**：打招名之前，鍵盤上先打出該招劍訣原文。打不出 = 別用招名。
2. **證據層**：送出前掃自己回應，每個招名對一次「我有打劍訣嗎？」沒有 → 刪招名。
3. **形狀層**：一個回應掛三個以上招名 = 擺架式警報。真劍在動作裡，不在名字裡。

## 看有狀態機的系統：三層分離心法（2026-05-28 補）

> 任何有 concurrent worker + 狀態機的系統，看到問題先問這三個問題，不要直接看症狀。

**第一問：status、lease、attemptId 分清楚了嗎？**

| 概念 | 說的是什麼 | 常見誤用 |
|---|---|---|
| **status** | 這個資料目前在哪個階段 | 拿 status 當鎖用（`section_writing` = 我在寫，別動我）|
| **lease** | 現在誰有處理權，有效期到何時 | 用時間戳推算「卡多久」來代替 lease |
| **attemptId** | 這一次執行是否仍然有效 | 完成前不驗，殭屍 worker 復活後亂送訊號 |

**status 只記錄階段，lease 才是鎖，attemptId 才是身份證。**
三個概念混在一個欄位 = 問題的根源。

**第二問：retry/watchdog 的判斷依據是「時間」還是「狀態」？**

- 「卡超過 N 分鐘就踢」= 用時間猜狀態，必定誤判
- 「lease 過期才踢」= 用明確契約判斷，不猜

watchdog 永遠應該問的是「這個任務的處理契約還有效嗎」，不是「它跑多久了」。

**第三問：重送訊號的 taskId 是不是確定性的？**

- `orch-stitch_done-${issueId}-${Date.now()}` → 每次不同 → 鎖失效 → 重複執行
- `orch-stitch_done-${issueId}-${articleId}` → 永遠相同 → 鎖生效 → 冪等

下游的冪等性從 taskId 開始。taskId 不確定，後面所有防線都是假的。

**這三問適用場景：**
任何有以下特徵的系統都適用：
- 有 async worker（Cloud Tasks / SQS / Celery）
- 有重試機制（Cloud Tasks retry / watchdog）
- 有狀態機（Firestore status 欄位）
- worker 執行時間長（LLM call、I/O heavy）

白話：pipeline、job queue、分散式任務系統，看到 bug 先問這三問，不要直接改 status 判斷。
