# LESSONS 2026-05-23b · ANEWS 狀態機測試驗收

## L1：approve endpoint 忘記帶 taskId，orchestrate 靜默 400
- 現象：GATE 2 approve 後 issue 永遠停在 awaiting_review，不推進
- 根因：`approve/route.ts` 呼叫 orchestrate 沒帶 `taskId`，`missing_params` 400 → 靜默失敗（endpoint 自己回 500 但測試沒抓到）
- 下次：每一個呼叫 orchestrate 的地方，寫完立刻確認 `taskId` 在 body 裡；同型 endpoints (`approve-blueprint` / `approve-section-review`) 對比一下有沒有少
- 對應 feedback：靜默失敗用「缺席的 log」診斷

## L2：Cloud Tasks 本機完全不觸發——callbackOrchestrator 也包含在內
- 現象：image workers 跑完但 `images_all_done` 不發；export workers 跑完但 `export_done` 不發
- 根因：`callbackOrchestrator` 內部呼叫 `enqueueTask` → GCP Cloud Tasks queue → 打不到 localhost
- 下次：本機測試凡是 `callbackOrchestrator` 或 `enqueueTask` 的地方，全部在測試腳本手動補一次對應的 orch 呼叫；測試腳本頂部加一條規則：「orchestrate→worker 方向走 CT（本機無效）；worker→orchestrate 方向走直接 HTTP（本機有效）」
- 對應 feedback：能本機重現就不要等遠端 cycle

## L3：28 pass 0 fail 代表什麼——和不代表什麼
- 代表：狀態機轉場邏輯正確，四個人工閘門（source_thin / blueprint_review / section_review / content_review）都能阻斷和繼續，idempotency 有效
- 不代表：LLM workers 真實輸出品質、Cloud Tasks 排隊延遲、Vercel 300s 壓測、多期並存沒踩鎖
- 下次：28 pass 是 Round 1+2 基線，Round 3（5 articles × 26 sections × 28 images）才是全系統壓測
