# LESSONS 2026-06-13 · ailivex 即時語音 v5/v6/v8 發言權能力

## L1：generate_reply-in-handler + StopResponse 會卡死框架回話迴圈
- 現象：v8「被點名抓麥克風」上線後，張立抓過一次麥克風就對所有後續 turn 都不回 = Adam 看到的「卡住」。
- 根因：在 LiveKit `on_user_turn_completed` 裡**手動 `self.session.generate_reply(allow_interruptions=False)` + 又 raise StopResponse**。這個組合把框架的回話排程迴圈卡死，之後不再生成任何回話。
- 下次：在 turn-completed handler 裡只該做「攔截/放行」（raise StopResponse 或 return），**不要手動 generate_reply**。要改回話參數得從 session 層（中斷門檻 / turn_handling），不在 handler 動。
- **對應 feedback**：`feedback_dryrun_before_test`（探索性測試先驗）+ `feedback_mvp_include_input_entry`（沒端到端跑過不算完成）。最痛的是——**動手前我自己在計畫裡白紙黑字標了這個技術風險（「generate_reply-in-handler timing-sensitive，要先驗」），結果沒在本機驗就推上去，照樣踩**。標了風險 ≠ 驗證了風險。

## L2：中文 regex 點名偵測，一字多義高頻口語必誤判
- 現象：v5/v6 讓位偵測把「晚一**点**如果」「晚一**点**再跟你聊」誤判成「點名第三方」，導致角色被 StopResponse 閉嘴 = 第一次「卡住」。
- 根因：`点/點` 在意圖詞集裡（原意「點名」），但口語「晚一点/有点/差点」超高頻全命中。
- 修法：意圖詞 + 名字 + **說話動詞**三件齊全才算讓位（修法 B）。排除 `点/找/叫` 高頻字，給讓位偵測獨立保守集 `_REDIRECT_PRE`，不動多角色用的 `_ADDR_PRE`。
- 鐵則：**寧可漏判（角色多回一句）也不要誤判（角色閉嘴）**。閉嘴體感最差。
- 對應 feedback：`feedback_defend_at_convergence_point`（把判斷收斂在乾淨的咽喉）。

## L3：天條張力——判斷題本可用 LLM，但即時語音延遲不允許
- 「這句是不是叫第三方說話」本質偏判斷題，照天條可用 LLM。但放 LLM 在每句回應路徑 = 每輪加延遲，即時語音很傷。
- 決策：先用結構化 regex 收斂（句型夠結構化抓得住），漏判不致命。**怎麼修都按不住、誤判層出不窮時**才是升級「LLM 輕判斷」的訊號。現在還不到。
- 天條不是「判斷題一律丟 LLM」，是「在成本/延遲容許下，判斷題才用 LLM」。

## L4：gcloud config project 會被別的 session 切走
- 現象：撈 v6 log 撈不到、`services describe` 報 Cannot find service。
- 根因：`gcloud config get-value project` 變成 `udnnews`（別的 session/工作切的），不是 `ailivex-2026`。
- 下次：查 ailivex 一律顯式帶 `--project=ailivex-2026`，**不要動全域 config**（會影響別的工作）。

## L5：AEC 回音真機驗到——角色自己的 TTS 污染逐字稿
- 現象：張立/Tracy 自己 TTS 講的話被麥克風（手機免持）收回，diarization 標成「另一個人 #2/#3」，60 秒後又吐回 STT。
- 影響：污染逐字稿 + 判斷腦拿回音當輸入評估 + 可打斷回話會被自己回音掐斷。
- 性質：裝置層 AEC 問題，非 agent code 能根治。「被點名不可打斷」原想順帶解決（講完整段回音動不了），但實作踩了 L1 的雷。
- 待辦：安全版用 session 中斷門檻（min_words/min_duration 調高，短回音打不斷），不碰回話迴圈。
