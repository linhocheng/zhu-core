## 1. Skills 防重複：tool loop 會讓 save_skill 被呼叫兩次

### 現象
對話中說「把這個技巧記下來」，Firestore 裡出現兩條一模一樣的技巧。

### 心
Claude tool use loop 的機制：第一輪 tool_use 結束後，Claude 拿到 tool_result，有時會再發起第二輪確認——如果 handler 每次都寫入，就會重複。不是 bug，是 loop 的正常行為，要在 handler 層防禦。

### 法
在 `save_skill` handler 寫入前，先查同角色同名是否已存在：
```typescript
const existingSnap = await db.collection('platform_skills')
  .where('characterId', '==', characterId)
  .where('name', '==', name)
  .limit(1)
  .get();
if (!existingSnap.empty) {
  return `技巧「${name}」已經記下來了，不用重複建立。`;
}
```

### 解
第二次呼叫 `save_skill` 被攔截，回傳「已記下來」，Firestore 不重複寫入。

## 2. Skills 系統設計模式：「記憶會模糊，技巧定型化」

### 現象
用戶說「以後這個流程就走這樣」，但存在記憶裡的 insight 會衰退、壓縮、消失。

### 心
記憶（insights）是流動的，適合「我學到了什麼」。
技巧（skills）是定型的，適合「遇到這種情況，我永遠這樣做」。
兩者本質不同，不能混用同一套系統。

Skills 的電流：
```
用戶說「把這個技巧記下來」
→ save_skill tool 觸發
→ 寫入 platform_skills（enabled: true）
→ 每次對話開始讀取啟用的 skills
→ 組成「我的定型技巧」區塊注入 system prompt
→ 角色永遠記得這個流程，不因記憶壓縮而消失
```

### 法
照 posts 的模式平行建構 skills：
- `/api/skills` CRUD（GET/POST/PATCH/DELETE）
- `save_skill` tool 加進 `PLATFORM_TOOLS`
- system prompt 組裝時讀 `platform_skills where enabled==true`
- `/dashboard/[id]/skills` 後台管理頁

### 解
阿肯、Doory 兩個角色都驗收通過：
- 對話說「記下來」→ tool 觸發 → 存入
- 下次新對話 → 技巧自動注入 → 角色照流程走
- 後台可開關、編輯、刪除、手動新增

## 3. Firestore compound where 的限制再確認

### 現象
`where('characterId').where('name')` 兩個 where 條件，runtime 沒有爆，正常運作。

### 心
兩個 where 在同一個欄位類型（都是 string equality）不需要 composite index，只有 where + orderBy 不同欄位才需要。這次沒踩雷，但要記住邊界。

### 法
- `where A == x` + `where B == y`：通常 OK，string equality
- `where A == x` + `orderBy B`：需要 composite index，改成 JS filter
- 遇到 index 錯誤：先移除第二個 where，JS 端 filter

## 4. ailive-platform 和 dreamfactory 花費追蹤的共同漏洞

### 現象
兩個平台的 `costMetrics.totalCostUSD` 都偏低——生圖費用完全沒有計入。

### 心
生圖是獨立的 lib（`generate-image.ts`），不走 dialogue 的 trackCost 路徑。兩個平台的開發者（就是築）在蓋生圖時只管「圖有沒有出來」，沒有回頭接費用追蹤。費用追蹤只有在 API route 層有意識地加，lib 層沒有自動追蹤。

### 法
**ailive-platform（Gemini 生圖，按 token）：**
1. `gemini-imagen.ts` 的 `GeminiImageResult` 加 `inputTokens / outputTokens`
2. return 時從 `data.usageMetadata` 抓 `promptTokenCount / candidatesTokenCount`
3. `generate-image.ts` 三條路徑各加 `trackCost(characterId, result.model, result.inputTokens, result.outputTokens)`
4. `cost-tracker.ts` 補 `gemini-2.5-flash-image` 定價（input $0.075 / output $0.30）

**dreamfactory（Grok 生圖，按張）：**
1. `cost-tracker.ts` 加 `PER_IMAGE_PRICING` + `calcImageCostUSD` + `trackImageCost`
2. `grok-imagine-image` 定價 $0.07/張
3. `generate-image.ts` 三條路徑各加 `trackImageCost(characterId, result.model)`

### 解
兩個平台 tsc 乾淨，deploy 後生圖費用開始正確計入。歷史數字無法補回。

## 5. knowledge-parse 圖片描述費用也沒追蹤

### 現象
上傳 docx/pdf 含圖片時，Haiku 會幫每張圖寫描述，但沒有 trackCost。

### 心
`describeImage` 是 helper function，呼叫完直接 return string，呼叫者沒有機會拿到 usage。

### 法
改 `describeImage` 回傳 `{ description, inputTokens, outputTokens }`，呼叫後立刻 `trackCost`：
```typescript
const descResult = await describeImage(client, base64, ct);
trackCost(characterId, 'claude-haiku-4-5-20251001', descResult.inputTokens, descResult.outputTokens).catch(() => {});
```

### 解
兩個平台都補齊，知識庫圖片描述費用開始計入。

## 6. GCP Firebase Function operation lock

### 現象
`firebase deploy --only functions:ailiveScheduler` 連續回傳：
`An operation on function ailiveScheduler in region asia-east1 is already in progress. Please try again later.`

### 心
GCP 對 function update 是排隊處理。前一次 deploy 的 operation 還沒釋放鎖，第二次請求就被擋。`gcloud functions describe` 看到 `status=ACTIVE` 但 `updateTime` 是舊的，代表鎖還在。不是代碼問題。

### 法
等 2-5 分鐘讓 GCP 自己釋放，不要反覆重試（越試越擠）。
確認方式：`gcloud functions describe ailiveScheduler --region=asia-east1 --format="value(status,updateTime)"`
看到 updateTime 更新才代表上一次 operation 完成。

### 解
等候後重試 deploy 成功。代碼完全沒動。

## 7. ailiveScheduler 與 task-run 的記憶寫入重複問題

### 現象
角色 Lili 有 10 條 `scheduler_post` 記憶，title 是「post 排程執行 2026-03-29」，無 embedding，hitCount 被反覆命中接近升 core 門檻。所有 learn/reflect/sleep 任務也有雙重記憶（一條有 embedding，一條沒有）。

### 心
兩個系統對同一任務各自認為自己要存記憶：
- task-run：saveInsight 有 embedding、有完整流程
- ailiveScheduler：用 Haiku 提煉標題後寫入，embedding 是空陣列，Haiku 失敗 fallback 為「learn 排程執行 YYYY-MM-DD」

兩邊沒有協議，結果是每次任務執行都產生兩條記憶，其中一條是殭屍。

### 法
記憶寫入職責統一由 task-run 負責。ailiveScheduler 移除所有 platform_insights 寫入邏輯。
ailiveScheduler 的唯一職責：掃任務 → 時間對了呼叫 task-run → 記 last_run。

### 解
ailiveScheduler executeTask 從 60 行簡化為 22 行，移除 Haiku 呼叫、toolsUsed、reply 變數、LINE 假代碼、ANTHROPIC_API_KEY secret。GCP deploy 後新任務不再產生殭屍記憶。歷史殭屍條目等 14 天 decay 自然 archive。

## 8. 行銷文案直接向量化會產生雜訊

### 現象
Vivi 被問「完美淨顏慕絲花的成分是什麼」，回答「知識庫裡有定位描述，但成分清單是空的」。

### 心
行銷文案的格式是給人看的，不是給向量化看的：
- emoji 小標題（💧 保濕修護）會被切成獨立碎片
- 購買連結、品牌故事、引言語句都進了知識庫
- 每個碎片失去上下文，角色查到殘缺資訊就幻覺

### 法
上傳前先整理成結構化 Markdown：
- H1 = 產品名稱
- H2 = 段落（產品基本資訊 / 核心成分 / 功效 / 使用方式 / 適合對象 / 注意事項）
- 成分用表格，不用 emoji 列點
- 移除行銷語氣、購買連結、引言

### 解
產出 11 份整理好的 .md 範本，Vivi 能正確回答成分問題。

## 9. .md 上傳走的是 upload-url → parse，不是 knowledge-upload

### 現象
在 knowledge-upload 加了 .md 支援，deploy 後 dashboard 還是說「只支援 .pdf 和 .docx」。

### 心
dashboard 的上傳流程是三段式：
1. `knowledge-upload-url`：生成 Firebase Storage 的 signed URL
2. 前端直接上傳到 Storage
3. `knowledge-parse`：從 Storage 下載後解析存入知識庫

知識庫 API 有三條路徑，改一條不夠，三條要同步。

### 法
每次改知識庫上傳邏輯，三個地方都要改：
- `/api/knowledge-upload-url`：副檔名白名單
- `/api/knowledge-parse`：解析邏輯
- `/dashboard/[id]/knowledge/page.tsx`：UI 的 accept 屬性和說明文字

### 解
三條路徑全部補齊後 .md 上傳成功。

## 10. chunkMarkdown H1/H2 沒有帶產品名稱錨點

### 現象
上傳兩個產品的 md 後，知識庫出現兩條 title 都叫「核心成分」的條目，Vivi 分不清哪個是哪個產品。

### 心
chunkMarkdown 按 `## 段落名稱` 切塊，title 只保留段落名稱，H1 的產品名稱被丟掉了。角色語義搜尋「完美淨顏慕絲花的成分」，兩條「核心成分」的 cosine 分數相近，隨機命中其中一條。

### 法
chunkMarkdown 改為雙層結構：
- H1 = parentTitle（產品名稱）
- H2 = sectionTitle（段落名稱）
- title 格式：`產品名稱 — 段落名稱`

### 解
`AVIVA 完美淨顏慕絲花 — 核心成分`、`AVIVA 水光澤潤白凝霜 — 核心成分`，各自清楚。

## 11. knowledge-parse 的 chunkText 會把 Markdown 表格切壞

### 現象
上傳 .md 檔後，成分表格的第一行 `| 胺基酸型潔淨因子 | 溫和不過度去脂 |` 變成了獨立知識條的 title。

### 心
dashboard 走的是 `knowledge-parse`，而 `knowledge-parse` 用 `chunkText`，`chunkText` 有一個條件：「全大寫短行 = 新標題觸發點」，表格的分隔線和表頭行觸發了這個條件，把表格切碎。

`knowledge-upload` 用的是 `chunkMarkdown`（按 # / ## 切），兩個函數邏輯不同。

### 法
`knowledge-parse` 的 .md / .txt 分支改用 `chunkMarkdown`，不走 `chunkText`。
`chunkMarkdown` 只按 `#` / `##` 切，不管表格內容。

### 解
成分表格完整保留在「核心成分」那個知識條目裡。

## 12. mammoth convertImage handler 拿不到同格的圖說文字

### 現象
docx 裡圖說文字和圖片在同一個表格格（`<th>`），但 title 還是 `filename — 圖片 N`，圖說沒有對應到。

### 心
mammoth 的 `convertImage` handler 只能拿到圖片本體（base64 + contentType），拿不到旁邊的文字。圖說和圖片雖然在同一格，但在 mammoth 的 handler 執行時，文字上下文已經分開處理了。

### 法
不用 `convertImage` handler，改用：
1. `mammoth.convertToHtml` 拿到帶完整 base64 的 HTML
2. regex 按 `<th>/<td>` 為單位，同時抽同格的圖說文字和 base64
3. 圖說 = 移除 HTML 標籤後的純文字
4. title = 圖說文字，沒有才 fallback 到 `filename — 圖片 N`

### 解
`雪玉如初精華液 純產品視角正面`、`雪玉如初精華液與模特兒半身手持`——圖說對應準確。

## 13. Haiku 圖片描述會誤判產品名稱

### 現象
Vivi 知識庫的圖片條目，Haiku 把 AVIVA 描述成「ALIVA 精華液」，content 裡有錯誤描述，污染知識庫。

### 心
Haiku 做 OCR 描述對品牌名稱不準，尤其是細小文字或特殊字體。錯誤描述進了 content，語義搜尋時會帶出錯誤資訊。

### 法
圖片條目不再存 Haiku 描述，content 只存圖片網址：
`content = "圖片網址：https://..."`
title 靠圖說做語義搜尋，URL 靠 imageUrl 欄位傳給角色。

### 解
節省 token，圖片條目乾淨，不再出現誤描述。
