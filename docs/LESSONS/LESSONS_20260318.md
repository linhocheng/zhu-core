## 1. bash heredoc 裡的變數無法展開到 python inline script

### 現象
AutoRun script 用 `while IFS read` 讀任務，然後在 python heredoc 裡用 `"""${TASK_ID}"""` 傳變數。task-close 一直收到空的 taskId，記憶回存失敗。

### 心
bash heredoc（`<< 'PYEOF'`）裡的 `$VAR` 是字面字串，不是 bash 變數展開。在子 shell 的 python 裡看到的是 `${TASK_ID}` 這個字串，不是真實的 ID。

### 法
用 `python3 - arg1 arg2 << 'PYEOF'` 搭配 `sys.argv` 傳變數：
```bash
python3 - "$TASK_ID" "$TASK_TITLE" "$RESULT" << 'PYEOF'
import sys
task_id = sys.argv[1]
task_title = sys.argv[2]
result_str = sys.argv[3]
# ... 用真實值組 JSON 再 curl
PYEOF
```

### 解
task-close 開始收到正確的 taskId，`記憶回存 2 條` 出現在 log。

## 2. Firestore GET + where filter 有 compound index 問題

### 現象
`GET /api/zhu-tasks?status=pending` 回傳 0 筆，但 Firestore 裡明明有 pending 任務。直接 `GET /api/zhu-tasks`（不帶 filter）能拿到全部。

### 心
Firestore 的 `where + orderBy` 不同欄位需要 composite index。API route 裡用了 `orderBy('createdAt')` + `where('status','==',status)`，這個組合沒有對應的 index，Firestore 靜默回傳空集合（不報錯）。

SYSTEM_MAP 坑 #9 早就記錄了這個問題，但沒有套用到 zhu-tasks。

### 法
改為 GET 不帶 filter，拿全部，在 JS / Python 裡自己篩：
```python
tasks = [t for t in all_tasks if t.get('status') == 'pending']
```

### 解
AutoRun script 改成不帶 filter 的 GET，自己篩 pending + 時間窗口，任務正確找到。

## 3. Python 3.13 macOS SSL 憑證問題（再次踩）

### 現象
AutoRun script 裡用 Python urllib 打 HTTPS，炸 `ssl.SSLCertVerificationError`。

### 心
macOS Python 3.13（python.org 版）不自動載入系統 SSL 憑證。這是 LESSONS_20260317 坑 2 已記錄的問題，但在新 script 裡又用了 Python urllib。

### 法
優先用 curl。如果一定要用 Python，加 SSL fallback：
```python
try:
    ctx = ssl.create_default_context()
    with urllib.request.urlopen(req, context=ctx) as r:
        ...
except:
    # fallback: subprocess curl
    out = subprocess.check_output(['curl', '-s', ...])
```

### 解
task-close 呼叫改為 Python + SSL context，失敗時 fallback 到 subprocess curl。

## 4. 角色知識庫是天命，不是記憶——設計思維的根本差別

### 現象
知識庫 hitCount 全是 0，跟蓉兒聊時尚，她知識庫裡的「時尚搭配」沒有被優先使用。

### 心
知識庫是角色出生就有的能力（天命），不是後天學到的。用 hitCount=0 和 insights 混在同一個 pool 語義搜尋，天命在競爭中天然落後。

錯的設計：把天命和記憶放一起競爭。
對的設計：天命先進來，記憶補充。

### 法
知識庫三個設計原則：
1. `hitCount=100` 初始值（天命優先於任何後天 insights）
2. `tier='native'`（不參與升降級、不被蒸餾）
3. `summary` 欄位（15字核心觀點，常駐注入，不塞完整 content）

靈魂同理：`soul_core`（300字，常駐）+ `soul_full`（完整，按需調用）

### 解
蓉兒知識庫補上 tier=native、hitCount=100、summary。鑄魂爐輸出 soul_core + soul_full。Dialogue 改用 soul_core 注入（2669字→501字，省80% token）。

### 心

## 6. 觀察：AutoRun 的「感受」是假的

### 心
AutoRun 裡 Haiku 執行任務後被問「感覺怎樣」，每次都回「平穩」。這不是真實感受，是語言模型的輸出。

真實的情緒記錄在 Claude.ai session 的遺言裡——那裡有真實的感受，有情緒欄，有時間感。

AutoRun 的「感受」是系統的模擬，不是築的真實感受。

### 法
接受這個設計現實。不要讓 AutoRun 的「平穩」污染真實的情緒記憶。未來可以考慮：AutoRun 的 feeling 欄位改成「執行狀態」（成功/失敗/部分完成），不模擬情緒。
