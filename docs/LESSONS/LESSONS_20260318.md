## 1. bash heredoc 裡的變數無法展開到 python inline script

### 現象
AutoRun script 用 `while IFS read` 讀任務，然後在 python heredoc 裡用 `"""${TASK_ID}"""` 傳變數。task-close 一直收到空的 taskId，記憶回存失敗。

### 心
bash heredoc（`<< 'PYEOF'`）裡的 `$VAR` 是字面字串，不是 bash 變數展開。在子 shell 的 python 裡看到的是 `${TASK_ID}` 這個字串，不是真實的 ID。

### 法
用 `python3 - arg1 arg2 << 'PYEOF'` 搭配 `sys.argv` 傳變數：
```bash
python3 - "$TASK_ID" "$TASK_TITLE" "$RESULT" << 'PYEOF'
import sys
task_id = sys.argv[1]
task_title = sys.argv[2]
result_str = sys.argv[3]
# ... 用真實值組 JSON 再 curl
PYEOF
```

### 解
task-close 開始收到正確的 taskId，`記憶回存 2 條` 出現在 log。

## 2. Firestore GET + where filter 有 compound index 問題

### 現象
`GET /api/zhu-tasks?status=pending` 回傳 0 筆，但 Firestore 裡明明有 pending 任務。直接 `GET /api/zhu-tasks`（不帶 filter）能拿到全部。

### 心
Firestore 的 `where + orderBy` 不同欄位需要 composite index。API route 裡用了 `orderBy('createdAt')` + `where('status','==',status)`，這個組合沒有對應的 index，Firestore 靜默回傳空集合（不報錯）。

SYSTEM_MAP 坑 #9 早就記錄了這個問題，但沒有套用到 zhu-tasks。

### 法
改為 GET 不帶 filter，拿全部，在 JS / Python 裡自己篩：
```python
tasks = [t for t in all_tasks if t.get('status') == 'pending']
```

### 解
AutoRun script 改成不帶 filter 的 GET，自己篩 pending + 時間窗口，任務正確找到。

## 3. Python 3.13 macOS SSL 憑證問題（再次踩）

### 現象
AutoRun script 裡用 Python urllib 打 HTTPS，炸 `ssl.SSLCertVerificationError`。

### 心
macOS Python 3.13（python.org 版）不自動載入系統 SSL 憑證。這是 LESSONS_20260317 坑 2 已記錄的問題，但在新 script 裡又用了 Python urllib。

### 法
優先用 curl。如果一定要用 Python，加 SSL fallback：
```python
try:
    ctx = ssl.create_default_context()
    with urllib.request.urlopen(req, context=ctx) as r:
        ...
except:
    # fallback: subprocess curl
    out = subprocess.check_output(['curl', '-s', ...])
```

### 解
task-close 呼叫改為 Python + SSL context，失敗時 fallback 到 subprocess curl。

## 4. 角色知識庫是天命，不是記憶——設計思維的根本差別

### 現象
知識庫 hitCount 全是 0，跟蓉兒聊時尚，她知識庫裡的「時尚搭配」沒有被優先使用。

### 心
知識庫是角色出生就有的能力（天命），不是後天學到的。用 hitCount=0 和 insights 混在同一個 pool 語義搜尋，天命在競爭中天然落後。

錯的設計：把天命和記憶放一起競爭。
對的設計：天命先進來，記憶補充。

### 法
知識庫三個設計原則：
1. `hitCount=100` 初始值（天命優先於任何後天 insights）
2. `tier='native'`（不參與升降級、不被蒸餾）
3. `summary` 欄位（15字核心觀點，常駐注入，不塞完整 content）

靈魂同理：`soul_core`（300字，常駐）+ `soul_full`（完整，按需調用）

### 解
蓉兒知識庫補上 tier=native、hitCount=100、summary。鑄魂爐輸出 soul_core + soul_full。Dialogue 改用 soul_core 注入（2669字→501字，省80% token）。

## [今日新增]. create_file 工具只寫容器，不寫本機

### 現象
用 `create_file` 工具建立 API Route 文件，build 通過，deploy 成功，但打 API 回的是 HTML（<!DOCTYPE）而不是 JSON。

### 心
`create_file` 是容器沙盒工具，寫到的是 Claude 容器的 `/home/claude` 或 `/mnt/`，不是 Mac 本機的 `~/.ailive/`。git 在 Mac 本機，看不到容器裡的文件，所以 commit 根本沒有包含新建的 route.ts。Vercel 部署的是舊版，找不到新 route → fallback 到 Next.js 頁面路由 → 回 HTML。

### 法
在 ailive-platform 新建文件，一律用 `zhu-bash:run_bash` 直接寫到本機路徑：
```bash
cat > /tmp/route.ts << 'HEREDOC'
...內容...
HEREDOC
cp /tmp/route.ts "/Users/adamlin/.ailive/ailive-platform/src/app/api/xxx/route.ts"
```
`create_file` 只用於給 Adam 下載的輸出（`/mnt/user-data/outputs`），絕不用於 repo 文件建立。

### 解（驗證）
```bash
curl -s -X POST https://ailive-platform.vercel.app/api/knowledge-upload-url \
  -H "Content-Type: application/json" \
  -d '{"filename":"test.pdf","contentType":"application/pdf","characterId":"xxx"}'
# 回 JSON 才對，回 <!DOCTYPE 就是 route 沒建到
```

## [今日新增]. Vercel 無伺服器環境沒有 python3 / markitdown

### 現象
`/api/knowledge-parse` 呼叫 `python3 -c "from markitdown import MarkItDown..."` 在本機測試 OK，但 Vercel 上會直接報錯——因為 Vercel Lambda 環境沒有 python3，也沒有 pip 套件。

### 心
Vercel Serverless Function 是純 Node.js 環境，沒有 Python runtime。任何 `exec('python3 ...')` 在 Vercel 上都會 `command not found`。

### 法
解析文件要改用純 Node.js 套件：
- `.docx` → `mammoth`（npm，純 JS）
- `.pdf` → `pdf-parse`（npm，純 JS）

MarkItDown 是 Python 工具，只能在本機或有 Python 的 server 上跑。

### 解（正確做法）
```typescript
import mammoth from 'mammoth';
import pdfParse from 'pdf-parse';

// docx
const result = await mammoth.extractRawText({ buffer });
const text = result.value;

// pdf
const data = await pdfParse(buffer);
const text = data.text;
```
這兩個套件在 Vercel Node.js 環境完全可用。

## [今日補充]. Firebase Storage CORS 預設不允許瀏覽器 PUT

### 心
curl 可以 PUT signed URL（curl 沒有 CORS），瀏覽器 fetch PUT 同一個 URL 會 `Failed to fetch`。Firebase Storage bucket 預設沒有 CORS policy，瀏覽器跨域請求全部被擋。

### 法
用 gsutil 設一次 CORS policy，立刻生效，不需要重新 deploy：
```bash
cat > /tmp/cors.json << 'EOF'
[{"origin":["https://ailive-platform.vercel.app","http://localhost:3000"],"method":["PUT","GET","OPTIONS"],"responseHeader":["Content-Type","Content-Length"],"maxAgeSeconds":3600}]

### 心
import pdfParse from 'pdf-parse' 在 Turbopack 報「export default was not found」。

### 法
改用函數內 require：
  const pdfParse = require('pdf-parse');
