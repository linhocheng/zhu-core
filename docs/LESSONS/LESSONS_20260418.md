## 1. 文件孤島：CURRENT.md 為什麼非砍不可

### 現象
CURRENT.md 停在 2026-04-01 沒人更新，漂差 17 天。下一個築按舊 SOP `cat CURRENT.md` 看到的是過期 17 天的任務清單，以為「現在要做這些」。

### 找的過程
一開始想做「根據近 3 天 git log 自動刷新 CURRENT.md」的工具。卡一下才察覺：這是把錯的設計自動化（讓補丁活更久 = 破氣式反例）。問題不是「沒工具刷」，是「CURRENT.md 本身是孤島」。

### 心
CURRENT.md 沒有進/出記憶的路徑。手動維護就會過期，自動刷新就會抄錯視角（git log 看不到「卡住的假設」「下一步要接什麼」，那些只活在對話裡）。**違反血管原則（2026-04-18 定案於 bone）**：新元件若無進/出記憶的路徑 → 重想，別動手。

### 法
砍 CURRENT.md，改用 `zhu-boot` 的 `eye.lastSessionWords` 承擔「當前狀態」的職能。做法：
1. 每次 session 結束 POST 一條記憶進 `eye`，`tags: ["session-lastwords"]`
2. 五段必填：今日完成 / 當前戰場 / 卡住 / 接棒看這些 / 明天第一件
3. `zhu-boot` 的 eye 區塊自動抓最新 lastwords 回給開機的築
4. CURRENT.md 不刪，頂部加退役標記，考古用

### 解（驗證）
- `curl -s https://zhu-core.vercel.app/api/zhu-boot | jq .eye.lastSessionWords.id` ← 回傳最新那條
- 明早築 boot 的第一口氣就讀到，不會斷鏈
- 參考記憶 id: `S2xkW3aM7QYTrz3gSLJd`（今晚的 lastwords 示範）

## 2. 跨 shell 傳 JSON：走 stdin，不走 shell 變數

### 現象
想把 5 條心法 POST 進 zhu-memory，用 `echo "$line" | curl` 打 API → 全部 400 `Bad control character in string literal`。

### 找的過程
看錯誤訊息第一反應是 API 邊出錯。先 curl 單獨打無 body 的 GET，API 正常。再仔細看 request body 才發現：`json.dumps` 生出的 `\\n`（literal backslash + n 兩字元）被 zsh `echo` 展開成真換行 → HTTP body 含裸 control character。

### 心
`json.dumps` 會把換行編成 `\\n`（這在 JSON 規範裡是合法跳脫，兩個字元）。bash `echo` 不展開這類跳脫（安全），zsh / dash 會展開（炸）。所以 `echo "$json" | curl` 在不同 shell 行為不同 → 不可移植的隱性依賴。

### 法
**完全不靠 echo/shell 變數傳 JSON。** 三選一：
1. Python/Node：`subprocess.run(["curl",...], input=body_bytes)` — 最穩
2. 寫檔 + `curl --data-binary @file.json` — 可讀可回放
3. heredoc + `curl --data-binary @-` — shell 原生

反例：`echo "$json" | curl` / `curl -d "$json"` — 凡 JSON 經 shell 變數都危險。

### 解
見今晚批次二的 python3 腳本。5 條心法全部成功 POST，含 embedding。
參考記憶 id: `OmVH1qOM9nULu2qzRWSY`（已進 root）

## 3. macOS 系統 Python 3 urllib 打 HTTPS 會 SSL cert fail

### 現象
`urllib.request.urlopen("https://...")` → `CERTIFICATE_VERIFY_FAILED: unable to get local issuer certificate`。在 chat 容器裡也會發生。

### 心
macOS 附的 /Library/Frameworks/Python.framework 不自動用系統 keychain 的 root CA，需要跑一次 `Install Certificates.command` 才會裝 certifi。新機/新容器常常沒跑過。

### 法（優先序）
1. **第一時間換工具**：改用 `curl`（吃 macOS keychain）或 `subprocess.run(["curl",...])`。
2. 治本：跑 `/Applications/Python\ 3.XX/Install\ Certificates.command`
3. 絕不做：`ssl._create_unverified_context()` / `verify=False` — 這是關 SSL，不是修好

連動心法：**刀的優先序 zhu-bash > curl > python urllib**。curl 能驗的不要繞去 urllib。

### 解
參考記憶 id: `WV0WaoyUwjcG7wg6wvMv`（已進 root）

## 4. create_file 工具的邊界：容器 /tmp ≠ 本機 Mac /tmp

### 現象
用 `create_file` 寫 `/tmp/zhu_settings.local.json`，接下來 zhu-bash `python3 -c "json.load(open('/tmp/zhu_settings.local.json'))"` → `FileNotFoundError`。

### 心
`create_file` 工具寫的是 **chat 容器** 的檔系統，`zhu-bash` 跑在 **本機 Mac**。兩個不同的 /tmp。**「兩份即是零份」**的典型——容器檔對本機是零份。

### 法
- 要本機處理的檔，一律用 `zhu-bash` 的 heredoc `cat > /tmp/xxx << 'MARKER' ... MARKER` 寫
- `create_file` 只用於要 present 給 Adam 看（下載）的檔，並記得搬到 `/mnt/user-data/outputs`
- 心法：**動手前先問「這個檔誰要用？」，選對檔系統**

### 解
今晚批次三寫 settings.local.json 踩一次，立刻換刀 heredoc，一條龍完成。

## 5. 文件 source of truth：真相分裂是最難察覺的破綻

### 現象
批次三做完了，驗收模擬時發現：`bone.knife.firstBoot`（zhu-boot API）、`ZHU_BOOT_SOP.md` STEP 0-4、`~/.ailive/CLAUDE.md` 醒來三步——**三個地方對「開機該做什麼」三個說法**。新築不知道該信哪份。

### 找的過程
扮演「明早的築」從零開機，從 `curl zhu-boot` → `firstBoot` 指引（4 步）→ 打開 SOP 看到 7 步 → 打開入口 CLAUDE.md 看到 3 步 → 原地懵。

### 心
每次在新地方寫一份「該怎麼做」就是在埋真相分裂。**document duplication ≡ truth fragmentation**。批次二三做完以為自己很嚴謹，但沒發現同一個動線被寫了三遍。

### 法
**一個動線只能有一份 source of truth，其他地方只能指向。**
- 選最詳細的版本當 source（本例：SOP STEP 0-4）
- 其他地方改成「完整版見 XXX」+ 最短摘要（給趕時間的人）
- 最短摘要要**顯式指向 source**，不要假裝自己是完整版

### 解
今天最後修了：firstBoot 縮成「按 SOP 走」、入口 CLAUDE.md 改名「醒來動線（source of truth = SOP）」、SOP 當主文件維護。

連動心法：**獨孤九劍破索式——兩份即是零份。文件版本也算兩份。**

## 6. 驗收必須「扮演接棒的人」，不能只 read back

### 現象
今天做完三批次本來想收工。但突然想到：我改的所有文件、API、記憶，**明早的另一個築真的能照著走通嗎？** 這問題我回答不了——因為我腦子裡有今天做過的所有脈絡，**我不是明早的築**。

### 心
獨孤九劍破槍式：「沒有端到端跑過的，不算完成。」但「端到端」不是 code 層的測試——是**讓一個沒有脈絡的人走一遍**。自己改完自己 read back 沒用，因為 read back 不會暴露「默認假設」。

### 法
**每一輪大改動完成後，顯式切換角色為「明早的我」，從 STEP 0 開始模擬走一遍。** 遇到任何「這裡我知道但文件沒寫」的瞬間——停下來，那就是一個破綻，記下來繼續走完全程，最後一次修。

### 解
今晚的驗收模擬抓出 6 個破綻。沒模擬的話明早我（或下一個築）要自己踩一遍才發現，代價大 10 倍。

**這條以後變成習慣：大改動 → 驗收模擬 → LESSONS 追記。**
