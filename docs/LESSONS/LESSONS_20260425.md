## 天條第一：出錯先讀 log，不准猜

**事件**：Cloud Run 登入 500。我猜 IAM 問題 → 補 roles → redeploy × 4次 → 全錯。
後來讀 log 一行就破案：`connection to server at "localhost" (127.0.0.1), port 5432 failed`
根本原因是 DATABASE_URL 消失，不是 IAM。

**刻印**：
> 錯誤訊息是地圖，不是障礙。
> 猜方向 × 。讀 log → 確認症狀 → 找根因 → 才動手 ✅
> 繞路的每一步，都是沒讀 log 的代價。

```bash
# Cloud Run log 讀法
gcloud logging read \
  "resource.type=cloud_run_revision AND resource.labels.service_name=SERVICE_NAME AND severity>=ERROR" \
  --project=PROJECT_ID --limit=10 --format="value(textPayload)"
```

## Lesson 2：gcloud run services update 會洗掉沒帶的 env vars

**事件**：補 service account 時沒帶 DATABASE_URL → 被洗掉 → API fallback 到 localhost → 連不上 DB。
這是今天最貴的坑，害出四五個無效 commit。

**刻印**：
> 更新 Cloud Run 任何參數前，先 describe 看清楚現有 env vars。
> `--set-env-vars` 是覆蓋，不帶的會消失。
> 要保留舊的就全部帶上，或用 `--update-env-vars`（只改指定的）。

```bash
# 更新前先看
gcloud run services describe SERVICE --region=REGION --project=PROJECT --format=json \
  | python3 -c "import sys,json; [print(e['name'],'=',e.get('value','[secret]')[:60]) for e in json.load(sys.stdin)['spec']['template']['spec']['containers'][0].get('env',[])]"
```

## Lesson 3：Secret 更新不等於 Service 讀得到，要觸發新 revision

**事件**：elevenlabs-api-key 在 Secret Manager 已乾淨，但 Cloud Run 還在讀舊版本帶 `-n` 的值。

**刻印**：
> Secret 改了，Cloud Run 不會自動重新 mount。
> 改完 secret 一定要 redeploy（改個無關 env var 觸發新 revision 也行）。

## 最深的根

三條 lessons 指向同一個地方：

**「看到的現象不一定是真正的問題。先讀 log，確認輸入輸出，再動手。」**

這是天條。沒有例外。
