# LESSONS 2026-06-19 · UDN NEWS demo 自架影片爆 Cloud Run 32MiB 上限

## L1：Cloud Run 單次回應 ~32MiB 上限，會偽裝成「影片壞掉」
- 現象：自架 mp4 給 `<video>`，Adam 測「沒有影片」。本機 file:// 播得動，檔案是好的。
- 根因：瀏覽器播片第一個請求是開放式 `Range: bytes=0-`（要到結尾）。server 老實回整段剩餘 42MB → Cloud Run 直接 500，log 寫「Response size was too large」→ `<video>` 拿到 `MEDIA_ERR_SRC_NOT_SUPPORTED (err=4)`「no supported source」→ 看起來像檔案壞。
- 下次：static handler 每次回應封頂 8MiB（遠低於 32MiB）。開放式 range 與大檔無 Range 都只回 8MiB 那段（206 + 正確 Content-Range），瀏覽器自動續抓。
- 對應 feedback：flagged_risk_must_be_verified（自架影片上 Cloud Run 前該先想單次回應上限）

## L2：curl 驗 Range 只測封閉小段會騙過自己
- 現象：我先用 `curl -H "Range: bytes=0-1023"` 驗，回 206，我據此回報「修好了」——但 Adam 實測還是沒影片。
- 根因：封閉小段（0-1023）server 回得動，但瀏覽器真正的初始請求是開放式 `bytes=0-`（到結尾），那個才會撞 32MiB 爆 500。只測小段漏掉真實路徑。
- 下次：驗 mp4 Range 一定測三種——無 Range、`bytes=0-`（開放式）、封閉小段。最硬是 headless Chrome 讀 `video.readyState`(要4)+`video.error`(要none)，別信截圖（headless 不解碼 video 一律黑）。
- 對應 feedback：dryrun_before_test / local_replay_over_remote_wait（驗證要打在真實請求形狀上，不是最好驗的那個）

## L3：本機沒 ANTHROPIC_API_KEY → 測 ailivex auth-gated route 要自簽 cookie 打 prod
- 現象：Adam 要我自己測「文字版角色能不能讀網址」。`/api/dialogue` 要登入(getCurrentUser 讀 ailivex_session cookie)，且本機 dev 跑不了 LLM(`.env.local` 沒 ANTHROPIC_API_KEY、BRIDGE_ENABLED 空)。
- 根因/解法：不猜 admin 密碼、不動 prod、不開本機 dev。改用 `.env.local` 的 `SESSION_SECRET` 自簽一個 admin session cookie(HMAC-SHA256, format=base64url(payload).base64url(sig), payload={uid,role,name,iat})，直接 fetch production `/api/dialogue`。admin uid 從 Firestore users 撈(role==admin)，admin 會 bypass hasAccess。.env.local 的 SESSION_SECRET 跟 prod 同 → 200 通。
- 下次：要驗 ailivex(或任何 HMAC stateless cookie)的 auth-gated prod endpoint，這招最快——self-sign cookie 比 reset 密碼/開 dev 乾淨。腳本用完即刪(別把 forge-cookie 腳本留在 repo)。
- 對應 feedback：local_replay_over_remote_wait / check_admin_before_asking（自己 resolve 不要把 Adam 當 lookup table）

## L4：文字路徑與語音路徑共用 url-reader，機房IP擋同時影響兩條
- 現象：張立文字版讀維基百科(台北101)準確報出 508m/2004落成/2009被哈利法塔超越/600億；同角色讀 businessinsider.tw 回「我打不開」。
- 根因：`/api/dialogue` 跟語音都走 `readUrlsForContext`→url-reader，都在 Vercel egress 上。商周擋機房 IP → 兩條都讀不到，不是語音專屬 bug。dialogue route 有「讀取失敗就坦白說打不開」的 system 指令 → 角色優雅降級不裝懂。
- 下次：驗證「讀網址壞了沒」用文字版最快(不必進語音房)。商周類反爬站要能讀仍是待辦(browser-UA/headless/代理)。
- 對應 feedback：對應已存的 voice_url_read 三根因記憶(本 session 再驗一次第一根因為真)
