# LESSONS 2026-06-01 · MACS dir2 對質一輪 + 成本計算 + Cloud Run 硬化

## L1：Vercel 上「多次序列 LLM 呼叫」的 worker 會撞 300s，而且 timeout 變成「永久卡死」不是「重試」
- 現象：dir2 的 cross-review 在 Vercel 一個函式內跑 5 趟序列 bridge 修訂（每趟 ~60-90s），第 5 趟撞 Vercel 300s 上限被砍。案子卡 `對質=4/5` 不動、worker_run 永遠 "running"、repair=0，不會自己好。
- 根因：Vercel 砍函式 → Cloud Tasks 在 **10 分鐘 lock 窗內**重送 → `acquireWorkerLock` 看到 status=running 回 `already_running` → harness 回 **200** → Cloud Tasks 以為成功、**停止重試**。所以一次 timeout 被「鎖 + 200」轉成永久 stall，不是預期的重試。
- 下次：worker 若會跑「N 次序列 LLM 呼叫 / 預期 >300s」→ 一律放 Cloud Run（無時限），不要放 Vercel。這是 `reference_websearch_cloudrun_not_vercel` 的**推廣**：不只 web_search，任何多呼叫長 worker 都適用。
- 修法（三層）：① skip-done（已 crossReviewed 的 memo 跳過 → 可續跑）② reconciler 偵測 stale-running cross-review（>10min 未 done）自動重發 ③ 終極：搬 Cloud Run，5+ 條一次跑完零停頓。
- 對應：reference_websearch_cloudrun_not_vercel、reference_vercel_300s_lambda_limit、獨孤九劍狀態機三問（lock/lease/attemptId）。

## L2：stub 測試過 ≠ 真案會過——真案才抓到 300s 雷
- 現象：orchestration stub 測試 21/21 全綠（驗 barrier→cross-review→synthesis 編排正確），但**真案第一跑就卡死**在 300s timeout。stub 用假 enqueue、零 LLM，測不到「實際 LLM 呼叫累計時間 > Vercel 限制」。
- 下次：編排正確性 stub 測得到；**資源/時間上限只有真案測得到**。新增「多呼叫 worker」後，端到端真跑一次是必要驗收，不能只靠 stub 綠燈。
- 對應：feedback_one_inch_cake_mvp（沒端到端跑過＝不存在）、破劍式。

## L3：不要把「會常改的設定」vendor 進第二個部署環境（真相分裂）
- 現象：cross-review 搬 Cloud Run 時，需要分析師人設。Adam 常改 `defaults.ts` 的 roleFraming 人設——若 vendor 一份進 Cloud Run，他改 Vercel 那份、Cloud Run 那份就漂移。
- 做法：Cloud Run 對質**不 vendor 人設**，改用「soul + 通用 analysisBase + 工作流標籤」較輕人設（memo 本身已帶 round-1 人設聲音），soul/analysisBase 仍從 macs DB `settings/roles` 即時讀。代價：對質回合人設較淡，但矛盾照樣攤得出（真案驗證過）。
- 下次：跨部署複製時，**只 vendor 穩定的東西**；常變的設定走共用 DB 或單一源 fetch，別留兩份。
- 對應：sync-truth-principle（兩份即零份）、reference_bridge_not_in_git。

## L4：dir2 對質一輪確實有效——tension map 直接點名其他分析師
- 現象：對質後的 synthesis，tensionMap 出現「Murphy 估算的現金流斷裂窗口」「Evan 的 CAC 分析顯示低價毛利歸零」「掉進 Kage 定義的無人區」——分析師互相引用、攤出真實衝突，洞察沒被磨鈍反而更尖。
- 設計關鍵（成立的原因）：① 一輪固定、不繞圈 ② 收尾交給 synthesis 拍板，**不要求分析師投票出共識** ③ 修訂 prompt 明令「禁止為和諧磨鈍、矛盾要留著」。
- 對應：feedback_courage_in_the_moment（不為順暢妥協）、本檔 dir2 設計。

# 下午場 · ANEWS source A/B 雙管道上線

## L5：改了 worker + 後端 API，忘了部署 Vercel 寫入端 → B issue 跑成 A
- 現象：建了 sourceProvider="B" 的 issue，worker log 卻顯示 `provider=A`。
- 根因：部署了 Cloud Run worker（讀 issue.sourceProvider），改了 `editorial-jobs/route.ts`（寫 sourceProvider）+ dashboard 表單——但 **route 改動只在本機，prod Vercel 還是舊 code**，沒把 sourceProvider 寫進 issue doc → worker 讀到 undefined → 預設 A。
- 流動斷裂：選擇（表單）→ 寫入（API）→ 讀取（worker）三段，只部署了讀取端，整條鏈斷在中間。
- 下次：跨 Vercel + Cloud Run 的功能，**兩邊都要 deploy 才算上線**。端到端驗證前先列「這條資料流經過哪些部署單位」，逐個確認部署過。
- 對應 feedback：interface_blood_vessel_check、backend_client_must_sync。

## L6：harness 壓線過 ≠ prod 過；B 綜述首跑就撞 bridge 524 + JSON 截斷
- 現象：B 兩篇 source，一篇 `bridge 524`（CF 邊緣超時），一篇 `B_PARSE_ERROR` JSON position 3205 截斷。
- 根因：驗證 harness 那次綜述 109.5s「逼近」130s 天花板剛好過；prod 真量（含我加的 query-gen 一跳）一翻就過 524。JSON 截斷疑似 Sonnet 4.6 預設 extended thinking 吃掉 output budget。
- 「逼近天花板」被我寫成計畫的「撞到再說」——結果第一次真跑就撞。逼近 = 紅燈不是綠燈。
- 下次：latency 逼近硬上限的方案別當「可上線之後再說」，上線前就降到安全帶，或明標「首跑可能就掛」。
- 對應：reference_sonnet46_effort_low（綜述要加 --effort low）、vercel_300s_lambda_limit。

## L7：needs_repair 不自動 fallback 的設計，第一次真實失敗就證明價值
- B 兩篇都掛、停 needs_repair，**完全沒 fallback 回 A 燒付費 key**——天條守住。
- 刻意設計：安全來自「建立時可選 + 兩條都活著」，不是靠自動切換。實戰驗證：失敗時安靜停住、可控、不偷花錢，比自動 fallback 健康。
- 對應：feedback_bridge_silent_fallback_double_burn、feedback_no_paid_api_without_consent。
