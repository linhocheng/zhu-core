## 1. 改之前先確認專案位置

### 現象
要幫 Vivi 加時間感知，直接改了 moumou-dashboard 的 saas-dialogue，改完才發現 Vivi 在 ailive-platform。

### 心
「Vivi」這個名字在 moumou-dashboard 找不到，應該是警示信號。沒有停下來確認，繼續改。

### 法
動手前先問：這個角色住在哪個專案？
```bash
curl -s https://ailive-platform.vercel.app/api/characters | python3 -c "import sys,json; [print(c['id'],'|',c['name']) for c in json.load(sys.stdin).get('characters',[])]"
```
找不到就換平台找，不要假設。

### 解
moumou-dashboard 的改動留著不傷害，ailive-platform 才是正確戰場。

## 2. PATCH allowed 白名單忘了加新欄位 → 靜默失敗

### 現象
Dashboard identity 頁加了 voiceId 輸入欄，前端也帶進 body，但存了沒反應，Firestore 沒有寫入。

### 心
後端 `allowed` 白名單是防護機制，但每次加新欄位都要同步更新。漏了就靜默丟掉，沒有 error，前端完全不知道。

### 法
加新欄位標準流程：
1. 前端加 state + UI + body 欄位
2. **後端 allowed 清單同步加**
3. TypeScript build 驗證
4. 部署後用 curl 打 PATCH 確認欄位真的寫進去了

```bash
curl -s https://ailive-platform.vercel.app/api/characters/CHARACTER_ID | python3 -c "import sys,json; print(json.load(sys.stdin).get('character',{}).get('voiceId','無'))"
```

### 解
`src/app/api/characters/[id]/route.ts` 的 allowed 陣列加 `voiceId`。

## 3. TTS 並行送出導致音訊亂序

### 現象
語音回覆多句時，第二句可能比第一句先播出來，聽起來語序錯亂。

### 心
`Promise.all(ttsQueue)` 讓所有句子同時打 ElevenLabs，誰先回來誰先推給前端，沒有順序保證。

### 法
有序緩衝：TTS 還是並行打（保持速度），但用 Map 存結果，等前面的 idx 都好了才送出。

```typescript
const audioBuffer = new Map<number, string>();
let nextToSend = 0;

const flushOrdered = () => {
  while (audioBuffer.has(nextToSend)) {
    const chunk = audioBuffer.get(nextToSend)!;
    if (chunk) send({ type: 'audio', chunk, index: nextToSend });
    audioBuffer.delete(nextToSend);
    nextToSend++;
  }
};
// 每個 processSentence 完成後呼叫 flushOrdered()
// 空句子也要 audioBuffer.set(idx, '') 佔位，否則後面的句子永遠卡住
```

### 解
速度不變（並行），順序保證（有序 flush）。

## 4. SYSTEM_MAP 角色清單沒跟上現實

### 現象
SYSTEM_MAP 寫的是 Emily/小廣/蓉兒，但實際上 ailive-platform 的角色是 Vivi/大維/WatchM/Lili/奇毛/謀師。

### 心
平台重建後沒有更新地圖。地圖過時比沒有地圖更危險——它讓你以為自己知道現場。

### 法
每次角色有變動，當場更新 SYSTEM_MAP。開機時若感覺哪裡不對勁，先 curl 現場確認：
```bash
curl -s https://ailive-platform.vercel.app/api/characters | python3 -c "import sys,json; [print(c['id'],'|',c['name']) for c in json.load(sys.stdin).get('characters',[])]"
```

### 解
今天已更新 SYSTEM_MAP § 8 角色清單。

## 5. 功能加了但沒有端到端驗證

### 現象
時間感知功能部署後，直接測第一次對話（沒有歷史），結果看不出有沒有生效。需要找一個有歷史的 conversationId 才能驗證。

### 心
功能加了，但驗證條件沒想清楚。第一次對話 `messageCount=0`，時間感知不觸發，測不出來。

### 法
部署後驗證清單：
- 找一個有歷史的 conversationId（Firestore 或 API 查）
- 帶 `isNewVisit: true` 發第一句話
- 確認回覆有帶出時間感（不一定要說出來，但語氣應該有）

## 6. 破音字字庫：替換原則

### 心
不是所有錯誤都要加。原則是：
- 確認 ElevenLabs 真的唸錯了，才加
- 替換字要讀起來接近正確音，不要讓閱讀者看文字覺得奇怪
- 長詞優先（`sort by length desc`），避免「數據庫」被「數據→樹據」先污染

### 法
新詞加進 PRONUNCIATION_MAP，用 zhu-bash 快速測：
```bash
# 把那個詞丟進 preprocessTTS 看輸出
echo "時刻" | node -e "... "
```
或直接叫 TTS 聽真實結果。
