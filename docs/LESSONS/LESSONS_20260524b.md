# LESSONS 2026-05-24b · ANEWS prod test script — writeWorkerTrace targetId bug

## L1：Firestore trace 欄位名稱不等於 payload 參數名稱
- 現象：`parseErrIds` 永遠是空 Set，PARSE_ERROR detection 從來不觸發，alignment 無限卡住
- 根因：`writeWorkerTrace` 存的是 `targetId`（harness 的通用欄位），不是 `articleId`；script 讀 `d.data().articleId` = undefined
- 下次：查 Firestore 欄位名稱，一定要看 trace.ts 的 TraceData interface，不能靠參數名稱猜
- 對應 feedback：「記憶會說謊，越具體的記憶越要去現場驗證」

## L2：prod Cloud Tasks pipeline 需要三層 alignment 恢復路徑
- 現象：alignment 掛掉的方式有三種：(A) 所有 article alignment_done 但 orchestrate callback 靜默失敗、(B) article 永遠在 blueprint_ready 有 PARSE_ERROR trace、(C) article 到 needs_repair 超過重試上限
- 根因：Cloud Run → Vercel callback 不保證到達；alignment LLM 有機率 PARSE_ERROR；Cloud Tasks 有重試上限
- 下次：prod test script 的 poll loop 每次都要檢查三種情況，不能只想到一種
- 對應 feedback：「補 session/auth 後必跑完整 dry-run — 斷鏈很少一個原因，第一層通了不算完成」

## L3：每次「卡住」先加診斷 log 再猜原因
- 現象：alignment_running 卡住 41+ polls，完全不知道 article 層級發生什麼事
- 根因：script 只印 issue.status，沒印 article 狀態，無從判斷是 A/B/C 哪種
- 下次：poll loop 內加 per-article 狀態 log（每 N polls 印一次），debug 速度提升 10x
- 對應 feedback：「能本機重現就不要等遠端 cycle」→ 看不到中間狀態等於在猜
