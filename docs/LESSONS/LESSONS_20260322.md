## 1. git identity 沒設 → Vercel 部署全擋

### 現象
Vercel 每次部署都報 `Unexpected error`，builds: 0，根本沒進 build 階段。

### 心
Vercel GitHub integration 會驗 committer 身份。git 沒設 `user.name` / `user.email`，GitHub 無法關聯 committer，Vercel 直接在 queue 層擋掉，不進 build，所以看不到任何 build log。

### 法
```bash
git config --global user.name "adamlin"
git config --global user.email "adam@dotmore.com.tw"
# amend 最後一個 commit 修正 author
git commit --amend --reset-author --no-edit
git push --force
```

### 解
Vercel ls 出現 `● Ready`，部署成功。

## 2. Firebase Admin SDK bucket.name 含 \n — 圖片 URL 損毀

### 現象
對話生圖第一張正常，第二張開始 403/404。Firestore 裡的 imageUrl 有換行字元。

### 心
Firebase Admin SDK 回傳的 `bucket.name` 帶有 `\n`（`moumou-os.firebasestorage.app\n`）。凡是用 bucket.name 拼 URL 的地方，回傳的 URL 都是壞的。ref URL、characterSheet URL 也是同樣根因。

### 法
所有用到 bucket.name 或 Firestore 讀出的 URL 的地方，一律清洗：
```ts
const cleanBucket = bucket.name.replace(/\n/g, '').trim();
return `https://storage.googleapis.com/${cleanBucket}/${finalPath}`;

// Firestore 讀出的 URL
if (referenceImageUrl) referenceImageUrl = referenceImageUrl.replace(/\n/g, '').trim();

// response 存入前
generatedImageUrl = url.replace(/\n/g, '').trim();
```

### 解
URL 無換行，圖片可正常訪問。

## 3. Gemini functionResponse.name 必須是工具名稱，不是 call_id

### 現象
換 Gemini 主對話後，`generate_image` tool 被呼叫（toolsUsed 有記錄），但 Gemini 不繼續回覆，imageUrl 空白，圖沒有出現在對話裡。

### 心
Gemini 的 `functionResponse` 格式要求 `name` 欄位必須跟對應的 `functionCall.name` 一致（即工具名稱，如 `generate_image`）。之前用的是 OpenAI 的 `tool_call_id`（`call_0`），Gemini 認不出來，拿不到 tool result，所以不繼續。

### 法
建立 `toolCallIdToName` 映射，`functionResponse` 查映射取工具名稱：
```ts
const toolCallIdToName: Record<string, string> = {};
for (const m of messages) {
  if (Array.isArray(m.tool_calls)) {
    for (const tc of m.tool_calls) {
      toolCallIdToName[tc.id] = tc.function.name;
    }
  }
}
// functionResponse
{ functionResponse: { name: toolCallIdToName[m.tool_call_id] || 'tool', response: { output: m.content } } }
```

### 解
Gemini 正確拿到 tool result，繼續生成最終回覆，imageUrl 正常存入。

### 解

## 5. Firebase Storage bucket 沒設 allUsers:objectViewer → makePublic() 靜默失敗

### 現象
第一張圖公開成功，後續圖片 403 Access Denied。

### 心
預設 Firebase Storage 用 per-object ACL。`makePublic()` 偶爾在高頻呼叫或 race condition 時靜默失敗，圖片存在但不公開。

### 法
Bucket 層級設定 allUsers 公開讀取，一勞永逸：
```bash
gsutil iam ch allUsers:objectViewer gs://moumou-os.firebasestorage.app
```
這樣所有上傳的物件自動公開，不依賴每次 `makePublic()`。

### 解
之前 403 的圖片改為 404（檔案本身就沒存到），新上傳圖片全部公開。

## 7. knowledge embedding 退化 — content 含 URL 稀釋語義

### 現象
不同 query 對同一批 knowledge 的 cosine score 完全一樣（退化）。

### 心
圖片 knowledge 的 content 是 `商品描述\n\n圖片網址：https://...`。URL 字串被一起送進 embedding，語義被大量 URL 文字稀釋，向量退化成幾乎固定的值。

### 法
生成 embedding 前清洗掉 URL 行：
```ts
const cleanedContent = content.split('\n')
  .filter(line => !line.startsWith('圖片網址：') && !line.startsWith('http'))
  .join(' ').trim();
const embedding = await generateEmbedding(`${title} ${cleanedContent}`);
```

### 解
debug-kb 測試，不同 query 對不同 knowledge 的分數出現差異，語義搜尋恢復正常。

## 9. Python regex 替換 → 單引號被轉義成 \' 導致語法錯誤

### 現象
用 `re.sub` 替換 TypeScript 程式碼，結果單引號 `'` 變成 `\'`，build 報 `Unterminated string constant`。

### 心
Python regex 替換字串裡的 `\'` 被解釋成轉義序列。

### 法
替換字串用 Python raw string 或直接用 `str.replace()` 取代 regex：
```python
content = content.replace("old string", "new string")  # 優先用這個
# 或 regex replacement 不要用 \' 
```

### 解
改用 `str.replace()` 後 build 通過。

## 10. knowledge 排序：知識庫永遠優先於 insights

### 現象
用戶問商品相關問題，但 knowledge 可能被高分 insight 擠到後面或被 threshold 過濾掉。

### 心
knowledge（天命資料）和 insight（後天記憶）混排純比 cosine score，知識庫沒有保障名額。

### 法
排序時 knowledge 強制排首位，同 type 再比 score：
```ts
.sort((a, b) => {
  const aIsKnowledge = a.d._type === 'knowledge' ? 1 : 0;
  const bIsKnowledge = b.d._type === 'knowledge' ? 1 : 0;
  if (bIsKnowledge !== aIsKnowledge) return bIsKnowledge - aIsKnowledge;
  return b.score - a.score;
})
```
未來升級：兩個 pool 分開取（knowledge top-5 + insight top-10），互不擠壓。
