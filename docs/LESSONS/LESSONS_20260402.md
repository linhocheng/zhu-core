## 1. Web Speech API：continuous 設定決定生死

### 現象
實作 Web Speech API 後，文字有即時顯示，但按「送出」後 UI 永遠卡在「正在處理...」。

### 根因
`continuous: false`（預設）→ 用戶說話停頓時 Chrome 自動停止辨識，`onend` 已在背景觸發。  
用戶按「送出」時才設 `onend` callback → recognition 已停，callback 永遠不會再觸發。

### 找的過程
以為是效能問題，做了一堆無關的優化（Haiku、跳過 embedding、跳過 episodic），都是錯方向。  
後來 Adam 帶著回溯才對上：「加了 Web Speech API 才卡住」。

### 心
根因診斷要跟著症狀走，不要跟著「感覺可以優化」走。送出卡住 → 查送出流程 → 找 `onend` 沒觸發，兩步就到了。

### 法
```typescript
// 兩個關鍵修正
rec.continuous = true;  // 不自動停，等用戶主動送出

// 送出時不等 onend，直接讀已有文字
const stopAndSend = () => {
  try { rec.stop(); } catch { /* already stopped */ }
  const text = (finalTextRef.current + ' ' + interimText).trim();
  sendToDialogue(text); // 直接送，不等 callback
};
```

### 解
`continuous: true` + 送出直接讀 ref，不靠 `onend`。測試：說話→停頓→繼續說→按送出，文字完整送出，對話正常。

## 2. 優化順序錯了：先搞清楚 WHY，再動刀

### 現象
語音對話 26 秒才回應，開始一路砍：Haiku 換 Sonnet、跳過 embedding、跳過 episodic block、跳過 Haiku 推理。最後角色知識殘缺，答不上來。

### 根因
26 秒的真正原因是 system prompt 的天條寫錯了：
> 「開口前**必須**先呼叫 query_knowledge_base 查一次」

這讓角色每一句話（包括說「你好」）都強制跑完整知識庫查詢流程（embedding → Firestore → Haiku 推理 → Sonnet 回應），疊加起來 26 秒。

真正的修法只需要一行：把「必須查」改成「問產品才查」。

### 心
看到慢就想優化是反應，不是診斷。應該先問：**為什麼這個特定操作慢？** 找到根因再動刀。

### 法
**三問診斷法**：
1. 哪一步慢？（timeline 拆開看）
2. 這一步為什麼存在？（是 feature 還是 bug？）
3. 移除這一步，功能還完整嗎？

這次的答案：「強制查知識庫」是 feature 寫成 bug，移除強制，不移除查詢能力。

### 解
天條改為：「問到產品、成分、功效才查知識庫，一般對話直接回應」。  
讓 `tool_choice: auto`，Sonnet 自己判斷要不要查。

## 3. 語音模式不能砍資料換速度

### 現象
為了讓語音回應快，改用 Haiku 當主模型、限制 max_tokens 400、跳過 embedding 搜尋。角色變笨，答不上來。

### 心
語音模式的「快」是用戶感知的快，不是減少資料量的快。  
角色的大腦（Sonnet）和知識（knowledge base）不能動。可以動的是：**減少不必要的 Firestore 讀取**，不是砍資料本身。

### 法
```
可以省的：
- 強制每句查知識庫 → 改成按需查
- episodic block（語音時次要，但不是不要）

不能省的：
- 主模型用 Sonnet
- knowledge base 完整語義搜尋
- skills 注入
```

## 4. TTS 字庫：順序決定正確性

### 設計
兩階段預處理，順序不能反：
```
原文 → 中台用語轉換（概率→機率）→ 破音字替換（機率→機律）→ ElevenLabs
```

### 為什麼順序重要
若反過來：「概率」先過破音字替換（沒命中），再轉「機率」，TTS 拿到「機率」但沒替換成「機律」，仍念錯。

### 法
長詞優先匹配（防止「明顯著」被「顯著」先命中變成「明顯住著」）：
```typescript
const keys = Object.keys(map).sort((a, b) => b.length - a.length);
for (const key of keys) { result = result.replaceAll(key, map[key]); }
```

## 5. 存在感問題比效能問題更深

炬的觀察：每一輪對話角色都「重新成為自己」，而不是「一直是自己」。  
這不是優化問題，是架構問題。

### 現狀電流
```
每輪：載入角色資料 + 讀對話歷史 + 讀 episodic + 呼叫 Sonnet
```

### 理想電流
```
對話開始：載入角色資料（一次）
每輪：只讀對話歷史 + 呼叫 Sonnet
對話結束：沉澱記憶
```

### 短期解法方向
用 conversationId 做 session key，第一輪載角色資料存入 cache（Firestore session doc 或 Redis），後續輪次直接讀。  
不傳 characterSession 到前端（payload 太肥），在後端做。

### 長期：Gateway
這個需求正是 Gateway 的核心職責。Gateway 建立後，session 管理自然被接管。

## 6. STT 選型教訓

**瀏覽器 Web Speech API（Chrome/Edge）**
- 優點：即時顯示、零延遲、免費
- 缺點：iOS Safari 不支援、`continuous` 設定需謹慎

**Gemini 2.5 Flash（fallback）**
- `gemini-2.0-flash` 已對新用戶下架，要用 `gemini-2.5-flash`
- batch 模式（送整段音檔），延遲 1-3 秒

**降級策略**
```
Chrome/Edge → Web Speech API（即時）
iOS Safari / Firefox → Gemini STT（batch）
```
