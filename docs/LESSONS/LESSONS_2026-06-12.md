# LESSONS 2026-06-12 · ailivex 即時語音 v2 掛斷記憶收尾 + 上次對話連貫

## L1：LiveKit agent 掛斷後做 LLM 收尾，必須拉高 `shutdown_process_timeout`
- 現象：v2 掛斷時的記憶提煉（lastSession + 記憶）一直沒寫進去，log 連 `[finalize:` 第一行都沒有，像是整段沒跑。
- 根因：`WorkerOptions` 的 `shutdown_process_timeout` 預設 **10 秒**。room 斷 → job 子行程 shutdown → `add_shutdown_callback` 跑收尾，但收尾第一步就在打 bridge LLM（>10s）→ 子行程被 SIGKILL，還沒 log 就死。`room.disconnect()` 在 callback 之前先跑（job_proc_lazy_main），所以 callback 裡無法再對前端 publish data。
- 下次：任何「掛斷後在 shutdown callback 裡跑 LLM/網路 I/O」的 agent，先把 `shutdown_process_timeout` 拉到 60–90s；收尾順序＝最不能丟的先做（先秒存逐字稿、第一行就 log 證實有跑，再做慢的 LLM）。
- 對應 feedback：silent_failure_absent_log（缺席的 log 要當靜默失敗診斷）。

## L2：「有記憶但不連貫」＝開場念摘要 + lastSession 寫入的回播時間差
- 現象：撥回同角色，它知道很多事但開場硬，扯回更舊的話題（投资人），沒接你結尾剛交代的最新事（UDN 等一下聊）。
- 根因有二：①greeting 指令叫它「把摘要自然帶出」→ 它就複述摘要的開頭句（舊話題），沒去接結尾的最新/未完線。②lastSession 靠掛斷後 LLM 萃取，要 ~30s 才寫完；回播太快 → 新通話載到舊的 lastSession。Adam 自己驗證：等久一點再撥就接得上。
- 解（三管齊下）：①prompt 注入「上次聊到最後」的**原話結尾**（逐字稿 step① 秒存、最快最不 stale）讓它從真話接，不從摘要接；②greeting + lastSession 區塊明確編碼「**最新未完的事第一優先，不扯回舊話題**」；③finalize 把 lastSession 與記憶提煉改 `asyncio.gather` 並行，寫入 ~30s→~15s 縮短回播窗口。
- 殘留：秒回播（<~5s）連原話結尾都還沒存完，仍可能差一拍。根治要「通話中即時滾動存逐字稿」（未做）。
- 對應 feedback：memory_can_lie（去現場讀真實逐字稿才看出是 greeting 念稿，不是沒記到）。

## L3：看現場讀 Firestore 用自己的 gcloud token 走 REST，不碰 SA 密鑰
- 要驗 conversation doc 的 lastSession / 逐字稿時，用 `gcloud auth print-access-token` + Firestore REST `GET .../documents/conversations/{id}`，不去抓 Secret Manager 的 service account JSON（守紅線：不暴露密鑰）。

## L4：同一功能兩條派工路徑＝真相分裂，修一條不等於修好
- 現象：ailivex 文件功能「卡住」。Adam 給的交接只修了 `documents.ts`（cleanEnv 洗 env 字面 `\n`）並 deploy，但語音撥進去叫角色寫文件，document 仍停 pending、doc-worker log 零新活動。
- 根因：**文字路徑**（`/api/dialogue` → TS `dispatchDocumentJob` → 直接 fetch worker）跟**語音路徑**（Python agent `firestore_loader.create_document_job` → `_enqueue_job` → Cloud Tasks）是**兩套完全不同的派工機制**。TS 側早改成直接 POST、Cloud Tasks 廢棄；但 Python 側還在走 Cloud Tasks，而 agent 根本沒設 `DOC_TASKS_QUEUE`/`DOC_WORKER_URL` → `_enqueue_job` 每次靜默 `留 pending`。鐵證：agent log `[doc] created ...` 緊接 `[enqueue] Cloud Tasks 未設定，job ... 留 pending`。
- 修法：把 Python `_enqueue_job` 改成 fire-and-forget 背景 thread 直接 POST worker（跟 TS 同招、消滅 Cloud Tasks 依賴）+ yaml 加 `WORKER_SECRET`(secretRef)/`DOC_WORKER_URL`。重部署 agent（00017-rqb）。
- 下次：debug 一個功能前先問「這功能有幾條進入路徑？」。破綻三處之「真相分裂」——同一件事兩份不同步的實作，過了一條別宣稱修好。
- 對應 feedback：backend_client_must_sync（兩側 contract 要對齊）、solve_root_not_symptom。

## L5：worker 磁碟源碼 ≠ 線上部署版，照線上現實建 fix
- 現象：照磁碟 `cloud-run/doc-worker/src/index.ts` 寫的是 `app.post('/process')` 無鑑權；手動 curl `/process` 回 `Cannot POST /process`，curl 根路徑 `/` 無 header 回 `{"error":"unauthorized"}`。
- 根因：線上部署的 worker 是**另一版**——handler 在根 `/` 且要 `x-worker-secret`。無 git 的老問題（磁碟/線上分裂）。
- 下次：對 worker 建 client 前，先用 curl 探真實 endpoint + 鑑權，照**線上現實**建，不照磁碟源碼。記憶會說謊，源碼也會。
- 順帶：`gcloud run services logs read` 撞 NoneType 直接 crash → 改用 `gcloud logging read 'resource.type=cloud_run_revision AND resource.labels.service_name=...'`。
- 對應 feedback：memory_can_lie（延伸：disk source 也會 lie）、diagnosis_verify_before_write。

## L7：記憶會說謊，但要逐檔驗、別一竿子打翻
- 現象：lastwords 警告「AIR 磁碟落後、page.tsx 會踩回線上」。我實查 `page.tsx`：版本標籤=`c-voice-ws`、mtime 與生產部署吻合 → 本機就是部署源頭，**該警告過時**。但同一份磁碟的 `documents.ts` 卻**真的**是舊碼。
- 根因：雙機分裂是「逐檔」的，不是「整台」的。某些檔 AIR 領先/同步、某些落後。一句「磁碟落後」會誤導。
- 下次：驗「磁碟落後」要落到**具體檔**（mtime + 版本標籤 + 對賬部署時間），不要整台下結論。
- 對應 feedback：`feedback_memory_can_lie`。

## L8：cloudbuild 複製版號忘了改 entry（讀碼就抓到，沒等部署炸）
- 現象：`agent/cloudbuild-v3.yaml` 部署目標服務名改成 `ailivex-realtime-agent-v3`，但 `--args` 還是 `-m,agent.main_v2,start`。一部署 V3 服務就會跑成 V2、註冊成 `ailivex-realtime-v2`，收不到 v3 dispatch。
- 根因：cp v2→v3 只改了一半。和 v3 前端頁送 `{v2:true}`、token route 無 v3 分支 = 同一批「半成品未差異化」。
- 下次：cp 一個版本當新版時，grep 全檔所有 `_v2`/`v2`/舊版號，逐一確認該改的都改了再部署。

## L6：「查清楚再動手」救一整輪迴——框架可能早就內建你要的東西
- 現象：v4 單機群聊，PLAN P2 假設要自己手搖「per-participant STT + 合併 context」（最大工程）。Adam 提「一支手機靠聲紋給名字」又自己補「聲紋是假議題」。
- 查清楚後翻盤：①Soniox `stt-rt-v4` 原生支援 speaker diarization（`enable_speaker_diarization=True`，每字帶 speaker）②livekit-plugins-soniox 1.5.1 把 `speaker_id` 透到 `SpeechData`③livekit-agents 1.5.1 的 `UserInputTranscribedEvent` 帶 `speaker_id`，**還內建 `MultiSpeakerAdapter`**（包 STT、自動標 primary/background speaker）。→ 群聊單機可行、不用造輪子、不用聲紋生物辨識。
- Adam 的直覺對：要的是 **diarization（分得出不同聲音）+ 自報名**，不是聲紋建檔。聲紋是假議題。
- 方法：動手前 `python3 -m pip download <pkg>==<確切版> --no-deps` 解開讀源碼（不靠記憶、不靠 main branch、不靠 PLAN 假設）。grep `enable_`/`speaker`/`class XxxAdapter`。
- 對應 feedback：framework_vs_reflex（先查框架有沒有，別反射造）、diagnosis_verify_before_write、one_inch_cake_mvp。

## L7：主動發話「選沉默」不是 bug 是性格；「還在嗎」像罐頭是品質不是機制
- 現象：v3 擬真 backoff 機制完美（間隔 3→6→13→39s 退讓、抖動、自我重排都對），但 im=3 內斂角色**每次評估都選沉默** → Adam 聽不到聲、以為沒觸發。
- 根因：①選沉默是 **soul 驅動的設計目標**——換 im=5 熱情角色就冷場 8s 開口（「你在想。」）。同機制、性格不同。②長靜默時 LLM 收斂到通用 check-in（還在嗎／你還好嗎），是新生成但內容太安全 → **感覺像罐頭但不是固定字串**。
- 解：prompt 明令禁通用問候、要求「從上下文＋角色＋當下默契長出具體話」「越被晾語氣越淡但不准退化成罐頭」。機制不動，只調判斷的 prompt。
- 對應 feedback：memory_can_lie（log 看似沒觸發，其實觸發了只是選沉默）、soul_design_narrative。
