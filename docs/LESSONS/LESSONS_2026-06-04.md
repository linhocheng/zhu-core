# LESSONS 2026-06-04 · MACS Mode 2 首跑端到端

## L1：Cloud Run 有自己的 schema，Vercel lib 改了沒用
- 現象：`lib/llm/synthesis.ts` 有完整 `HybridSynthesisSchema`，但 synthesis 跑在 Cloud Run，Cloud Run 只認識 Mode 1。Recommendation 讀 `hs.dataAnchoredTruth.summary` crash。
- 根因：synthesis 超過 300s（兩次 bridge call），Vercel 擋不住，搬去 Cloud Run。Cloud Run 是獨立 source，Vercel lib 改動對它完全無效。
- 下次：改 lib/pipeline 之前先問「這個 worker 實際上跑在哪？」Vercel lambda < 300s / Cloud Run 長任務，兩邊要分開改。
- 對應 feedback：後台改動必同步前台（harness）+ 記憶會說謊（改了 Vercel 以為改完了）

## L2：每個 route.ts 都要獨立讀 strategyMode，不會自動繼承
- 現象：lib/pipeline 全面 hybrid 化，但 roadmap/partner-review/export 三個 route 都沒讀 `c.strategyMode`，仍然走 Mode 1 路徑。
- 根因：route 是 harness，pipeline lib 是邏輯層。harness 不讀 mode，pipeline 就永遠跑預設（market_evidence）。
- 下次：改完 lib/pipeline 之後，每個 route 必須過一遍 checklist：「有讀 strategyMode 嗎？有往下傳嗎？readArtifact 的 generic type 改成 union 了嗎？」
- 對應 feedback：改 lib/pipeline 必同步改 app/api/workers

## L3：needs_repair SOP — 從失敗的 stage 重進，不是從 brief_intake
- 現象：本想從 brief_intake 重打，但 synthesis 失敗只需要重打 synthesis，多打前段是浪費。
- 根因：之前的 SOP 寫「reset 到 brief_intake」，適合完全重來，但 mid-pipeline 失敗有更精準的入口。
- 下次：看 status 判斷從哪裡重進：synthesis_running → 重打 synthesis；roadmap_running → 重打 roadmap。
- 對應 feedback：needs_repair SOP 更新進 MODE1_TO_MODE2_LESSONS.md

## L4：Mode 1 only 函式（assembleDeliverables）必加 mode guard
- 現象：export worker 呼叫 `assembleDeliverables`，裡面讀 `recommendation.options.find()`，hybrid 沒有 `options` → crash。
- 根因：assembleDeliverables 是早期 Mode 1 寫死的，沒預留 hybrid 路徑。
- 下次：任何讀 `recommendation.options`、`synthesis.keyInsights` 等 Mode 1 專屬欄位的地方，先加 mode guard。Hybrid 的報告由 `runReportBuild` HTML 路徑處理，不依賴 Mode 1 markdown。
- 對應 feedback：心法二（TypeScript 型別安全 ≠ 執行期安全）

## L5：完整心法集輸出比零散 debug 更有傳承價值
- 現象：今天踩了 10 個雷，每個雷都有根因 + 對策。
- 根因：系統級的「踩雷地圖」比個別 commit message 更容易被下一個築快速吸收。
- 下次：每跑完一個新模式的首次端到端，就整理一份 MODE_TO_MODE_LESSONS。一次對一次，不積累。
- 對應 feedback：memory 格式 v2（規則/Why/觸發信號）
