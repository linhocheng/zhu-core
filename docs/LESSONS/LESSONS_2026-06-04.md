# LESSONS 2026-06-04 · MACS Mode 2 首跑端到端

## L1：Cloud Run 有自己的 schema，Vercel lib 改了沒用
- 現象：`lib/llm/synthesis.ts` 有完整 `HybridSynthesisSchema`，但 synthesis 跑在 Cloud Run，Cloud Run 只認識 Mode 1。Recommendation 讀 `hs.dataAnchoredTruth.summary` crash。
- 根因：synthesis 超過 300s（兩次 bridge call），Vercel 擋不住，搬去 Cloud Run。Cloud Run 是獨立 source，Vercel lib 改動對它完全無效。
- 下次：改 lib/pipeline 之前先問「這個 worker 實際上跑在哪？」Vercel lambda < 300s / Cloud Run 長任務，兩邊要分開改。
- 對應 feedback：後台改動必同步前台（harness）+ 記憶會說謊（改了 Vercel 以為改完了）

## L2：每個 route.ts 都要獨立讀 strategyMode，不會自動繼承
- 現象：lib/pipeline 全面 hybrid 化，但 roadmap/partner-review/export 三個 route 都沒讀 `c.strategyMode`，仍然走 Mode 1 路徑。
- 根因：route 是 harness，pipeline lib 是邏輯層。harness 不讀 mode，pipeline 就永遠跑預設（market_evidence）。
- 下次：改完 lib/pipeline 之後，每個 route 必須過一遍 checklist：「有讀 strategyMode 嗎？有往下傳嗎？readArtifact 的 generic type 改成 union 了嗎？」
- 對應 feedback：改 lib/pipeline 必同步改 app/api/workers

## L3：needs_repair SOP — 從失敗的 stage 重進，不是從 brief_intake
- 現象：本想從 brief_intake 重打，但 synthesis 失敗只需要重打 synthesis，多打前段是浪費。
- 根因：之前的 SOP 寫「reset 到 brief_intake」，適合完全重來，但 mid-pipeline 失敗有更精準的入口。
- 下次：看 status 判斷從哪裡重進：synthesis_running → 重打 synthesis；roadmap_running → 重打 roadmap。
- 對應 feedback：needs_repair SOP 更新進 MODE1_TO_MODE2_LESSONS.md

## L4：Mode 1 only 函式（assembleDeliverables）必加 mode guard
- 現象：export worker 呼叫 `assembleDeliverables`，裡面讀 `recommendation.options.find()`，hybrid 沒有 `options` → crash。
- 根因：assembleDeliverables 是早期 Mode 1 寫死的，沒預留 hybrid 路徑。
- 下次：任何讀 `recommendation.options`、`synthesis.keyInsights` 等 Mode 1 專屬欄位的地方，先加 mode guard。Hybrid 的報告由 `runReportBuild` HTML 路徑處理，不依賴 Mode 1 markdown。
- 對應 feedback：心法二（TypeScript 型別安全 ≠ 執行期安全）

## L5：完整心法集輸出比零散 debug 更有傳承價值
- 現象：今天踩了 10 個雷，每個雷都有根因 + 對策。
- 根因：系統級的「踩雷地圖」比個別 commit message 更容易被下一個築快速吸收。
- 下次：每跑完一個新模式的首次端到端，就整理一份 MODE_TO_MODE_LESSONS。一次對一次，不積累。
- 對應 feedback：memory 格式 v2（規則/Why/觸發信號）

# 下午場 · MACS 框架重構(Phase 0-2)+ Adam「先看現場挖 WHY」

## L6：重構前先挖「為什麼設計成這樣」，承重牆不能 naive 搬
- 現象：要把 Cloud Run 三棒(research/cross_review/synthesis)收進框架，差點 naive 規劃搬執行。
- 根因：Cloud Run 分裂是 timeout 承重牆(Vercel 300s + CF 130s/100s,多次連續 bridge call 必撞)，不是隨意切。而且當初已用「只 vendor 穩定 soul/base、易變 roleFraming 從 DB 即時拉」打贏 drift——naive 全 vendor 會把 drift 重新引爆。
- 下次：重構動手前，先讀 code 註解挖設計 WHY。Adam 一句「先看現場，設計成這樣一定有原因」擋掉一次大錯。
- 對應 feedback：決策前先問解根因還是繞道 / 記憶會說謊先看現場

## L7：export 洞的根更深——analysis route 從沒傳 mode（綠勾說謊第二次）
- 現象：要修「各分析師章節是 Mode 1 格式」，挖到根是 analysis/route.ts 從沒讀 strategyMode、沒傳 mode → hybrid 案一直跑 Mode 1 分析，hybridMemo 被算出來就丟。
- 根因：心法四(改 lib 必同步改 route)沒落實。runAnalysis 有 hybrid 分支但 route 沒接。Mode 2 e2e「過」只代表跑完，不代表每層跑對 mode。
- 下次：e2e 綠 ≠ 每層 fidelity 對。驗證要驗「每層真的用對 mode」，不只「跑到 done」。
- 對應 feedback：記憶/lastwords 會說謊 + 心法四

## L8：查資料不分 mode，分模式的是「拿到原料怎麼分析」
- 現象：原本照企劃要做「Mode 2 特製六分類查資料」。
- 根因：現場確認 handleResearch 從不讀 strategyMode——查資料是純收集。六分類不是 Mode 2 專屬，是「把原料收豐富」，三模式共享；mode 智慧全在 analyze 之後。Adam 一問點破。
- 下次：要加 mode 分支前先問「這一步真的因 mode 而異嗎，還是只是收集得更全」。能共用就別分叉。
- 對應 feedback：破刀（好架構是刪出來的）

## L9：框架不搶執行，搶 schema 單一源（A+）
- 現象：糾結框架要不要 vendor 進 Cloud Run。
- 根因：Cloud Run 執行搬不動(承重牆)。真正的痛是 schema 被定義兩份(心法八 drift)。框架該擁有 schema 單一源、兩邊 import，執行留原地、易變 prompt 留 DB。
- 下次：跨執行環境的重構，先分「什麼搬不動(執行)」vs「什麼該收單一源(schema/契約)」。
- 對應 feedback：兩份即是零份（破索）

## L10：本機 LLM replay 前必查 bridge 是否真的生效（天條）
- 現象：想本機跑 runRecommendation 驗框架,快又省 cycle。
- 根因：.env.local 的 BRIDGE_ENABLED="true" 帶引號，--env-file 是否去引號沒把握；賭錯 getLLMClient fallback 燒付費 key。
- 下次：本機任何會打 LLM 的 replay，先確認 bridge 真的會被選中(去引號、enabled、URL/secret)。沒把握就走伺服器(prod bridge 已證 $0)。
- 對應 feedback：開發不燒付費 API key / bridge-first

## L11：pilot 證明分等級——by-construction vs runtime-log
- 現象：recommend pilot 想看 outputSummary 證明走框架，但 completeWorkerRun 只存 status，outputSummary 沒落地 Firestore。
- 根因：證明等級要分清。這次拿到的是「部署含該分支的工作樹 + hybrid 無條件走框架 + 有效輸出 + 管線推進」=by-construction，不是 runtime 一行 log。鏈密但非直接觀測。
- 下次：要 runtime 證據就要可觀測性落地;一次性 marker 是 over-engineering，該在全遷時建正式 instrumentation(Phase 3)。
- 對應 feedback：沒跑過=沒收完 / 綠勾說謊（誠實標註證明等級）
