# LESSONS 2026-05-31 · MACS 平台從零建置（ANEWS 概念 → AI 顧問公司）

## L1：bridge（Max）不支援 tool_use，結構化輸出要退回 <result> JSON + Zod
- 現象：synthesis 第一版用 `tool_choice: {type:"tool"}` 強制結構化輸出，bridge 回來的 content 裡沒有 tool_use block，直接拋「model did not emit tool」。
- 根因：bridge VM 那層（claude CLI gateway）不支援 server-side tools / tool_use——這條 ANEWS 的 `cloud-run/source-worker/src/index.ts:13` 早寫了註解「bridge doesn't support tools/web_search」，我沒先讀就盲套 SDK 慣用法。
- 下次：走 bridge/Max 的結構化輸出一律用 ANEWS 成熟做法——prompt 要求 JSON 包在 `<result></result>`、regex 抽取、JSON.parse、Zod 驗證，範例給【完整欄位】（省略欄位會讓 LLM 自創 key → Zod undefined）。只有 web_search 那種 server-side tool 才走 getLLMClientDirect（API key）。
- 對應 feedback：feedback_20260502_awareness（API 盲猜 vs 先讀源碼）、feedback_blueprint_schema_example。

## L2：動態 fan-out 的正解 = barrier + completion_log 為真相
- 現象：MACS 的 workstream 數量在 issue-tree 跑完才知道，不能像 ANEWS 靜態 DAG 預建 dependsOn（會踩 dispatcher 引用不存在節點永久卡死）。
- 根因 / 做法：materializer 動態建 N + barrier 動態收 N。關鍵不變式——completedCount 只在「寫 completion doc 的同一 Firestore txn」內變動（兩者原子，不會分裂）；enqueue synthesis 在 txn commit 之後（Cloud Tasks 不能進 txn）；正因如此 reconciler 從 nice-to-have 升為【必需】，兜 commit 後 enqueue 遺失/崩潰，從 completion_log 重推真相。
- 下次：任何「動態分支數 + 收斂」都用這個 pattern，不要讓下游 dependsOn 還不存在的節點。
- 對應：sync-truth-principle（兩份即零份）、獨孤九劍狀態機三問（status/lease/attemptId、taskId 確定性）。

## L3：一吋蛋糕——先燒最高風險的兩塊，不先鋪滿 12 個 worker
- 現象：12+3+1 的大架構，誘惑是從 Phase 0 一路線性蓋到 Phase 7。
- 做法：先單獨驗「synthesis 質感」（go/no-go 閘門，假 memo 餵 bridge，看能不能 derive reframe）+「orchestration 正確性」（stub 對真 Firestore 跑 21 斷言，零 LLM 成本）。這兩塊是「蓋下去才知道行不行」的真風險，而且都能用便宜方式單獨驗。確認口味造型再量產，後面 ANEWS 換皮的 worker 風險低放後面。
- 結果：synthesis go=GO、orchestration 21/21，兩個閘門都過才往下，沒有白蓋。
- 對應：feedback_one_inch_cake_mvp、破槍式（沒端到端跑過的等於不存在）。

## L4：記憶/子代理摘要會說謊，靠去現場核對抓到兩個錯
- 現象：① 子代理摘要說 ANEWS 全用 claude-haiku-4-5；現場 `bridge.ts:66` 是 `claude-sonnet-4-6`。② 記憶（含 LAST_WORDS）說「anews-platform 不是 git repo」；現場 `.git` 今天才建、已是 git。
- 根因：子代理摘要是二手、記憶是過去快照，都不是現在的真相。
- 下次：對位正確的前提是回現場讀真檔（grep 真值、ls 真目錄），尤其在「照抄既有架構」時，抄的是現在的它不是記憶裡的它。
- 對應：feedback_memory_can_lie、feedback_diagnosis_verify_before_write。

# LESSONS 2026-05-31（晚場）· MACS 套部門魂 + HTML 報告 + 後台 + 部署踩雷

## L5：抄基建要抄「部署拓撲」，不只抄程式碼——web_search 必須 Cloud Run，放 Vercel 會 timeout→重試風暴燒 key
- 現象：MACS 部署後跑第一個真案，research（web_search）放在 Vercel route，撞 Vercel 函式 timeout，Cloud Tasks 預設無上限重試 → 同一個 research task `dispatchCount=9`，每次都燒 web_search API key（≈9 倍冤枉錢），Adam 喊停才發現。
- 根因：我「抄 ANEWS 80% 基建」時只抄了程式碼，沒抄**部署拓撲**。ANEWS 的鐵律寫在 `anews-platform/app/api/workers/orchestrate/route.ts:103` 註解：「Source worker runs on Cloud Run (SOURCE_WORKER_BASE_URL) to escape Vercel 300s limit」——web_search + 長生成一律放 `cloud-run/source-worker`，Vercel route 用 `enqueueTask(..., overrideBaseUrl=SOURCE_WORKER_BASE_URL)` 指過去。`cloudTasks.ts` 我抄進來時連 `overrideBaseUrl` 參數都帶了，卻沒用它，research 還是 enqueue 到 WORKER_BASE_URL（Vercel）。
- 下次：「對位正確」要包含 runtime/hosting 對位，不只 API/schema 對位。照抄成熟系統時，先盤「哪些 worker 不在 Vercel、為什麼」，那些例外往往是前人踩過坑換來的。對應 feedback：feedback_one_inch_cake_mvp（沒端到端跑過＝不存在）、reference_vercel_300s_lambda_limit、bridge_silent_fallback_double_burn（燒錢的雷要先防）。

## L6：上線前每個 Cloud Tasks 佇列必設 maxAttempts 上限，否則單點失敗 = 重試風暴放大器
- 現象：research 失敗本身只是「一條 worker 掛了」，但佇列預設 retry 無上限，把一次失敗放大成 9 次燒錢。
- 根因：`gcloud tasks queues create` 沒帶 `--max-attempts`，吃預設（很大）。ANEWS 因為長任務不在 Vercel 所以沒爆，但 MACS 把雷點（research）留在 Vercel + 佇列無上限 = 雙重失誤疊加。
- 下次：建佇列即設 `--max-attempts=3 --max-backoff=...`，把任何 worker 失敗的 blast radius 封在個位數。止血永遠先做（暫停佇列：`gcloud tasks queues pause`，可逆）。
- 對應：feedback_no_paid_api_without_consent（天條：燒錢的路徑要有護欄）、獨孤九劍狀態機三問。

# LESSONS 2026-05-31（深夜場）· MACS research 上 Cloud Run + 端到端首跑通

## L7：卡在 needs_repair 的真因不是燒錢的 worker，是觀察層自己同步拋錯
- 現象：把 research 搬上 Cloud Run（532s 單跑、單 dispatch、零重試燒 key，錢 bug 確實修好）後重跑臻品案，case 仍被打成 `needs_repair`。直覺以為又是 research/analysis 出事。
- 去現場（用 ADC + firebase-admin 讀 worker_traces）才發現：全部 LLM 階段（framing→issue-tree→research→analysis×4→synthesis→recommendation→roadmap→storyline→partner-review）trace 都是 `ok`、deliverable 全寫好了。報錯的只有 **materialize / export** 這兩個「不呼叫 LLM」的 worker。
- 根因：`lib/workers/trace.ts` 的 `writeWorkerTrace` 號稱 fire-and-forget（`.add({...}).catch()`），但非 LLM worker 回 `llmUsage: undefined`，Firestore SDK 在 `.add()` **同步驗證階段就拋錯**，`.catch()` 只接 promise rejection、攔不到同步拋錯 → 例外逃進 harness 錯誤路徑 → 誤分類 LLM_ERROR、bump repairAttempts、把做完工作的健康 case 打成 needs_repair。觀察層（trace/log）反過來拖垮工作層。
- 修法（根因）：寫入前剝除所有 undefined 欄位 + 外層同步防護，讓 trace 失敗只記 warn、不外溢業務流程——這才是讓函式真正符合它自己宣稱的「never block or fail the pipeline」。
- 下次診斷起手式：case 莫名 needs_repair 但 deliverable 都在、所有 LLM 階段 trace 都 ok → 直接去查「無 LLM 的 worker」（materialize/export）的 trace/log 寫入，那裡最容易藏 undefined。
- 對應：破綻三處（邊界模糊：觀察層不能讓工作層當掉）、feedback_memory_can_lie（先去現場讀 trace 不要靠直覺猜是 research）。

## L8（待觀察）：export 一次 schema-invalid blip，被 maxAttempts=3 韌性接住
- 現象：重跑 export 時 05:02 出現一次 `schema invalid（expected string）`，90 秒後重試一次就過了，case→done。
- 意義：①maxAttempts=3 護欄發揮了該有的韌性（單次 blip 不致命）。②但根因未查，可能偶發重現——標記為待觀察，不默默當沒事（feedback_surface_technical_debt）。
