# LESSONS 2026-06-18 · UDN NEWS UI 修繕

## L1：nav stale closure — 切換專案後 URL 帶著舊 ticket
- 現象：openTask 切換到新專案，URL 仍帶舊的 work_order_id
- 根因：`nav("kanban")` 沒傳 `workOrderId: ""`，nav 的 closure 讀到舊 activeTicket
- 下次：切換任何上下文（project/user/session）時，一律明確傳 `{ projectId, workOrderId: "" }` 並清 activeTicket state
- 對應 feedback：backend_client_must_sync（改一邊不夠，state 和 URL 要同時清）

## L2：製圖風格 UI 先上，後端沒接 = 假功能
- 現象：08 頁加了風格選擇器，但 executeImageMaker 完全不讀 image_style
- 根因：UI 和 backend 兩邊要同 session 同步改，不然是假中台
- 下次：新增 operator 輸入欄位時，馬上問「後端讀了嗎」，確認 data flow 兩端都通才算完成
- 對應 feedback：interface_blood_vessel_check（介面建完強制問血管接通了嗎）

## L3：React useCallback 的 stale closure 是 URL 污染的根源
- 現象：`nav` 是 useCallback，依賴 workflow.project.project_id；但切換過程中舊 workflow 還在 state，導致新的 nav call 用舊 projectId
- 根因：async 函數裡 React state 不是同步更新，要的新值要直接傳進去不能靠 closure
- 下次：任何 async 流程裡呼叫 nav，一律顯式傳 `projectId`，不依賴 closure 捕獲的 state

# 第二 session · StraTA 學習 + HD 進度收束

## L4：「寫註解」不是反射加註解，先核既有再決定缺什麼
- 現象：Adam 要「HD 專案寫註解跟目前進度」，我第一反應差點直接補一堆註解
- 根因：全局天條是「默認不寫註解，只在 WHY 非顯而易見時寫」。兩個決策點（Tone 來源、重疊半圓配色）上一輪已有 WHY 註解
- 下次：收到「加註解」先 grep/Read 既有註解，判斷真正缺的是什麼（這次缺的是進度檔不是行內註解），別把 noise 當交付
- 對應 feedback：deterministic_work（過度註解＝認知負擔）+ ui_conform_no_patch 的精神（先看現場慣例）

## L5：StraTA 是「機制證明了直覺」不是新銀彈
- 現象：讀完 StraTA 容易興奮想全套搬 RL 訓練
- 根因：我們走 bridge/API 不自訓，RL 半部用不到；真正可搬的只有編排層三模式（Top-δ評分/最遠點多樣性/校準自審）+ plan→condition→execute = 三段公式
- 下次：讀前沿論文先分「機制證明既有信念」vs「真正新東西」，前者拿來當硬證據不是換劍
