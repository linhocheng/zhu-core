# LESSONS 2026-06-07 · MACS Mode 1 管線重構（Victoria/Marcus 中途 worker + issue-tree 雙階段）

## L1：export 純渲染靠 preBuiltChapters bypass，不是刪掉 fallback

- 現象：export 原本 inline 呼叫 Cloud Run structure-analysis + integrate-chapters（同步 HTTP），新管線這兩步已在 pipeline 跑完並存 artifact。
- 根因：export 不知道 artifact 存不存在，不能強制要求一定有（舊案沒有）。
- 下次：新增 optional `preBuiltChapters` 參數做快路徑，沒有時 fallback 舊路徑。新舊案都能跑，不破向後相容。
- 對應 feedback：改動必須保留舊案 fallback，不要假設所有案都走新路線。

## L2：issue-tree 雙階段——中間 schema 不含 workerType，配兵官才加

- 現象：原本一個 LLM call 做全部（問題分解 + workerType 指派），現在要拆成 Eric + 配兵官。
- 根因：兩個責任混在一個 prompt 裡，角色角色邊界不清，也無法分別控 token 預算。
- 下次：Eric schema 不含 workerType（EricIssueTreeSchema），配兵官拿 Eric 輸出當 user context，輸出含 workerType 的最終 schema。usage 加總回傳。
- 對應 feedback：確定性的分工（誰問問題/誰配兵）用結構保證，不靠 prompt 內「記得做兩件事」。

## L3：Cloud Run deploy 流程確認（build submit → run deploy 兩步）

- 現象：MACS research-worker 改動後要重 deploy，路徑是 gcloud builds submit + gcloud run deploy。
- 根因：Cloud Run 不像 Vercel 有 git hook，需要手動兩步。
- 下次：build submit 先拿到 image hash，deploy 指定 image，分開跑比較清楚進度。
- 對應 feedback：deploy 完要等 revision serving 100% 才算上線。

## L4：天條的「收斂點」打法——修 12 個呼叫點不如修 1 個共用閘門（esc 物件崩潰）

- 現象：export 炸 "e.replace is not a function"。根因是 LLM 把宣告為 string 的欄位（keyFindings）回成 `{finding,...}` 物件，物件流進 render 層被 `.replace()` 呼叫就炸。
- 用相同概念查 Mode 2/3：兩個框架（hybrid/report.ts、creative-lead/report.ts）有一狗票同類欄位（stopConditions / whatWorks / territoryName / actions…）直接餵 bullets/table/cards，全部沒有 data 層正規化（不像 Mode 1 有 flattenKeyFindings）。逐點修要改十幾處、跨三個檔，還會漏。
- 根因修（天條姿態）：三模式所有動態值都流經**單一收斂點** `esc()`。把 `esc(s: string)` 改成 `esc(s: unknown)`，在這一個確定性閘門 coerce（string / null→"" / object 取 .finding|.text|.claim 否則 JSON.stringify / 其他 String()）。一處上游走樣 → 降級成可讀文字，永不崩潰。一個 commit（v0.11.3.005）守住全部三模式。
- 心法：找「資料必經的最窄處」下手。不是把防禦撒在每個生產端，而是釘在唯一消費端。esc() 就是 render 層的咽喉——任何 mode 任何 block 的字串都得過它。
- 對應天條：LLM 輸出一律當不可信文字，**在邊界**用確定性程式 coerce/validate；不是在 prompt 拜託模型「回字串別回物件」。

## L5：丟棄式診斷腳本會無聲炸掉 production build（這次最大的隱形坑）

- 現象：Mode 1 export 一直 300s timeout、status 卡 "running"。表面看是 Cloud Run 慢。
- 真根因：Vercel 跑的是**舊 code**（沒有 preBuiltChapters 快路徑），因為從 v0.11.3.001 起每次 build 都靜默失敗——我自己丟在 scripts/ 的一次性診斷腳本 `_normalize_kf.mts` 有個 TS error（`workstreamId` 不存在於型別），`next build` 全綠不了，Vercel 部署一直是舊版。一個 throwaway 腳本擋了好幾個版本的部署，而我以為在 debug 管線。
- 修法：當下 `(ch as any)` cast 先過；永久修是 tsconfig `exclude` 加 `"scripts"`（v0.11.3.004），讓診斷腳本永遠不進 production typecheck。
- 心法：**「我部署了」≠「Vercel 跑我的新 code」**。改完沒驗到新行為真的上線前，先確認 build 綠。診斷腳本與 production 同一個 tsconfig 範圍 = 定時炸彈，第一天就該隔離。
- 對應 feedback：補一層（fix）後必跑完整 dry-run／真的看到新行為，不能假設第一層通了就是好了。
