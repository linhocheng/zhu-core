# LESSONS 2026-06-25 · UI display fallback ≠ state

## L1：`??` fallback 只管顯示，不寫進 state
- 現象：admin 後台 emotion 設成 neutral，存進 Firestore 還是 `{}`
- 根因：`<select value={vs.emotion??'neutral'}>` 讓 UI 顯示 neutral，但 `vs.emotion` 仍是 undefined；用戶沒有互動 dropdown 就送出 → state 沒有 emotion key → backend `typeof vs.emotion === 'string'` 為 false → 不存
- 下次：凡是 display fallback（`?? 'default'`），都要確認 state 初始化也有設定這個 default，不能只靠 JSX 的 `??`
- 對應 feedback：介面建完強制問「血管接通了嗎」

## L2：從 log 排除法縮小根因
- 現象：達賴語音變女聲，不確定根因
- 做法：依序排除：多 region？→ 只有 asia-east1。多 TTS 實例？→ 只有一個。WS fallback？→ log 全是 WS done，沒有 REST fallback。3a 插話？→ 走同一個 session.say()
- 最後留下的可能：emotion auto-inference + Firestore voiceSettings 沒存進去
- 下次：「突然變女聲」不一定是 runtime 問題，先查資料層（Firestore 欄位值）再看 code
- 對應 feedback：記憶會說謊，動手前先看現場

## L3：UI bug 修法——初始化時填預設，不是在 JSX 補 fallback
- 現象：想用 JSX `??` 補 default，但這只是 controlled component 的顯示值
- 下次：正確做法是在 `useState` 或 `setEditing` 初始化時就把 default 填進去，`{emotion:'neutral', ...serverData}` pattern

## L4：Harness 心法——進入 harness 不是成為 harness（第十八 session）
- 現象：SOP 每個 Phase 都是機械步驟，很容易照著跑忘了監造者視角
- 根因：ZHU_CONTEXT.md 寫的鎖死根因——進了 SOP 就失去監造者，變機器人
- 下次：每輪 REFLECT 問自己「我還在監造嗎」
- 對應 feedback：懂劍法跟真的用

## L5：閻羅拿摘要不夠——確定性驗收比 LLM re-judge 更可信
- 現象：前兩次呼叫閻羅都給摘要，說 CONTINUE；改用 python3 assert 直接 DONE
- 根因：摘要是壓縮版，閻羅再判一次 = 機率×機率
- 下次：審計類任務驗收用 python3 確定性程式，不是 LLM re-judge
- 對應 feedback：確定性的工作用程式不要丟 LLM

## L6：靜態掃描有天花板——Live 狀態要用動態工具確認
- 現象：M2（CR service 存活）和 L1/L2（v11 狀態）只能說「待確認」
- 根因：grep/diff 告訴代碼怎麼寫，Cloud Run 現況要 gcloud run services list
- 下次：審計任務起頭先分「靜態可知」vs「動態才知」，後者提前說清楚需要哪個工具
- 對應 feedback：標了風險不等於驗了風險
