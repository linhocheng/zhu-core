## 2. Firebase admin default app vs named app — `admin.apps.length` 是錯的 check

**症狀**：v0.0.0.008 deploy 後 6 個 cron 全 HTTP 500，body 空，Vercel logs streaming 看到：
```
Error: The default Firebase app does not exist
```
Auth 過了（不是 401），但 `admin.firestore()` 拿不到 default app。

**根因**：`src/lib/firebase-admin.ts` 用：
```ts
if (!admin.apps.length) { admin.initializeApp({...}); }
```
但 `zhu-vitals/firestore.mjs` 先 `admin.initializeApp({...}, 'zhu-vitals')` 註冊了 **named app** → `admin.apps.length` 已 ≥1 → 跳過 default app 初始化 → 之後 `admin.firestore()` 找 default 找不到，throw。

**解法**（v0.0.0.009）：
```ts
const defaultExists = admin.apps.some((a) => a?.name === '[DEFAULT]');
if (!defaultExists) { admin.initializeApp({...}); }
```

**衍生天條**：
> `admin.apps.length` 在「有人先註冊 named app」的環境下會說謊。
> 共存場景一律 `admin.apps.some(a => a?.name === '[DEFAULT]')`。

**已寫進 CLAUDE.md**：施工規範 → BUILDING_PROTOCOL v0.2 → 踩過的雷第 2 條。

## 4. IAM grant 後 propagate 延遲 — 不要當「不能」，要 poll

**症狀**：grant `roles/iam.serviceAccountTokenCreator` 給 `adam@dotmore.com.tw` 後立刻 `gcloud auth print-identity-token --impersonate-service-account=...` → `PERMISSION_DENIED iam.serviceAccounts.getAccessToken`，本能想「啊還是不行」要回去查 IAM。

**根因**：GCP IAM policy 寫進去到全 region propagate 完成有 10-30s 延遲。
**正解**：
```bash
until TOKEN=$(gcloud auth print-identity-token --impersonate-service-account=...) && [ -n "$TOKEN" ]; do
  sleep 10
done
```
約 30s 後就拿到了。

**衍生天條**：IAM PERMISSION_DENIED **立刻重試**，10-30s 後再 fail 才是真 fail。

**已寫進 CLAUDE.md**：BUILDING_PROTOCOL v0.2 → 踩過的雷第 4 條。

## 5. vendor 策略 + sha256 lock — 寫了 VENDOR.md 不等於有 lock

**狀態**：3 個 worker 早已有 VENDOR.md，但只記「Version vendored: 0.1.1 / 0.1.2」字串，沒有 sha256 lock。**今天才補進去 6 個檔案的 sha256 + 來源 commit hash**。

**意外**：補 sha256 過程中發現一段「看似有 drift 其實沒 drift」的虛驚 — 第一次 parallel `shasum` 輸出顯示 strategy-worker `bridge-call.mjs = 02b6...` 跟 zhu-core source 不同，第二次重算所有人都是 `a0e0...`。沒查到原因（可能 fs cache、可能 parallel tool 排程錯誤輸出），但**教訓**：sha256 結果靠單次 read 不安全，要重算驗證才能下結論。

**衍生天條**：sha256 lock 寫進去 = 凍住「這一刻」。下次改 source 一定要：
1. 改 `~/.ailive/zhu-core/zhu-vitals/src/`
2. cp 進 5 個 worker（含 bridge VM scp）
3. 在每個 VENDOR.md 重算 sha256 更新 lock
4. CI 之後可加 diff 兩邊 sha256 報警

**今天落地**：4 個 vendor 點（molowe-platform、strategy-worker、strategy-html-worker、bridge VM）都已寫 VENDOR.md w/ sha256 lock + source commit `495e2058`，`check-manifest.mjs` strict mode 強制存在。

## 5b. 校正：5/12「虛驚」其實是真 drift（5/13 補）

5/13 動 B3 (callBridge purpose 細分) 第一動作 = sha256 對賬，molowe vendor `bridge-call.mjs` = `02b69a04`，跟 5/12 第一次看到的一字不差。

5/12 第二次 shasum「全 a0e0」是錯判 — 可能 shell 排序撈到 source 路徑、或 parallel tool 輸出對錯了 row。當時我把對的當錯（第一次）、把錯的當對（第二次），收進 LESSONS 第 5 條叫「虛驚」。

**真相**：5/12 補 vendor 時 molowe 那份 `bridge-call.mjs` + `manifest.schema.mjs` + `manifest.types.d.ts` 三個檔 cp 漏了，VENDOR.md sha256 lock 寫成 source 的 `a0e0a9ff`，但事實檔還是 `02b69a04`（0.1.1）。**lock 跟事實不一致整整 24 小時**，期間 prod molowe 6 cron + 8 worker 全程跑 0.1.1 字串分支，沒人發現。

**衍生天條**：
- shasum 兩次不一致 → 先信第一次 + 重核「我這次打的是哪個路徑」，不要相信「重做就好了」
- 「VENDOR.md 寫 lock」≠「事實對齊」。lock 是宣告，事實要另外 shasum 對賬
- CI sha256 drift check（B1）此刻價值翻倍 — 24h 沒人察覺 = 沒守門哨

**5/13 落地**：cp 三個 drift 檔（bridge-call / manifest.schema / manifest.types.d.ts）+ wrapper `bridge.ts` 加 `await ... .text` unwrap（0.1.2 return object 不是 string）+ 8 caller 全傳 purpose。

## 6. bridge VM CJS 載 ESM — 動態 import 是唯一解

**狀態**：claude-bridge index.js 是 CJS（systemd service），要用 zhu-vitals 的 `withVitals`（純 ESM `.mjs`）。

**正解**：
```js
let _bridgeVitals = null;
async function loadBridgeVitals() {
  if (_bridgeVitals) return _bridgeVitals;
  const v = await import('./zhu-vitals/index.mjs');
  const intelManifest = (await import('./manifests/bridge-intel.mjs')).manifest;
  const discoveryManifest = (await import('./manifests/bridge-discovery.mjs')).manifest;
  _bridgeVitals = {
    trackedIntel: v.withVitals(intelManifest, runMoloweIntel),
    trackedDiscovery: v.withVitals(discoveryManifest, runDiscoveryCycle),
  };
  return _bridgeVitals;
}
async function runMoloweIntelTracked() {
  try { return await (await loadBridgeVitals()).trackedIntel(); }
  catch (e) { console.error('[vitals] fallback raw:', e.message); return runMoloweIntel(); }
}
```

`scheduleMoloweIntel`/`scheduleDiscovery` 改呼叫 `runMoloweIntelTracked`/`runDiscoveryCycleTracked`。lazy load + fallback to raw，第一次 tick 才 import，失敗不會弄壞既有 worker。

**踩過小雷**：function 宣告要 hoisted（沒用 `const` arrow function），不然 `scheduleMoloweIntel` 在 `runMoloweIntelTracked` 宣告前被呼叫會 `TDZ`。
