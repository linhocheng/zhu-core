# LESSONS 2026-05-25 · ANEWS DAG dispatcher + cold-start crash

## L1：getFirestore() 直接呼叫 = cold-start 定時炸彈
- 現象：orchestrate 部署後返回 HTTP 500 empty body（content-length: 0）
- 根因：`idempotency.ts` 用 `getFirestore()` 直接呼叫，不走 lazy `db` Proxy。Firebase app 未 init 時就爆 `app/no-app`。之前之所以能跑，是因為 warm start 時其他 route 已初始化 Firebase。Turbopack chunk 重組後 cold-start 路徑暴露了這個 bug。
- 下次：凡是 firebase-admin/firestore 相關操作，一律走 `db` Proxy（`lib/firestore/admin.ts` export 的那個），不直接呼叫 `getFirestore()`。
- 對應 feedback：fix_one_layer_dryrun_all（修完一層就要驗整條鏈）

## L2：shadow mode 驗證腳本是架構遷移的安全網
- 現象：DAG dispatcher 上線後，worker_traces 有 25 筆，但 workflow_nodes 只有 14 筆，diff=9
- 根因：blueprint/alignment/stitch 用 createHarnessWorker 但沒設 nodeType，shadow write 不觸發
- 下次：新 worker 接 createHarnessWorker 時第一步就補 nodeType，不是等 verify 腳本跑出 diff 才補
- 對應 feedback：surface_technical_debt（驗出來就立刻修，不默默繞過）

## L3：dispatcher fire-and-forget 在 serverless 不可靠
- 現象：reconcileIssue 如果 fire-and-forget，Vercel 可能在 response 送出後就終止 function
- 根因：serverless 函數生命週期和回應綁定，response 後的非同步 code 不保證跑完
- 下次：關鍵路徑（dispatcher poke）要 await，60s cron 才是 fire-and-forget 的正確位置
