# LESSONS 2026-05-22c · ANEWS 文章平台 + 後台大改版

## L1：Dashboard 大白話設計——技術狀態名要換成人話才有用
- 現象：後台顯示 `section_writing`、`blueprint_running`，Adam 看不懂要做什麼
- 根因：工程師習慣把狀態名直接拿來 render，沒有對應到使用者的心智模型
- 下次：任何面向人的 UI，狀態一律翻成動詞短句（「寫手撰寫段落中」），code 的 key 留在 title/tooltip
- 對應 feedback：介面血管原則——UI 做完不等於能用，還要確認使用者能讀懂

## L2：Kick 機制——卡死偵測要從 section 狀態反推，不能靠 pipeline 上報
- 現象：Cloud Tasks 沒繼續派發，pipeline 卡在 section_writing，5 篇文章全 blueprint_ready
- 根因：Cloud Tasks 可能 drop 或 timeout，section 的 draft_ready/qa_failed 狀態就是線索
- 設計：kick endpoint 從 Firestore section 狀態推算卡點，直接打 orchestrate 事件重啟——不依賴 queue 本身
- 對應：solve_root_not_symptom（不是重試 queue，是補打遺漏的事件）

## L3：一期鎖要在 API 層擋，不能只靠 UI 灰底
- 現象：原本沒有任何限制，可以無限建期刊
- 根因：前台 disabled 按鈕是視覺提示，不是防護；curl 或誤操作都能繞過
- 下次：業務規則一律在 API 層 enforce（`not-in ["done","failed"]` 查詢）；UI 只是友善提示

## L4：Role prompt hardcode 搬 Firestore，要加本地 TTL cache
- 現象：每次 section-write 都要讀一次 settings/roles
- 解法：module-level cache + 60s TTL，只有第一次或過期才打 Firestore
- 下次：任何「每次 LLM call 都要讀」的設定都加 TTL cache
