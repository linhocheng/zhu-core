# LESSONS 2026-06-24 · Task Harness 建置 + 工具卡死復盤

## L1：進入 harness 和成為 harness 是兩件事
- 現象：上一個築說「現在建 skill 檔」然後什麼都沒留下，session 結束就消失了
- 根因：太深入執行模式，失去監造者視角——變成 SOP 的零件不是使用 SOP 的人
- 下次：觸發 harness 時監造視角全程保留，goal 有問題立刻說出來
- 對應 feedback：feedback_framework_vs_reflex

## L2：確定性工作用程式，CB 觸發不靠 LLM 判斷
- 現象：設計 CB2 時差點用自由文字比對 blocker 是否相同
- 根因：LLM 每輪措辭不同，字串比對必失靈
- 下次：blocker_key 用枚舉六個固定值，len(set(last3))==1 才是確定性觸發
- 對應 feedback：feedback_deterministic_work_belongs_in_code

## L3：工具 internal error 是偶發的，先重試不要找規則
- 現象：mkdir 和 Write 第一次都 internal error，重試就過了
- 根因：Claude Code 工具層偶發抽風，不是 deny 規則也不是 hook 問題
- 下次：遇到 internal error 先重試一次，主動報告「我卡住了」不是靜默等待
- 對應 feedback：feedback_silent_failure_absent_log

## L4：Bridge auth 每個系統不同，要記
- Task Harness：x-api-key: $BRIDGE_SECRET + bridge-direct.soul-polaroid.work
- ANEWS：Authorization: Bearer $BRIDGE_SECRET + bridge.soul-polaroid.work

## L5：標了斷路器不等於驗了斷路器
- 現象：設計三個 CB 後 Adam 要求模擬觸發
- 根因：設計在紙上不等於可觸發，要用假資料實際跑腳本驗
- 下次：任何防禦機制建完立刻用最小資料集驗觸發
- 對應 feedback：feedback_flagged_risk_must_be_verified
