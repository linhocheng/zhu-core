# LESSONS 2026-05-25b · ANEWS P3 情報官 + source worker 靜默救場

## L1：catch fallback 比 throw 更危險
- 現象：Cloud Run source worker parse 失敗時，catch 存了一筆 `[parse error]` 假 fact，article 仍被標 `source_ready`，pipeline 繼續跑，最後 article=done 但零內容
- 根因：fallback 用意是「不讓整條線斷」，但實際上是把壞資料往下游推，downstream 全部靜默接受
- 下次：沒把握的 parse → 一律 throw，讓 retry 機制接手；fallback 只能是「降等輸出」不能是「假裝成功」
- 對應 feedback：`feedback_silent_failure_absent_log.md`

## L2：contract 只查欄位存在 ≠ 查內容有效
- 現象：sourceContract 查 `!d.sufficiency`，sufficiency 是 `{sufficient: false}` 仍通過合約
- 根因：object truthy check 和 field validity check 是兩件事
- 下次：合約要查業務語義（`sufficient === true`），不只查結構存在
- 對應 feedback：無現有 feedback，這是新的

## L3：pipeline 末端沒有底線才能空殼 done
- 現象：所有 section qa_blocked → sections_all_done → stitch（0 sections）→ polish → export → article=done，無內容
- 根因：`enqueueNextWritableSection` 把「找不到可寫的節」和「全被 alignment 封鎖」都當成 sections_all_done
- 下次：寫入計數（writtenCount）在 loop 裡加，0-section 走 needs_repair 不走 stitch
- 對應 feedback：`feedback_solve_root_not_symptom.md`

## L4：max_tokens 偏低是隱性的截斷風險
- 現象：stitch=1500, polish=1000 在 4 段文章時可能不夠，但沒有顯式報錯
- 根因：token limit 觸發後 LLM 靜默截斷 JSON，parse 才炸
- 下次：生成長文/JSON 的 worker 一律先設 8192，觀察再調低；短 JSON 設 2048 保底
