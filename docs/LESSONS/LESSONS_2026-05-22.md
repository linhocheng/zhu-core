# LESSONS 2026-05-22 · ANEWS S3 QA + Stitch + Polish + Export 全通

## L1：qa_blocked 的 skip 邏輯要在 worker 層做，不能在 orchestrator callback 做
- 現象：qa_blocked auto-skip 部署後，舊 blocked sections 沒有 skip，pipeline 卡死要手動 kickstart
- 根因：skip 邏輯在 orchestrator `section_qa_failed` handler 裡，但舊版 section-qa 直接 update status 沒有 callback → orchestrator 永遠不知道 blocked 了
- 下次：qaAttempts 計數 + blocked 判斷全部在 section-qa worker 做完，直接 enqueue 下一段，不走 orchestrator callback
- 對應 feedback：solve_root_not_symptom（symptom = 手動 kickstart，root = callback 不對稱）

## L2：Storage URL 可能帶換行符，必須 .replace(/\n/g, "")
- 現象：export worker 讀 finalMarkdownUrl 失敗，filePath 多了 `\n` 導致 Storage 找不到檔案
- 根因：Firestore 存 URL 時中間有換行（stitch worker 拼 gs:// + path 時換行混入）
- 下次：寫入 Storage URL 前統一做 `url.replace(/\n/g, "")`，讀取時也加防護
- 對應：Pipeline 防禦性編程——外部資料進來就清洗

## L3：QA Haiku 審自己生的稿太嚴格（qa_blocked 比例 50%+）
- 現象：主文 8 段有 5 段 qa_blocked，sub-article 也有 1-2 段 blocked
- 根因：LLM 生的稿 word_count check（實際字數達目標 80%）+ no_unsupported_claims 這兩項太容易 fail
- 下次：word_count 門檻降到 60%，或直接移除 no_unsupported_claims（long-form 深度文本斷言難以逐一佐證）
- 對應：一吋蛋糕——先跑通再調品質，不要一開始就卡在 QA 嚴格度

## L5：`language="zh"` 從第一個 commit 就存在，不是新引入的 bug
- 現象：Adam 說馬雲以前能聽中英文，懷疑是聲紋識別改動所致
- 根因：`git show a77341e:agent/realtime_agent.py` 驗證，`language="zh"` 在最早 commit 就有；記憶說謊，不是聲紋識別引入，而是從未支援過雙語
- 下次：越具體的「以前可以，現在不行」先查 git blame，不跟著記憶方向跑；修法是 `detect_language=True`（commit c778556），需 Cloud Run redeploy 才生效
- 對應 feedback：`feedback_memory_can_lie.md`

## L6：知識庫 Gemini summary 造成 Vercel 120s timeout
- 現象：上傳知識庫項目後卡住
- 根因：POST /api/knowledge 每筆都呼叫 callGemini() 同步等 1-3s，14+ 項堆疊直接超時
- 下次：Vercel lambda 路由裡 LLM call 要估乘積時間，超 100s 就必須異步或移除
- 對應：`reference_vercel_300s_lambda_limit.md`

## L4：全自動模式的「人工閘門放最後」要從架構上確保 pipeline 不卡
- 現象：`awaiting_review` 後需要手動 fire `review_approved`，全流程測試要手動補
- 根因：orchestrator 設計是「人工閘門阻斷」，但 Adam 想要的是「全自動跑到底，最後有個確認鈕」
- 下次：T9 Dashboard 審核按鈕是接下來第一件，做完才算真正「全自動到底」
- 對應：介面血管原則——功能做完但沒有 UI 接口 = 血管沒通
