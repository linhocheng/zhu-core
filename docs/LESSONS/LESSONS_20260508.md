## 1. AI 多層 prompt pipeline：結果不對先讓真相可見再診斷

**症狀**：vivi 生圖背景全黑，加「明亮」brief 也壓不住。

**第一次猜（F1）**：以為是 gemini 模型版本問題，升級 model → 還是黑。
**第二次猜（F2）**：以為是 prefix／soul 問題，往 prefix 看 → 命中。

**根因**：shun-001 的 `imagePromptPrefix` 在 Firestore 寫死了「dark background, chiaroscuro lighting, ...」，每張圖都串這串在最後。靈魂後台表面上能編，實際看不見 prefix 欄位的存在感，brief 寫多亮都被蓋過。

**真因不是 prefix 寫死本身**，是**沒人能看到「真正送進 Gemini 的 final prompt」**。所有人都只看自己的 brief，不知道 pipeline 後面被加了什麼。

**血管接口**：
- 寫回真相到 DB：`output.geminiPrompt / imagePromptPrefix / refsUsed`（dot notation 不污染主流程）
- UI 燈箱顯示「真相鏈」：`brief → prefix → final prompt → image`，逐層對賬
- 衍生天條：「應該是 X 的問題吧」連續兩次都沒中 → 立刻停手，先把每層真相外部化，不再用直覺

**已寫進 memory**：`skill_ai_pipeline_blackbox_debug.md`

## 2. CRON 401 是「靜默失敗」最完美樣本——schedule 對得上就是它

**症狀**：molowe pipeline 從 5/7 晚上之後完全沒新 published，但 dashboard 看起來正常。

**排雷**：
1. 檢查 vercel logs `--no-follow --since=10m --json` → 撈到一堆 `*/5 * * * *` 觸發 cron 全部 401
2. 對 schedule：cron tick 時間跟 401 時間秒對秒對齊，**這個對齊本身就是診斷**
3. 根因：`vercel cron` 帶 `Authorization: Bearer $CRON_SECRET` header，但 production env 沒設 `CRON_SECRET` → route 的 `authCheck()` 拒絕

**修法**：
```bash
KEY=$(openssl rand -hex 32)
printf '%s' "$KEY" | npx vercel env add CRON_SECRET production
npx vercel --prod --yes
```

**血管接口**：
- Vercel cron 必設 `CRON_SECRET`，否則 silent 401
- 排靜默失敗用「缺席的 log」對齊「應該存在的事件時刻表」（呼應 `feedback_silent_failure_absent_log`）
- `printf` 不用 `echo`（呼應 `feedback_secret_manager_printf` — echo 留 \n 進 header 會炸）

## 3. GEMINI key 被 Google revoke 是因為「明文進 git memory」

**症狀**：CRON_SECRET 修完後，content 仍 fail，error 換成：
```
gemini 403: Your API key was reported as leaked. Please use another API key.
```

**根因**：之前重建 `ai.zhu.migrate.plist` 時，把 GEMINI_API_KEY 寫進 `~/.ailive/zhu-core/memory/reference_zhu_migrate_plist_keys.md`（明文）→ commit → push → Google 巡到 → revoke。

**修法**：
1. 開新 key → curl `generativelanguage.googleapis.com/v1beta/models?key=$KEY` 驗 200
2. `vercel env rm GEMINI_API_KEY production` + add 新 key + redeploy
3. 把 10 篇 fail 的 content 重置回 pending／drafted（依失敗 stage）
4. 消毒 memory 檔：明文 key → `AIzaSy****REDACTED****`

**衍生天條**：
- API key 永遠不入 git，包括 memory／lessons／README／plist commit
- 即使是「私人 repo」也不寫，因為記憶會被推到 zhu-core 公開倉庫
- 寫範例 plist／env 時用 `<REDACTED>` 或 `<from_secret_manager>` 佔位

## 4. 監督層砍稿不是 bug 是 feature

**現場**：cron 修完後跑第一輪，pipeline 沒 publish 而是 `verdict=KILLED_AFTER_REWRITE`，原因「『不是預言是鏡子』框架已重複三次」。

**第一念**：「監督太嚴吧？」
**校準**：看歷史 failed 的 4 篇監督層 reason，全都是「框架重複／沒下一步／未提供新資訊」 → **是內容問題不是技術問題**。三層 AI 編輯部設計的目的就是擋這種稿。

**血管接口**：
- 區分技術失敗（writer stage / 403 / 500）vs 內容失敗（editor stage / KILLED_AFTER_REWRITE）
- 後者出現代表編輯部在做事，不要動編輯部，去看題材池
- 中午我以為「pipeline 壞了」，下午發現「編輯部正常運作只是不滿意內容」 → 別把標準誤認為斷路
