# LESSONS 2026-06-08 · MACS 報告篇幅旋鈕一處接通三模式

## L1：收斂點打法不是萬靈丹——要先盤點「繞過收斂點的旁路」

- 現象：把篇幅 directive 注入釘在 `callStructured`（結構化輸出唯一咽喉）後，以為三模式全收。實際上有兩條旁路會漏：
  1. **Mode 1 analysis 是 raw `llm.messages.create`**（prose 輸出，不走 callStructured）→ 不補就永遠收不到 directive。
  2. **Mode 3 creativeLead 原本自己也注入過一次** → callStructured 再注入就變雙重注入。
- 根因：收斂點只覆蓋「流經它的」producer。釘咽喉前要先 grep 出所有同類 producer，分兩種——走咽喉的（自動收）vs 繞過咽喉的（手動補）vs 已自己做的（要拆掉避免重複）。
- 下次：釘收斂點前先列「producer 清單 + 各自路徑」，標明哪些流經、哪些旁路、哪些已自理。三類都處理完才算收斂。
- 對應 feedback：`feedback_defend_at_convergence_point`（補一條但書：收斂點要配合旁路盤點）

## L2：UI 旋鈕只接到一個 mode = 血管不通的假中台

- 現象：篇幅三檔（精簡/標準/深入）UI 存在、設定有寫進 Firestore、文案寫「統一控制各內容步驟」，但實際上只有 Mode 3 讀它。Adam 跑的 Mode 1/2 + Cloud Run 章節（最大宗文字）完全沒接 → 拉桿是裝飾品。
- 根因：旋鈕建立時只接了當時在做的 mode，沒回頭盤點「所有消費端」。典型 interface_blood_vessel_check 三題沒問完：誰讀／何時讀／沒讀怎樣。
- 下次：任何「統一控制」型設定上線前，列出宣稱涵蓋的全部消費端逐一驗證接通，別讓文案承諾跑在管道前面。
- 對應 feedback：`feedback_interface_blood_vessel_check`、`sync-truth-principle`（後台必同步前台）

# LESSONS 2026-06-08 夜 · MACS complete-B route 泛型化 + Mode 4 e2e

## L3：「零 route 改動」的承諾，只跟最後一道 mode 耦合一樣真

- 現象：10 條 worker route 全改成 framework 驅動（`isVercelNative` / `stage.writes` / `buildReport`），tsc 綠、build 綠、M1/M2/M3 全 e2e 過。Mode 4 第一次跑就在 issue-tree 炸 `needs_repair`（repair=3 耗盡）。
- 根因：issue-tree 把 territory 轉 workstream 的 **compat 映射層**還藏著一行 mode 專屬欄位讀取——`hypothesis: t.coreEmotion`、`requiredEvidence: [t.worldview]`。那是 `CreativeTerritory`（Mode 3）的欄位；`ProposalTerritory`（Mode 4）叫 `emotionalCore`、根本沒 `worldview` → `undefined` → Firestore 拒寫。整條 route 99% 泛型化，就這一行把 Mode 4 釘死。
- 下次：泛型化要泛到葉節點。**compat / adapter / 映射層是 mode 耦合最後的藏身處**——它長得像「無害的資料搬運」，最容易在 review 時被放過。檢查法：凡讀 `t.某欄位`，問「這欄位是所有 mode 共有，還是某一個專屬？」共有才能讀。正解是只讀共享的 `territoryId`/`territoryName`，其餘給安全佔位（下游 analyze 本來就從 artifact 依 territoryId 重讀完整 territory，這些欄位是純 fan-out carrier，從不被消費——**先驗證下游不消費才敢簡化**，不是塞 fallback 繞過）。
- 對應 feedback：`feedback_solve_root_not_symptom`、`feedback_defend_at_convergence_point`
