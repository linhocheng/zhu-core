# LESSONS 2026-05-23c · ailive 即時語音 Soniox 換裝 + on_disconnected async 修法

## L1：asyncio.ensure_future + to_thread 被 framework executor shutdown 打穿
- 現象：改用 ensure_future(_async_on_disconnected) + 全部 to_thread 後，insights/promise/profile/cost 都報 "Executor shutdown has been called"
- 根因：LiveKit framework 在 on_disconnected 返回後立刻關 asyncio thread pool executor；後續 to_thread 呼叫全失敗
- 下次：asyncio.to_thread 不是 "把 sync 變 async" 的萬能解，它依賴的 executor 的生命週期受 framework 控制；脫離 framework 的生命週期要用 threading.Thread
- 對應 feedback：決策前先問「解決問題還是繞開根本問題」

## L2：threading.Thread(daemon=False) 擋不住 SIGUSR1
- 現象：改用 threading.Thread(daemon=False) 後，save_conversation 成功，但 insights/promise/profile 仍被 SIGUSR1 kill
- 根因：daemon=False 只讓 Python interpreter 等 thread 跑完；但 LiveKit framework 直接發 SIGUSR1 殺 process，不通過 Python cleanup 機制
- 下次：只要 cleanup 時間 > framework kill timeout（~25s），後段 task 必死；真正修法是 Cloud Tasks 非同步（掛斷時 enqueue，執行在別的 worker）
- 對應 feedback：決策前先問「解決問題還是繞開根本問題」

## L3：save_conversation 是唯一在 timeout 內完成的 critical task
- 現象：save_conversation（22s）剛好在 kill（25s）之前完成；insights 等沒機會跑
- 根因：save_conversation 包含兩個 Anthropic API call（extract_session_summary + summary compression），本身就很重
- 下次：Cloud Tasks 方案：on_disconnected 只做一件事 → enqueue cleanup job；insights/promise/profile/cost 全在 job 裡跑
- 對應 feedback：能本機重現就不要等遠端 cycle

## L4：Soniox API key 充值要幾秒生效
- 現象：加錢後立刻打，仍 402；等一分鐘後通
- 根因：Soniox balance refresh 有短暫延遲
- 下次：充值後等 30s 再測
