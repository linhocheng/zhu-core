## 1. Google Cloud SDK 在 Turbopack 下：拒絕 bundle，改 fetch + REST

**症狀**：
- `import { CloudTasksClient } from '@google-cloud/tasks'` static import 進 dialogue/route.ts → `npm run build` 直接炸 `Failed to collect page data for /api/dialogue` → `Cannot find module as expression is too dynamic at r.getJSON`
- 改 `const { enqueueStrategy } = await import('@/lib/cloud-tasks')` → build 通了但 production runtime 還是 `strategyEnqueueError: "Cannot find module as expression is too dynamic"`
- 加 `next.config.ts` 的 `serverExternalPackages: ['@google-cloud/tasks']` + static import → build 通了但 dialogue 直接 HTTP 500

**根因**：`@google-cloud/tasks` SDK 內部用 dynamic require（`r.getJSON` 從 proto descriptor 載 schema），Turbopack 的 static analyzer 解析不出來。`serverExternalPackages` 也救不回（Next 16.1.6 Turbopack 行為跟 webpack 不同）。

**解法**：**徹底捨棄 SDK，cloud-tasks.ts 改純 fetch + Node crypto**：
1. `node:crypto` 的 `createSign('RSA-SHA256')` 從 SA key 自簽 JWT
2. POST `https://oauth2.googleapis.com/token` 換 `access_token`（in-memory 50min cache）
3. POST Cloud Tasks REST v2 `https://cloudtasks.googleapis.com/v2/{parent}/tasks` 帶 Bearer

整支檔案剩 ~130 行，零 npm dependency 新增，build 秒過，runtime 直接 work。

**衍生天條**：
> Vercel Edge / Turbopack 環境下，Google Cloud SDK（@google-cloud/*）系列 default-不-bundle。
> 要嘛在 server-side route 用，要嘛改 REST API 自己 sign JWT。
> 三次嘗試（static / dynamic / serverExternalPackages）都打不通就停手，換思路。

**待寫進 memory**：`reference_google_cloud_sdk_no_bundle.md`

## 2. self-actAs：同一個 SA 自己 mint OIDC token 也要明確 grant

**症狀**：
- `enqueueStrategy` 終於跑進 Cloud Tasks REST，回 `403 PERMISSION_DENIED`
- 訊息：`The principal lacks IAM permission "iam.serviceAccounts.actAs" for the resource "strategy-enqueuer@..."`

**情境**：
- caller = `strategy-enqueuer` SA（用它的 key 從 Vercel 發 token）
- `task.httpRequest.oidcToken.serviceAccountEmail` = `strategy-enqueuer` SA（自己 mint 給 Cloud Run）

**直覺**：「同一個 SA 對自己 actAs 應該不用 grant 吧？」── 錯。GCP IAM 對 self-actAs 也要明確 binding。

**修法**：
```
gcloud iam service-accounts add-iam-policy-binding \
  strategy-enqueuer@zhu-cloud-2026.iam.gserviceaccount.com \
  --member="serviceAccount:strategy-enqueuer@..." \
  --role="roles/iam.serviceAccountUser"
```

注意 `iam.serviceAccountUser` 已包含 `actAs`，不需要另外加 actAs role。

**為什麼之前 SDK 路徑沒撞**：根本沒走過。bundle 一直炸，沒 reach REST 那層。fetch-based 重寫之後才是第一次真的 createTask call → 第一次撞 IAM。

**先前認知偏誤**：
之前修過一次「html-worker → enqueuer 的 actAs」覺得 IAM 都配好了。實際上「caller 對 oidcToken target」每組都是獨立 IAM 邊界，新組合 = 新 IAM。

**衍生規則**：
> 任何 SA-mint-OIDC-token 場景，明確列出 (caller, oidcToken.serviceAccountEmail) 兩個身份，
> 兩者不同要 grant 一次，**兩者相同也要 grant 一次**（self-actAs）。

## 3. 「dynamic import 救不了」的訊號：兩次 production 重現

**信號**：
1. build 過了
2. local `npm run build` 過了
3. 部署到 Vercel 後 HTTP 500 / strategyEnqueueError 寫 Firestore

**過去 5/10 對話有同樣 pattern**（conversation summary 顯示）：以為 `await import()` 是「Turbopack 救生圈」，實際上對含 dynamic require 的 npm package 沒用。Turbopack 還是會 trace import target，trace 失敗一樣炸。

**正解判斷流程**：
1. SDK 是不是純 ESM/CJS？（看 package.json `exports`）
2. 內部有沒有 dynamic require（proto loader、scoped fs load）？有 → bundle 必炸
3. 有 REST API 對應嗎？有 → 直接 fetch
4. 沒有 → 用 `serverComponentsExternalPackages` + Vercel Node runtime + 手動 ensure node_modules 進 lambda（複雜）

**衍生天條**：
> dynamic-import 不是 bundle 救生圈，是「我希望它是」的安慰劑。
> 第三次跑同樣的 expression-too-dynamic 訊息 = 不要再試 import 修法，換 REST。
