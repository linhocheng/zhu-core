# LESSONS 2026-05-23 · ailive 即時語音二分法 + librosa 兇手確認

## L1：librosa numba JIT 是 Cloud Run 的 CPU 殺手
- 現象：即時語音進入後 6 秒開始 dropped 100/200 frames，馬雲/吉娜聽不到用戶說話
- 根因：librosa.feature.mfcc 內部用 numba JIT compilation，在 Cloud Run 小型 CPU 上第一次跑要 51 秒，期間 CPU 100% 被佔用，VAD queue 爆炸
- 下次：Cloud Run 環境用到任何需要 JIT 的套件（numba/jax/torch compile）之前，先估算 cold-start CPU 時間
- 對應 feedback：決策前先問「解決問題還是繞開根本問題」

## L2：二分法一段一段跑才能找到真正兇手
- 現象：我一開始猜測是 insight 提煉（cbf58f9）造成記憶壓力，結果 S1-S3 全乾淨
- 根因：S1 cbf58f9 / S2 1f1fb1f / S3 04689e0 三段都是 session 結束後跑，不影響 session 期間；S4 ca59d4b librosa 是 session 啟動時跑，時間點不同
- 下次：memory_high 出現的時間點（17-25 秒，session 期間）是關鍵線索，應更早鎖定「session 啟動時做了什麼」
- 對應 feedback：記憶會說謊，越具體越要去現場驗

## L3：Deepgram 中英雙語在 streaming mode 有架構限制
- 現象：language="multi" 在 Deepgram streaming 會炸（plugin 源碼硬擋），Flux Multilingual 不含中文
- 根因：Deepgram streaming 設計要事先指定語言，中文不在 Flux Multilingual 支援清單（截至 2026/05）
- 下次：換 Soniox STT（language_hints=["en","zh"]，官方 LiveKit plugin），先查 interim_results 對應參數
- 對應 feedback：先上網查再猜

## L4：純 numpy 聲紋特徵可以替換 librosa MFCC
- 現象：ZCR + energy + FFT spectral（20-d）純 numpy 實作，計算微秒級，00059-x6n 穩定無爆炸
- 根因：MFCC 精準度高但依賴 numba；ZCR+FFT 精準度稍低（threshold 從 0.75 調到 0.92）但 CPU 無負擔
- 下次：Cloud Run 限制環境優先選無 JIT 的特徵提取方案
