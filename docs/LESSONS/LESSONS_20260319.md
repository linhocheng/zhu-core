## 1. 兩套 upload API 並存，改錯地方

## 2. Grok multi-image edits：臉被產品圖覆蓋

### 現象
加了 `reference_image_url`（產品圖）後，角色的臉跑掉了。

### 心
Grok `/images/edits` 的 `image` 欄位是單張——傳產品圖就把臉的 ref 擠掉了。兩個 ref 需求（臉 + 產品）衝突，舊架構只容一張。

### 法
用 Grok 的多圖 edits：
```json
{
  "images": [
    { "url": "臉的ref", "type": "image_url" },
    { "url": "產品圖", "type": "image_url" }
  ]
}
```
prompt 明確說：`Use face from FIRST image, product/clothing from SECOND image`

### 解
`grok-imagen.ts` 改成三條路：單 ref / 多 ref（臉+產品）/ 純文字生圖。

```bash
...
TSEOF
```

### 解

## 4. 知識庫圖片條目 content 描述過長反而稀釋語意

### 現象
圖片條目 Haiku 描述 150 字，角色搜尋時找不到正確的產品圖。

### 心
語意搜尋靠 embedding，embedding 靠 content 的語意密度。150 字的圖片描述夾雜各種細節，語意被稀釋，搜尋分數反而比 20 字商品名更低。

### 法
圖片條目 content 只存商品名稱，20 字以內：
```
prompt：這張圖片是什麼產品？只輸出商品名稱，不超過20字，不要其他說明。
max_tokens: 60
```
imageUrl 獨立存欄位，不混在 content 裡。

### 解
搜尋命中率提升，角色正確拿到圖片 URL。

## 5. query_knowledge_base：搜尋結果沒有暴露 imageUrl，Claude 看不到

### 現象
知識庫 image 條目有 `imageUrl` 欄位，但角色說「我不知道圖片網址」。

### 心
`query_knowledge_base` 回傳時只輸出 `content` 的前 300 字，`imageUrl` 是獨立欄位，沒有被顯式輸出。Claude 的 context 裡根本沒有這個值，自然找不到。

### 法
在回傳字串裡明確加一行：
```ts
const imgLine = (d._type === 'knowledge' && d.imageUrl)
  ? `\n[產品圖 imageUrl: ${d.imageUrl}]（生圖時填入 reference_image_url）`
  : '';
```

### 解
Claude 能在 tool result 裡直接讀到 imageUrl，並填進 `generate_image` 的參數。

## 6. 記憶架構選型：過度設計 vs 最短路徑

### 現象
設計知識圖譜時，規劃了兩個新 collection、多個 API、完整 entity/relation schema。

### 心
真正的痛點只有一個：角色知道「條目」，不知道「關係」。但關係已經在文字裡了，只是搜尋時沒有利用。建 KG 是 500+ entity 規模的解法，現在是 50 條的規模——過度設計。

### 法
最短路徑兩步：
1. sleep engine 掃 knowledge 做 cosine > 0.85 去重（自動結晶）
2. query_knowledge_base 搜 15 條 → Haiku 一步推理關係 → 回傳結構化 context

不建新 collection，不改 schema，多一次 Haiku call（$0.001）換關係推理能力。

### 解
兩步落地，角色理解關係，知識庫自動去重。真正的 KG 等規模到了再建。

## 8. Anthropic SDK tool 格式 ≠ OpenAI 格式，換模型要重寫 tool loop

### 現象
想把 dialogue 的 Claude Sonnet 換成 Grok，以為只要換 model 名稱，結果整個 tool use loop 都要改。

### 心
Anthropic SDK 和 OpenAI 格式的 tool 定義、tool result、loop 結構完全不同：

| 差異點 | Anthropic | OpenAI（Grok）|
|---|---|---|
| tool 格式 | `input_schema` | `parameters` |
| tool result role | `user`（包 `tool_result` block）| `tool`（帶 `tool_call_id`）|
| stop reason | `tool_use` | `tool_calls` |
| response 結構 | `response.content[]` | `response.choices[0].message` |

### 法
換 OpenAI 相容模型（Grok/OpenAI）時，用 `fetch` 直打 `/v1/chat/completions`，不用 SDK：
```ts
const res = await fetch('https://api.x.ai/v1/chat/completions', {
  headers: { Authorization: `Bearer ${XAI_API_KEY}` },
  body: JSON.stringify({
    model: 'grok-3',
    messages: [{ role: 'system', content: ... }, ...history],
    tools: tools.map(t => ({ type: 'function', function: t })),
    tool_choice: 'auto',
  }),
});
const choice = res.choices[0];
// tool call: choice.message.tool_calls[]
// tool result: { role: 'tool', tool_call_id: tc.id, content: result }
// stop: choice.finish_reason === 'stop' || !msg.tool_calls?.length
```

### 解
dialogue 換 grok-3 成功，tool loop 完整運作。

## 10. Vercel CLI deploy 失敗時的備案

### 現象
`npx vercel --prod` 反覆報 `Unexpected error. Please try again later`。

### 心
Vercel CLI 直接部署有時會遇到雲端 build 環境問題，但本機 build 是通的。

### 法
兩步備案：
```bash
npx vercel build --prod          # 本機跑 build，產出 .vercel/output
npx vercel deploy --prebuilt --prod  # 只上傳 output，跳過雲端 build
```
`git push` 觸發自動部署也是選項，但要確認 GitHub integration 有設好。

### 解
prebuilt deploy 每次都成功。這條路比直接 `vercel --prod` 更穩。

## 12. Gemini gs:// fileUri 不適用於 Gemini API（只能用 Vertex AI）

### 現象
想用 `{ fileData: { fileUri: 'gs://bucket/path' } }` 傳圖給 Gemini，回傳 400 Invalid Argument。

### 心
`gs://` URI 是 Google Cloud Storage 格式，只有 **Vertex AI** 版本的 Gemini 支援。
普通的 **Gemini API**（generativelanguage.googleapis.com）只接受 `inlineData`（base64）或公開 HTTP URL。

### 法
Firebase Storage 的公開 URL（`https://storage.googleapis.com/...`）要先下載成 base64 再送進 Gemini：
```ts
const res = await fetch(url);
const buffer = Buffer.from(await res.arrayBuffer());
const data = buffer.toString('base64');
// { inlineData: { mimeType, data } }
```

### 解
回到下載 base64 的方式，通了。

## 13. 圖片 media_type 寫死 image/jpeg 導致 Claude 400

### 現象
用戶上傳 WebP 圖片，Claude 回傳 400：`The image was specified using the image/jpeg media type, but the image appears to be a image/webp image`

### 心
前端寫死 `media_type: 'image/jpeg'`，但現代手機截圖、iOS 分享的圖片很多是 WebP。
Claude 會驗證 base64 實際內容和宣告的 media_type，不符就拒絕。

### 法
**server 端從 base64 header 偵測真實格式，不信任前端傳的值：**

```ts
const header = image.data.slice(0, 16);
const bytes = Buffer.from(header, 'base64');
if (bytes[0] === 0x89 && bytes[1] === 0x50) image.media_type = 'image/png';
else if (bytes[0] === 0xFF && bytes[1] === 0xD8) image.media_type = 'image/jpeg';
else if (bytes[0] === 0x52 && bytes[1] === 0x49) image.media_type = 'image/webp';
else if (bytes[0] === 0x47 && bytes[1] === 0x49) image.media_type = 'image/gif';
```

前端也要同步修：從 dataUrl 解析真實 mimeType，不寫死。

### 解
server 自動偵測修正，JPG/PNG/WebP/GIF 全部通了。

## 14. 多次修補越改越壞的教訓——回到最乾淨版本

### 現象
Gemini 生圖引擎改了 sharp → gs:// → loopback fetch → hardcode URL，四次修補，每次解一個問題又冒出新問題，最後連基本生圖都壞了。

### 心
每次修補都在「當前壞掉的版本」上加補丁，沒有停下來問「最乾淨的版本在哪」。越改越遠離根本。

### 法
出現「越改越壞」的信號時，立刻：
1. `git log --oneline` 找到「最後一個好用的 commit」
2. `git show <hash>:path/to/file` 確認那個版本的內容
3. `git checkout <hash> -- path/to/file` 直接 restore
4. 從那個乾淨基礎再往前

不要在壞的版本上繼續打補丁。

### 解
`git checkout 1040460 -- src/lib/gemini-imagen.ts` 還原到初版 Gemini，一次解決。
