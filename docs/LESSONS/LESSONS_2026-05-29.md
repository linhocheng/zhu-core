# LESSONS 2026-05-29 · Queue 契約 + Async Worker 五問心法

## L1：`failed` 在 TTL 內被當 `running` 擋掉 → Cloud Tasks 永久放棄

- 現象：OpenAI 502 → worker 拋錯 → `failWorkerRun(status=failed)` → Cloud Tasks 重試 → `acquireWorkerLock` 看到 TTL 未過 → 回 `already_running` → caller 回 200 → Cloud Tasks 以為成功 → 不再重試 → 任務永久卡住
- 根因：TTL 鎖沒有過濾 `status`，把 `failed` 和 `running` 一視同仁
- 修法：`if (d.status !== "failed" && withinTtl)` + `already_running → 409`
- 對應 feedback：queue 不讀心，code 即契約

## L2：`already_running` 回 200 是對 queue 說謊

- 現象：lock 回 200，Cloud Tasks 記下「完成」，不再重試，任務永久消失
- 根因：`already_running` 和 `already_done` 被同一個 200 回應
- 修法：`already_done=200`（停止），`already_running=409`（繼續等待）
- 對應心法：Queue 的眼睛只有 status code。說 200 等於立誓。

## L3：三問法的「心」在操作層才看得到

- 現象：三問法寫進了心法（獨孤九劍），但 code 還是沒有落地
- 根因：心法是觀念層，操作層沒有 checklist 對應
- 修法：建 `async-worker-checklist.md` skill，心法管眼睛，skill 管手
- 下次：蓋新 worker 時先過五問 checklist，不靠記憶

## L4：自創不存在的 class 名 = 沒查現場就動手（補丁）

- 現象：ailive client `IdentityScreen` 用了 `screen / screen-header / screen-title`——這三個 class 在 `client-v2.css` 根本不存在，全靠 inline style 撐版。跟其他分頁（`topbar`+`content`+`page-head`+`gallery-cell`+`dropzone`）兩套。Adam 一眼看穿「不能補丁」。
- 根因：加新畫面前沒 grep 同級 sibling component 的 wrapper 慣例，憑印象自創 class + inline style
- 修法：refactor 成既有設計系統（topbar/content/page-head/dropzone/empty/gallery-cell），新視覺元素（角度/主要 badge）加進 `client-v2.css` 成設計系統的一部分，不散在 JSX inline
- 對應 feedback：`feedback_ui_conform_no_patch.md`（新建）、`feedback_interface_blood_vessel_check.md`、CLAUDE.md UI 品味
- 下次：寫新畫面第一步 grep 同檔 sibling 的 className 結構照抄，inline style 是補丁的味道

## L5：改 live 前先確認「哪一份才是 production」（真相分裂）

- 現象：strategy 邏輯有兩份副本——`~/.ailive/strategy-worker/src/index.ts`（Cloud Run，真 live）和 `ailive-platform/src/app/api/specialist/strategy/route.ts`（Vercel，死副本）。上個 session 把 self 鬆綁修正套到了 Vercel 死副本，自以為上線了。這 session Adam 一句「是走 Max 的 key 吧」追問，才循 `cloud-tasks.ts` 的 STRATEGY_WORKER_URL 揭穿——enqueue 走的是 Cloud Run，Vercel route 全無人呼叫。
- 根因：同一邏輯存兩份沒同步 = 破綻「真相分裂」。grep `isSelfCommission` 在 strategy-worker 回空，本該當下警覺「我改的檔跟 live 不是同一份」，卻沒追。
- 修法：先用 `cloud-tasks.ts`／enqueue 路徑確認「誰是真正被呼叫的 endpoint」，再動手。修正套到 live Cloud Run 檔、deploy（revision 00005-frn 100% 流量）、端到端真跑馬雲 self 驗過，最後刪掉死副本（確認無人 import／fetch 才刪）。
- 對應 feedback：CLAUDE.md 破綻三處「真相分裂（兩份即是零份）」、`feedback_diagnosis_verify_before_write.md`、新建 `reference_ailive_strategy_pipeline.md`
- 下次：改任何 worker／route 前，先問「這支真的會被呼叫嗎」——順著 enqueue／URL 常數找到 live endpoint 再改。grep 找不到預期變數 = 紅旗，不是「命名不同」一句帶過。

## L6：修正放在匯流點 = 入口無關，不必每入口各測

- 現象：擔心 self 鬆綁要在 dialogue（文字）和 voice-stream（語音）各改一次。實際上兩入口都只是建 `requesterId===assigneeId` 的 platform_jobs doc + `enqueueStrategy`，全部匯流到同一個 strategy-worker。`isSelfCommission` 與 form 判斷都在 worker，改 worker 一處，兩入口自動都吃到。
- 根因：修正點選在資料流的匯流節點（worker），而非各入口。入口只負責產生契約一致的 job。
- 下次：判斷「要改幾處」先畫資料流——若多入口匯流到單一 worker，改 worker 即可；驗證選一個入口端到端跑通即代表全部（LiveKit realtime agent 是例外：它在遠端 VM，能否發委託未驗，源碼不在本機）。
- 對應 feedback：CLAUDE.md 破綻三處「流動斷裂」反面——找對匯流點就一次到位
