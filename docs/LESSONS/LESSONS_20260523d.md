# LESSONS 2026-05-23d · Harness Lite 五 worker 遷移

## L1：QA fail 不是 WorkerError——業務路徑 vs 基礎設施故障要分清楚

- 現象：section-qa worker 的 QA "fail" 結果（字數不足、來源不可追溯）看起來像「失敗」
- 根因：repair 機制是給基礎設施故障用的（LLM 炸、parse 炸、Storage 掛），不是給「文章品質不過關」用的
- 下次：先問「這個 error 是 infra 問題還是 domain 邏輯問題」，domain 邏輯的 bad path 不觸發 repairAttempts
- 對應 feedback：feedback_solve_root_not_symptom

## L2：worldStateVerify 是「寫入確認」，不是「業務正確性審查」

- 現象：stitch worldStateVerify 只查 status + stitchedMarkdownUrl 是否存在
- 根因：world state verify 的職責是確認 handler 的副作用有落地，不是重新執行業務邏輯
- 下次：verify 三問：（1）該寫的 doc 寫了嗎？（2）該更新的 status 更新了嗎？（3）該上傳的檔案有 URL 了嗎？

## L3：createHarnessWorker 的 repairAttempts 掛在 article/section doc，不在 worker_runs

- 現象：之前理解是 per-task 計數，其實是 per-doc 計數
- 根因：Cloud Tasks 可能重試同一 taskId，repairAttempts 要跨 task 累積，所以住在業務 doc 上
- 下次：設計 repair 計數時先問「這個計數跨哪個維度累積」
