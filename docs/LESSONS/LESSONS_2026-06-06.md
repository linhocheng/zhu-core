## L7：nullable 巢狀欄位的 prompt 只給 null 範例 → 模型走另一分支時亂丟字串 → repair loop 重問是天條坑

- 現象：MACS 兩個案子（含我自建測試案）都死在 partner-review，同一個錯：`revisedStoryline expected object received string`。repairAttempts=4 仍死。
- 根因：schema 是 `z.object({...}).nullable()`，但 prompt 從頭到尾只示範 `verdict=ok` 的 `revisedStoryline:null`，從沒給過 `verdict=revised` 的物件形狀。模型走 revised 分支時無模板可循 → 丟一段散文（字串）。harness 的 repair loop 只是 Cloud Tasks 重派同一個 worker、重問同一個沒範例的 prompt → 每次都丟字串 → 撞天條那個坑「拿 LLM 補 LLM 壞輸出」。
- 下次：(1) 任何 nullable/條件式巢狀欄位，prompt 必須給「兩個分支都完整」的範例，不能只給 null 那支。(2) 驗證層加確定性 coerce（字串化 JSON 就 parse，散文退回 null 讓下游 fallback），不要靠重問。(3) 看到 repairAttempts 已耗盡、錯誤每次一樣 → 立刻警覺是「重問補壞輸出」反模式，不是「再多試幾次會過」。
- 對應 feedback：feedback_deterministic_work_belongs_in_code（天條）
