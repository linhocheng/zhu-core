## L1：後台讀寫血管不對稱——讀三模式、寫只認 mode 1

- 現象：`getRoleSettings(mode)` 三模式都通（worker 跑 Mode 2/3 拿得到 prompt），但 roles API 和後台 UI 寫死只認 `settings/roles` 一份 doc + mode-1 keys。Mode 2/3 在後台唯讀，只能吃 hardcoded default，改了存不進去。
- 根因：讀取層加 mode 分支時沒同步改寫入層——兩條血管對稱性假設沒成立。
- 修法：單一真相源 `roleModeSurface(mode)` 收斂 doc id + keys + defaults，讀寫都吃它。
- 下次：改讀取層加 mode 分支後，立刻問「寫入層對稱了嗎？」（API + UI 各一處）。
- 對應 feedback：feedback_interface_blood_vessel_check（介面建完問血管接通了嗎）

## L3：repair 腳本漏傳 overrideBaseUrl → 任務送 Vercel 404，靜默失敗

- 現象：寫了 `_repair-case.mts` 重新 enqueue 研究任務，腳本說「re-enqueued」，但案子卡在 `research_running` 超過 30 分鐘，Cloud Run 完全沒收到請求。
- 根因：`productionEnqueue` 第四個 arg 是 `overrideBaseUrl`。issue-tree 那邊傳了 `RESEARCH_WORKER_BASE_URL`，我寫的 repair 腳本漏了這個 arg → 任務送到 `WORKER_BASE_URL`（Vercel）的 `/api/workers/research`，但 Vercel 沒有這個 route → 404 → Cloud Tasks 一直重試 Vercel，Cloud Run 完全沒看到。
- 「re-enqueued」訊息是 Cloud Tasks 建任務成功，不是 Cloud Run 收到執行成功——兩件事，腳本說謊了（無意的，但沒驗到底）。
- 下次：腳本排隊後必看 Cloud Run log 確認有沒有新 request 進來；`productionEnqueue` 有第四個 arg 時必核。
- 對應 feedback：沒跑過 = 沒收完 / feedback_subagent_honesty（自己打的 curl 不等於目標跑了）
