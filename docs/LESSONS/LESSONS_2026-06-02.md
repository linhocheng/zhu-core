## L1：bridge 524 = 單次生成超 ~100s，不是一次性 bug

- **現象**：export 的 `structureAnalysisChapters` 一次把 6 個分析備忘錄批次餵 bridge，6×6 欄位，output > 3K tokens，生成 ~150s → Cloudflare 524。
- **根因**：bridge 背後是 Cloudflare → ~130s 邊緣超時；任何 bridge call 不管在 Vercel 還是 Cloud Run，只要單次 LLM 生成 > 100s 就炸。
- **修法**：移到 Cloud Run，**每個 memo 一個獨立 bridge call**（maxTokens: 1200，~30s/次），6 個 memo 順序跑 = 180s 但每次 call 不炸 CF。
- **下次**：看到 bridge 524 先問「這個 call 生成多少 token？」不要直接加 retry。
- **對應 feedback**：`reference_bridge_v1messages_effort.md`、`reference_vercel_300s_lambda_limit.md`

## L2：curl timeout ≠ Vercel function timeout（鎖已建立）

- **現象**：curl `--max-time 60` 到了就 exit 28。但 Vercel 那端的 function 還在跑。下一個 curl 撞上 409 `already_running`。
- **根因**：curl timeout 是 client-side；Vercel function 獨立執行；harness lock 在 curl 發出後 < 1s 就建立了，TTL 10min。
- **下次**：看到 409 + `already_running` 先去 Firestore 查 `worker_runs/<taskId>.status`，若 `running` = 函式還活著，等。不要重發，否則又 409 浪費一次。

## L3：enqueueTask 第 4 參數是 delaySecs，不是 overrideBaseUrl

- **現象**：`scripts/_reset-crossreview.mts` TS error `Argument of type 'string | undefined' is not assignable to parameter of type 'number | undefined'`。
- **根因**：`enqueueTask(queue, path, payload, delaySecs = 0, overrideBaseUrl?)` — 第四個是 delay，第五個才是 URL。
- **修法**：`}, 0, process.env.RESEARCH_WORKER_BASE_URL)`
- **下次**：寫 enqueueTask 調用前先看 signature，別靠記憶猜參數順序。

## L4：Vercel env var 加了不等於生效，要 redeploy

- **現象**：`STRUCTURE_ANALYSIS_BASE_URL` 加到 Vercel 後，現有 prod 不感知。
- **根因**：env var 在 build time 注入，已部署的 Lambda 用的是舊快照。
- **修法**：`npx vercel --prod --yes` redeploy。
- **下次**：加 env var 後口頭說「要 redeploy 才生效」，不要假設「加了就通了」。
