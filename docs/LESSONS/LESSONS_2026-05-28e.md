# LESSONS 2026-05-28e · blueprint_running 後患與 delete-then-recreate

## L1：status flip 防重複寫入會讓 harness retry 路死掉
- 現象：blueprint handler 開頭設 blueprint_running，讓 precondition 在重試時擋住不重複建 docs
- 根因：harness 的 catch block 把 PRECONDITION error 也算失敗 → repairAttempts++ → 三次後 needs_repair；文章永遠卡在 blueprint_running
- 下次：防重複寫入不要靠 status flip，要靠寫入本身幂等（delete-then-recreate 或 set with deterministic ID）
- 對應 feedback：feedback_solve_root_not_symptom（解決問題還是繞開根本問題）

## L2：delete-then-recreate 讓重試路保持暢通
- 現象：blueprint handler 在 image_tasks loop 前先刪掉 articleId 相同的舊 docs
- 根因：precondition 依然只允許 source_ready → retry 重跑整個 handler → 先刪舊資料 → 再寫新資料 → 正常推進
- 下次：幂等寫入優先考慮「先清再建」或「set with deterministic ID」，不要改 precondition 邏輯
- 對應 feedback：feedback_memory_can_lie（越具體的記憶越要驗現場）

## L3：全鏈路兩連跑確認修法有效
- 現象：薑黃保健品市場 + 網紅行銷兩篇都跑到 done，無 needs_repair，無重複 image_tasks
- 根因：三個修法（watchdog enqueueTask + delete-then-recreate + skipGates）同時生效
- 下次：修好一個結構性問題後，跑兩次 end-to-end 才算驗證通過（一次可能是偶然）
