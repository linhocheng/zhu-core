# LESSONS 2026-05-28b · ANEWS CF 524 根治 + article chain 修復

## L1：Cloudflare 對所有代理流量的 100s 硬頂，包含你自己的後端
- 現象：article-write Cloud Run → bridge.soul-polaroid.work → 主文生成 127-163s → CF 524
- 根因：CF 的 Enterprise plan 預設 proxy timeout 100s，任何超過的 TCP 連線都會被砍，不管是前端還是 machine-to-machine
- 下次：Cloud Run / VM 之間的 m2m 流量，一律走 direct IP + port，不要繞 CF domain
- 對應 feedback：feedback_avoid_extra_security_layers（不要為了「安全多一層」加會壞的元件）

## L2：多條平行觸發鏈導致同一文章被寫 3 次
- 現象：K1f1eg4J35mrdeATP8Kx main 文章在 7:35、7:37×2 被寫三次
- 根因：auto-kick cron、Cloud Tasks retry、手動 curl 三者幾乎同時 fire，都用不同 taskId → 各自取得 idempotency lock → 各自完成寫作
- 下次：idempotency 鎖的 key 應該是 articleId 不是 taskId，或者 article 進入 section_writing 時就加一把文章級 mutex
- 對應 feedback：feedback_solve_root_not_symptom（方案成功後根因還在 = 繞道不是解法）

## L3：GCP firewall target-tags 踩雷
- 現象：建 allow-bridge-3001 firewall rule 用 --target-tags=zhu-bridge，但 VM 的 tag 是 zhu-dev → 規則沒作用
- 根因：target-tags 是「哪些 VM 接收這條規則」，設錯 tag 等於規則對 VM 視而不見
- 下次：建 firewall rule 前先 `gcloud compute instances describe <VM> --format="json(tags)"` 確認 tag
