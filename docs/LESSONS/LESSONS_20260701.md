# LESSONS 2026-07-01 · ailivex 影片生成 avatarUrl 根治

## L1：HeyGen talking_photo_id 是短效 ID，不能存起來之後用
- 現象：admin 上傳頭像後存 `heygenAvatarId`，幾天後拿去生成影片 → `avatar_not_found`
- 根因：`POST upload.heygen.com/v1/talking_photo` 回傳的 ID 是臨時的，HeyGen 不保證長效；「上傳完馬上用」才是設計意圖
- 下次：需要 talking_photo 的場景，每次生成前用圖片 URL 即時 upload → 拿新鮮 ID → 馬上用；不要存 ID 等之後用
- 對應 feedback：標了風險不等於驗了風險（我們以為存了 ID 就可以隨時用，沒驗過過幾天還有沒有效）

## L2：input 傳遞鏈每一層都要改，不能只改最底層
- 現象：改了 types.ts + heygen-video.ts 支援 avatarUrl，第一次 deploy 仍然報同樣錯誤
- 根因：worker.ts 是 input 的解構層，只明確取了 `input.avatarId`，`input.avatarUrl` 被靜默丟棄；heygen-video.ts 收到 undefined
- 下次：改任何 provider input 結構，必須同時追蹤：Vercel route → media-worker /v1/jobs body → worker.ts 解構 → provider.generate()，四層都確認
- 對應 feedback：能本機重現就不要等遠端 cycle（若能本機跑 worker 就能立刻看到 input 是什麼）

## L3：舊成功 job 的 Firestore 記錄是最快的 debug 線索
- 現象：花了時間查 heygen API 文件，卻是翻舊 mw_jobs 找到根因（成功的 job 用 avatarUrl 不用 avatarId）
- 下次：遇到「以前可以現在不行」的問題，第一步去 Firestore 找最近一筆成功 job 的 input 格式，比猜 API 快 10 倍
- 對應 feedback：能本機重現就不要等遠端 / 先讀現場不猜
