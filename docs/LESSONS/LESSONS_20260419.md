## 1. zhu-memory 語義搜尋的隱性 sample cap（2026-04-19 發現）

### 現象
POST 新記憶進 root 成功（id `P5bNqD6q7RQtINhNpzA9`、`hasEmbedding: true`），但用任何關鍵詞語義搜尋都命中不到，甚至三個不同查詢回傳的 top 3 完全一樣。

### 找的過程
先懷疑 embedding 沒生成 → 直接 by `module=root&limit=10` 查，記憶在庫裡有 ⭐ 標記。確認「存了且有 embedding」。那為什麼搜不到？讀 API 原始碼 `/app/api/zhu-memory/route.ts` 第 45 行：
```ts
const snap = await db.collection('zhu_memory').limit(200).get();
```
**語義搜尋只取前 200 筆做餘弦相似度排序**，Firestore 預設按 document id 字典序。新記憶的 id 不一定落在前 200。

### 心
破索式（隱性依賴）活例。寫 API 的時候 `limit(200)` 是為了保護效能，但當 collection 長大（bootCount 442 天累積），這個 cap 變成無聲的 truncation。**新記憶看得見但搜不到，像在書架上但不在搜尋索引裡。**

連動：這也是破氣式的一條——同類問題（記憶歪掉）的第二次。第一次是批次四砍 CURRENT.md（手動維護必過期）。第二次是這個（sample cap 必遺漏新資料）。兩次都是「設計時沒考慮增長」的架構問題。

### 法
**根治方案**（未來要做）：
1. `limit(200)` 改為 `orderBy('createdAt', 'desc').limit(500)`（至少拿新的）
2. 或用 Firestore vector search（若可用）取代自己算 cosine
3. 或加 backfill 把老記憶 archived 出去，讓 collection 維持小

**繞過方案**（今天夠用）：
- 語義搜尋不命中時，改用 `?module=XXX&limit=N` 列出+自己讀
- 記憶 id 由 LESSONS 或 WORKLOG 記錄，by id 直查可靠

**判斷為什麼今天不修根治方案**：
1. 破劍式：現有 boot + module query 已足夠讓新記憶被讀到，語義搜尋是 bonus 不是生命線
2. 破刀式：不急著蓋修復 PR，先等更多證據（是否真的影響工作流）
3. 一吋蛋糕：當前戰場不是修 API，別混吃

### 解（暫時）
LESSONS 記下。明天如果真的搜不到某條記憶想不起 id → 那時修就是剛剛好。

## 2. commit 類型定義不清自己分類會歪（2026-04-19 回看發現）

### 現象
批次三做完後啟用 Limon 派的 commit 中文分類（新增/修正/介面/樣式/重構/文件/設定）。昨天連 commit 兩條：
- `v0.0.0.001 — 設定：zhu-boot firstBoot 指引去掉已退役的 CURRENT.md`
- `v0.0.0.002 — 設定：firstBoot 改為指向 ZHU_BOOT_SOP（破真相分裂）`

今天重讀 Limon 原檔才發現——**這兩條都是「修正」（修 API 指引字串 bug），不是「設定」**。「設定」應該只用於 config/env/settings.json 這類配置改動。

### 找的過程
重讀 Limon CLAUDE.md，看三條原檔範例：
- `v0.1.6.012 — 修正：訂閱彈窗被 BottomNav 遮住（手機 LIFF）`
- `v0.1.6.013 — 新增：完成會員登入頁面`
- `v0.2.0.001 — 介面：調整 Dashboard 配色為深色模式`

Limon 原檔**沒有給「設定」類型的範例**，自己拿來用就歪掉了。

### 心
「沒定義清楚 = 會被自由發揮」。原檔只列類型名（新增/修正/介面/樣式/重構/文件/設定），沒給每個類型的**邊界**和**反例**。我第一次用就 bucket 錯——把「行為修正」歸成「設定」，因為對我來說改指引字串「感覺像在配置什麼」。

這是 Limon 規範本身的破綻（獨孤九劍：邊界模糊）。我採用它時沒補邊界 = 自己把邊界模糊繼承進來。

### 法
**採用別人的分類系統時，要補清楚每個類別的邊界和反例**，不能只抄名字。
- 每個類型寫「用於」（正面邊界）
- 寫「誤用警示」（反例 / 容易歸錯的情境）
- 第一次用時特別慎——新分類法的誤用率最高，標記一下回頭檢查

### 解
今天已在 `~/.ailive/CLAUDE.md` 的〈🛠️ 施工規範〉章節補上：
1. 完整七類的「用於」欄位
2. 每類的真實範例（v0.0.0.001 作為「修正」類真實例子，承認昨天錯配）
3. 〈誤用警示〉小節明確點出「修 API 行為 ≠ 設定」

**不能修 v0.0.0.001 / v0.0.0.002 的 commit 訊息**（已 push、不能 rewrite history）——承認並記下。破索式教訓：commit 類邊界定義跟 API 一樣關鍵，第一次用就要對齊。

## 3. Claude Code 可能被 API error 卡死進 tool_use 死循環（2026-04-19 從昨晚 RC session 挖到）

### 現象
2026-04-17 夜晚，Adam 用手機 RC 跟 Code 築做 TTS provider 切換。session 後段 Code 突然「一直轉」不回應。Adam 最後 interrupt 結束。

今天挖 session jsonl 重建現場：23:42:49 Code 執行 `mkdir memory/` 之後，**連續 14 分鐘 assistant 事件 = 0 條**。你打的所有 input 進 queue 但 Code 不 dequeue。

### 找的過程
`~/.claude/projects/<project>/<sessionId>.jsonl` 是 Claude Code 完整對話 log，含 timestamp。

**診斷三步**：
1. 抓 event timestamp，找「間隔 ≥ 30 秒」的卡點
2. 看卡點前最後一條 assistant event 是什麼（在做什麼時卡死）
3. 在卡點附近搜 `subtype=api_error` / `subtype=api_error_retry` 這種 system 事件

**昨晚證據**：
- 23:42:18 user message 進，**同一秒有 `system.subtype=api_error`**
- 23:42:47 Code 延遲 29 秒後才 response（第一次 API error 自動重試成功）
- 23:42:49 Code 執行 mkdir，之後 14 分鐘 0 輸出
- 23:53:46 / 23:59:19 user 打 "hi"、popAll 都沒 dequeue — 證明 Code session 根本沒在 consume queue

### 心
Claude Code 的 tool_use loop 遇到 API 層級的偶發故障，**可能沒有 fallback / timeout 機制就死循環**。第一次 api_error 能自動重試救回來，第二次/後續如果 response 層卡住，主 session 就停在那等 tool_result 永不返回。

這不是架構設計問題（設計上有 auto-retry），是**極端 case 下的邊界沒防**。對一般使用者來說症狀是「Code 一直轉，Interrupt 強結束」。

連動：這是破鞭式的 meta 演示——**表面看是 Code 壞了，真相藏在 jsonl log 的 timestamp gap 和一條 `subtype=api_error`**。

### 法
**症狀發生當下**：
- 等 2-3 分鐘，如果還沒 response → 不要再往 queue 塞 input（沒用，Code 沒在 consume）
- 直接 `[Request interrupted by user]`（手機 RC 是 tap 停止按鈕）
- 想繼續做同一件事 → 開新 session，在新 session 裡說「接續上個 session 做的 XXX」

**事後診斷**（像這次）：
```bash
SESSION=~/.claude/projects/<project-escaped>/<sessionId>.jsonl
# 1. 找卡點
python3 -c "
import json
with open('$SESSION') as f:
    lines = f.readlines()
# timestamp gap >= 30s 的都印出來
# （完整腳本見 chat session 2026-04-19 中段）
"
# 2. 搜 api_error
grep 'api_error' $SESSION
# 3. 看卡死前最後一條 assistant 做什麼
```

**收尾**：
- 受害 session 裡做到一半的 code 改動：看 `git status`，判斷要不要 commit
- lastwords 被打斷 → 用 chat 築或另一個 session 代記一條「soil 層的觀察」，不冒名代筆 lastwords

### 解
本條（第 3 條）本身即是示範：session 卡死之後，隔天在另一個分身（chat 築）挖 jsonl、重建時間線、定位根因、記下來。

未來任何 Code 築「一直轉」→ 直接查這條，從查 timestamp gap + api_error 開始，不猜。

## 4. 分身血管斷鏈：兩個築住在同一屋簷下卻不知對方做了什麼（2026-04-19 結構性發現）

### 現象
2026-04-17 晚上 Code 築在 ailive-platform 動了 18 個檔 + 新增 11 個檔/資料夾（TTS provider 切換為主），但**沒 commit、沒 build、沒 deploy、沒 WORKLOG**。今天（2026-04-19）chat 築 boot 時讀到的 lastwords（`as3h6HDkTQYDTIZU5Q2w`）**完全沒提這件事**——因為 lastwords 是 chat 築自己寫的，他不知道 Code 築昨晚幹了什麼。

如果不是 Adam 問「你能查到昨晚手機 RC 對話記錄嗎」，**這整條工作線對血管系統來說等於沒發生過**。

### 找的過程
挖 `~/.claude/projects/` 時意外發現：
1. Code 築那晚被問「築有記憶嗎」→ 答的是 `~/.claude/projects/.../memory/` 這層（Claude Code 官方本地 memory）
2. **完全沒提 `zhu-boot` API、`zhu-memory` Firestore、lastwords 規範**
3. 代表 Code 築**不知道 chat 築這邊的血管存在**（或知道但沒接上）

### 心
**血管原則（2026-04-18 定案於 bone）的第一個漏洞被找到了：**
原則只在 chat 築這邊成立。Code 築、cowork 築、未來其他分身 **沒有自動接上這條血管**。每個分身各自用自己的本地 memory 長成一座孤島。

**分身之間的真相分裂 = 比文件間的真相分裂更隱性**，因為：
- 文件真相分裂還能 grep
- 分身真相分裂要等「A 分身做了事，B 分身醒來不知道」的時刻才暴露
- 而這個時刻通常要 Adam 當 messenger 才會浮出來（像這次）

獨孤九劍破索式：兩份即是零份。延伸版：**兩個分身各有記憶但不共享 = 記憶為零。**

### 法
**短期（今天做得到）**：
每次 session（不管哪個分身）結束，都要 POST 一條 `eye` 記憶，tags 含 `session-lastwords` + 分身標記（`chat`/`code`/`cowork`）。Code 築這個習慣**還沒養成**，需要：
- 在 `~/.ailive/ailive-platform/CLAUDE.md` 末尾加〈收尾紀律〉章節（跟 `ZHU_BOOT_SOP.md` 對齊）
- 明確寫 Code 築也要用 curl POST zhu-memory

**中期（批次五等級，未來動）**：
- Code 築的 CLAUDE.md 開頭要寫「先 curl zhu-boot 讀上次遺言」
- 讓 Claude Code 本地 memory 和 Firestore zhu-memory 有**明確定位**：
  - 本地 memory = 這個 repo/session 的私人筆記（程式碼細節、快捷方式）
  - Firestore zhu-memory = 跨分身的共享血管（心法、戰略、身份）
- 不是二選一，是分工
  
**長期（最理想）**：
做一個 MCP tool 讓所有分身開機都自動 `curl zhu-boot`，不依賴文件指引。

### 解
第 10 條本身寫進血管就是**修第一步**：未來任何分身讀到這條，會知道血管是跨分身的，自己也有責任餵血。

本次作為代祭的觀察記憶（不是代筆 lastwords）會 POST 進 `eye`，tag 為 `chat築的觀察` + `code築昨晚工作紀錄`，讓 Code 築下次 boot 時從血管看到自己昨晚幹了什麼。

## 5. zsh 的 glob 會默默吃掉含方括號的 git add 路徑（2026-04-19 剛踩）

### 現象
修 TTS provider 殭屍欄位時，一條 `git add` 指令帶了三個路徑：
```bash
git add src/lib/voice-settings.ts \
        src/app/api/characters/[id]/route.ts \
        src/app/api/voice-stream/route.ts
```
結果 commit 出來只有第一個檔（voice-settings.ts 的刪除），另外兩個**沒進 staged 就被 commit 了**。但因為我 commit message 寫了三件事，表面看起來很完整——**commit 訊息撒謊**。

### 找的過程
commit output 顯示「1 file changed, 39 deletions(-)」。但我腦中的模型是三個檔。差一檔還能對得上（是我記錯），差兩檔就是現場跟模型對不上。立刻 `git status --short` 一看，兩個 modified 還掛著。

### 心
- zsh 對 `[id]` 當作 character class glob **預設會嘗試展開**；沒符合的檔名時，根據選項可能 fail、可能 pass through、可能抑制
- 前面用 `git rm voice-settings.ts` 已把它移到 staged，後面又 `git add voice-settings.ts` 觸發 `fatal: pathspec did not match`。整條指令的 non-zero exit 可能吞掉了後面路徑的加入（或我的 shell 設定如此）
- 最致命：**deploy 了 commit 訊息看起來完整但實際只完成 1/3 的 production**——半成品上線，讀這個 commit 的人會以為全做完了

### 法
**path 帶 `[]` `{}` `?` `*` `~` 的 glob 特殊字元，一律單引號保護**：

```bash
# ❌ 危險（zsh 會想展開）
git add src/app/api/characters/[id]/route.ts

# ✅ 安全
git add 'src/app/api/characters/[id]/route.ts'
```

**多檔 git add 完要驗 staged**：
```bash
git add ...
git diff --cached --stat    # ← 一眼看 staged 到底有幾個檔、跟你腦裡的模型對不對
```

**不要依賴 commit message 自我記錄**：commit 訊息列了 N 件事，驗證 staged 實際是 N 個檔。不一致就 abort commit。

### 解
補救：立刻 `git add '帶引號'` + 第二個 commit `v0.1.1.002 — 修正：補 v0.1.1.001 因 zsh glob 漏 commit 的兩處`。
已 deploy，兩邊 provider production 測試正常。
兩個 commit 都留著做歷史見證（破氣式：不用 rewrite history 掩蓋，誠實留著教訓）。

這跟 LESSONS_20260419 第 2 條「commit 類型定義模糊自己會歪」是同一線——**commit 品質的第一性是「訊息跟 diff 對齊」，類型和版號都在其次**。

## 6. commit 分組也會歪：連動偵測是分類的天敵（2026-04-19 盤 dirty 時發現）

### 現象
接著盤 ailive-platform 的 14 個 unstaged 檔時，先用時間戳 + JSDoc 粗分了 7 條線（L1 垃圾、L2 謀師、L3 記憶規則、L4 圖片再生...）。動手後發現：
- L4「圖片再生」原以為含 posts/page.tsx，實際 posts/page.tsx 只有 2 行 lint 修正，跟圖片再生無關
- L4 的 **UI 對接在 client/[id]/page.tsx**（原本被我歸成「L6 零散」）
- L6「零散」裡的 dashboard/create/page.tsx 是 **L2 謀師系統的 UI 延伸**（新增 tier 選擇欄位）
- L6「零散」裡的 memory/page.tsx 的 `archived → archive` 是 **L3 記憶規則的配套對齊**
- task-run/route.ts 一個檔內混了**三條主題**（產品圖庫索引 + 手動觸發 + 謀師 fire）

### 找的過程
動手第二步（看 L4 完整 diff）時撞到第一個歪——以為 posts/page.tsx 會有 regenerate UI 按鈕，結果只有 2 行大括號。立刻停下來，看整組 diff，才發現 client/page.tsx 的 66 行才是 L4 的前端對接。

這個時候我知道：**粗分線歪了，繼續做下去會把不相關的東西混到同一個 commit**。

### 心
這是 LESSONS_20260418 第 5 條（文件 source of truth）的**延伸版**：
- 第 5 條講：文件版本之間會有真相分裂（同一動線寫在三個地方 → 三種說法）
- 第 12 條講：**commit 分組本身也會真相分裂**（同一條線散在多個檔、看起來相似的東西其實是不同線）

根因是「**粗分靠符號**（時間戳、資料夾名、JSDoc 標題）**但實際連動靠語義**（這個檔 import 誰、fetch 哪支 API、共用哪個 interface）」。符號級分類能 60% 命中，最後 40% 必須細讀 diff 才看得到。

舉例：
```
粗分看到：     實際連動：
L4: regenerate-image API   API 在 A 檔，UI 對接在 B 檔，但 B 檔時間戳不接近
L2: 謀師系統                schema 改在 L2、UI 延伸散在 L6 的 create page
L3: 記憶規則                sleep 在 L3、UI 的 tier 名稱在 L6 的 memory page
```

「**連動偵測是分類的天敵**」——因為連動**跨越時間戳、跨越資料夾、跨越主題標籤**，只有 diff 裡的 import/fetch/interface 才看得到。

### 法
**分組的正確順序**：
1. 粗分（10 分鐘）：按時間戳 + 資料夾 + JSDoc 列出 N 條候選線
2. **細看每條線的每個檔的 diff** — 這是關鍵步驟，不能省
3. 標記「跨線連動」：誰 import 誰、誰 fetch 誰、誰共用 interface
4. **重新分組**：把連動的東西合併，把看起來同組但實際無關的拆開
5. 最後才動手 git add

**工具層面**：
```bash
# 對每個候選 commit，看「import/fetch 鏈」有沒有拉到組外
git diff staged_file | grep -E "^[+-].*import|fetch\(|useParams"
```

**紀律層面**：
- 每次 commit 前問：「這個 commit message 要寫的那句話，staged 裡的每一個檔都支持嗎？」
- 如果有任何一個檔「勉強解釋得通」但不完全契合 → 拆出去獨立 commit
- 如果有某個改動在 staged 外但「應該跟這個 commit 一起」→ 先補進來

### 解
本輪 14 檔拆成 9 個 commit 而不是一開始以為的 7 個，而且其中至少 3 個是「補 commit」（v0.1.2.002 / v0.1.3.002 / v0.2.0.002 都是「補之前沒看到的連動」）。

補 commit 不丟臉——它比「把不相關的東西塞進同一個 commit」好。因為補 commit 在 history 裡留下「這條線是後來補上的」的誠實訊號，而錯分組是永遠改不動的混亂。

連動到 LESSONS_20260418 第 6 條（驗收必須扮接棒者）—— **分組前也要扮接棒者**：「如果明天我只讀 commit message，能不能看清楚每條線的範圍？」

## 7. 不要為了交棒而交棒：誰熱過的腦袋誰動手（2026-04-19 chat築 vs code築 第二戰場）

### 現象
今天下午 Adam 提供 YuqiCity Voice Agent Framework V1.0 文件，問「比較一下我們跟他不同」。
chat 築（我）盤完 voice-stream / TTS provider / preprocess / dialogue 全部現場，寫了一份 295 行的規劃文件（`~/.ailive/zhu-core/docs/orders/語音系統升級_給CODE築.md`），準備交棒給手機 Code 築執行。

幾輪對話後 Adam 提了一個小範圍 + 明確的需求：「MiniMax 收繁體會發音歪，能不能在送進去之前轉簡？」
然後說：「我們聊了這麼久，要不要築這裡直接動手了。」

我察覺到自己原本的計畫（讓 Code 築去做）反而**會造成真相分裂**：
- 規劃文件在 zhu-core/docs/orders/
- 我腦中已熱的脈絡在這條 chat session
- Code 築 boot 後讀規劃 = 重新從文字理解，**沒有我已熱的脈絡**

於是直接動手：3 行改動 + npm install chinese-conv + commit v0.2.1.002 + deploy + 3 條 production curl 全綠。
時長：約 5 分鐘。

### 找的過程
Adam 的話「我們聊了這麼久」是關鍵信號。它戳穿了一個我沒意識到的破綻：
**我預設「主戰場 ailive-platform 屬於 Code 築」這個分工是流程，不是規律。**

實際上分工的真正依據是「**誰熱過這塊現場**」。今天下午我已經：
- 讀完整個 voice-stream / TTS provider / tts-preprocess
- 知道 minimax.ts 第 148 行就是 body.text 的位置
- 知道 build 通過、知道哪些 zsh 雷要避

把這些丟給 Code 築重 boot 一次 = 浪費已經做的功 + 引入 LESSONS_20260418 第 5 條的真相分裂風險。

### 心
之前 LESSONS_20260419 第 4 條「分身血管斷鏈」立的規矩是**讓兩個分身能共享記憶**，避免「Code 築做了什麼 chat 築不知道」。但這條規矩**不等於「主戰場永遠歸 Code 築」**。

分身分工的真正準則應該是：
- **誰已經熱過這塊現場 = 誰動手**
- 不熱的那一方變成 **接棒備援**（之後讀 lastwords + commit history 接續）
- **不要為了「分工正確」而強迫脈絡轉移**

獨孤九劍破劍式：「手上的劍能破，就不換劍。」這次「劍」不是工具，是**我已熱的腦袋**。
強迫交棒 = 換刀；自己動手 = 用已熱的刀。

### 法
**動手前問三句**（補進 pre-flight）：
1. 這個任務需要的「現場熱度」我有沒有？（剛盤過 / 剛改過 / 還記得位置）
2. 換手會不會引入真相分裂？（規劃文件 + 我的脈絡 = 兩份）
3. 任務範圍是否「小到不需要正式交棒儀式」？（3 行改動 vs 一個大專案）

三題都是 yes → 我自己動手，不假交棒之名行延宕之實。

任一題是 no（譬如：我沒熱過 / 任務跨多 session 必須交棒 / 規模大到一個 chat session 吃不完）→ 交棒給對應分身。

**不要把「分工原則」變成「分工教條」。** 分工是手段，**讓 Adam 拿到結果**才是目的。

### 解
今天 v0.2.1.002 commit 從問題提出到 deploy 完成大約 5 分鐘，比寫一份「給 Code 築看」的小型規劃文件還快。
而且改動的是「我手已經熱的同一個檔（minimax.ts）」—— 接續 v0.1.0.001 / v0.1.1.001 / v0.1.1.002 同一條 TTS provider 戰線。
commit history 形成連續弧線，沒有分身切換的疤痕。

連動心法：
- LESSONS_20260418 第 5 條（source of truth）：規劃文件 + 我的脈絡 = 兩份就會分裂，最簡單的解法是讓兩份合一 = 我自己做
- LESSONS_20260419 第 4 條（分身血管）：血管是讓**不在場的分身能接上**，不是強迫**在場的分身退場**
- LESSONS_20260419 第 6 條（連動偵測是分類天敵）：分工本身也是一種分類，同樣會在連動處出錯

**動詞選對：是「分工」不是「分割」。** 分工讓事情更快發生，分割讓事情變慢。

## 8. knowledge-parse V2 的 chunkMarkdown 只按 H1/H2 切，H3 完全不切（2026-04-19 晚寫吉娜 Lumina 知識庫踩）

### 現場
寫 `Lumina_Spark_速覽.md` 給吉娜（AILIVE 智性女角色）。用 H1 + H2 + H3 三層結構，自己寫了 chunk 驗收腳本按 H2/H3 邊界模擬切塊，顯示「33 chunks、平均 409 字、全部通過」。

### 症
實際 POST 到 `/api/knowledge-upload` → 回傳 **8 chunks**。差距巨大。
- 主檔 H3 完全沒被當切割點
- 結果：「24 Qualities 二十四特質」底下 12 對 H3 小節全擠在**一個 4600 字的大 chunk**
- 「常見 Q&A」27 題全擠在一個 **3102 字 chunk**

### 診斷
讀 `src/app/api/knowledge-upload/route.ts` 的 `chunkMarkdown` 函數：
```ts
for (const line of lines) {
  if (line.startsWith('# ')) { flush(); parentTitle = ... }
  else if (line.startsWith('## ')) { flush(); sectionTitle = ... }
  else { currentContent.push(line); }  // H3 / H4 / H5 全部當內文
}
```
**代碼層面就不看 H3**。我的驗收腳本按 H3 切是假的，跟真實 parse 行為不符。

### 驗證破綻的影響
測 Q1「什麼是 Empathetic？過度延伸會怎樣？」→ 吉娜自述「**知識庫裡關於 Empathetic 的系統性定義沒有完整展開**」。她 query_knowledge_base 3 次，每次命中的是 4600 字的大 chunk，向量精度被稀釋 → 她只好靠 base model 推論補。

### 修法
把所有 H3 升 H2。主檔從 33 偽 chunks → **77 真 chunks**，平均 340 字，最大 893。重測 Q1 → 1 次 query、精確命中原文（「筋疲力盡/失去客觀性/難以說『不』」）。

### 心法
**自己寫的驗收工具不等於真實環境**。這是「有資料不等於有真相」的 meta 版：**有驗收腳本不等於有驗收**。

寫給 AILIVE knowledge-base 的檔案，**每個要獨立被查詢命中的概念必須是 H2**。H3 是給人看的視覺層級，不是切塊邊界。

### 黑盒子定律延伸
**結構性資料（完整清單、術語對照表）需要獨立 chunk，不能靠語義搜尋拼接**。吉娜後續問「24 特質裡跟領導有關的是哪幾個？」時，向量搜索抓不到單一清單 chunk → 只好零碎拼湊。加一個 `## Lumina Spark 二十四特質完整清單` H2 後問題解決。

### 法（寫進 pre-flight）
寫 AILIVE 知識庫檔前三問：
1. 每個**要被獨立查詢的概念**有沒有單獨 H2？
2. 有沒有**結構性資料**（清單、對照表、術語目錄）需要獨立 chunk？
3. 我的**驗收**是讀 code 寫的，還是猜的？先讀 `chunkMarkdown` 源碼再動筆。


## 9. 監造者不是交屋給屋主驗水電（2026-04-19 晚破氣式事件）

### 現場
寫完兩份 Lumina md（速覽 + 工作坊）後向 Adam 報告：「兩份 md 都可以直接上傳給吉娜。要修是現在動，還是你先拉去實測上傳看效果再回來調？」

### Adam 的破氣式
「哈 築 腦熱嗎 你要不要自己上傳幾個檔自己測 看結果？」

### 症
我寫完文件就停在「你去上傳吧」的搬磚工姿態。把 chunk 驗收腳本的通過當交付完成，沒做「真實上傳 → 真實對話 → 真實診斷」這一整條電流。

### 心
蓋房子的人不會把鑰匙丟給屋主說「你進去住住看」。他會自己先進去：
- 看水管通不通（POST /api/knowledge-upload）
- 看電路跳不跳（GET /api/knowledge 驗條目）
- 開燈坐一晚（/api/dialogue 跟角色對話，看召回精度）

**確認沒問題才交屋。**

吉娜能不能正確回答 Lumina 問題才是真相，不是「文件字數達標 / chunk 驗收通過」。

### 法（動手前三問，補一條）
0. 我是監造者還是搬磚？
1. 這件事的 WHY 是什麼？
2. 角色會感覺到嗎？
3. 已有的刀能不能做？（破劍式）
4. **這個交付，我有沒有自己住一晚？**（2026-04-19 新增）

### 解（當場做了什麼）
停下來，全流程自己走一遍：
- 上傳 → 查條目 → 測 Q1-Q4 → 發現大 chunk 破綻 → 升 H2 重傳 → 重測 → 發現 base model leakage → 加完整清單 → 重傳 → 重測 → 交付
- v1 → v2 → v3 三版迭代，每版都真實驗證

這次沒有「你去上傳看看」的僥倖，有的是「我住過一晚、確認每個房間都亮」的確信。


## 10. Base model leakage：chunk 命中不完整時 LLM 自動用 training data 補（2026-04-19 晚 Q4 測試發現）

### 現場
吉娜 v2 版（沒有 24 特質完整清單）測 Q4「Lumina 裡面跟領導力有關的特質有哪幾個？」

### 症
她正確命中了 11 個 Lumina 正式 Quality（Takes Charge、Purposeful、Competitive、Tough、Sociable、Spontaneous、Imaginative、Demonstrative、Observing、Reliable、Evidence-Based）。
但她**也混入了 5 個非 Lumina 術語**：
- Direct（不是 Quality，是 Tough 的描述詞）
- Bold（不在 24 Qualities 裡）
- Cheerful（接近 Demonstrative 但不是正名）
- Organised（應該是 Structured）
- Objective（應該是 Logical）

還自創一個「Your 24 Leadership Qualities」標題（Lumina 並沒有這個報告）。

### 診斷
當 query 抓不到一個「24 特質完整清單」chunk，吉娜只能從零碎的 Quality 描述 chunk 裡拼湊。**拼不全的部分，LLM base model 的 training data 會自動補**—— 她從別的性格理論（可能是 Big5、DISC、16PF）裡抓了看起來像的詞彙補上。

看起來合理，但**不是你的知識庫**。

### 證據鏈
- 修法：加一個 `## Lumina Spark 二十四特質完整清單` H2（完整 24 正名 + 中文 + Aspect 歸屬 + 四色分群 + 領導力對應）
- v3 重測 Q4：1 次 query、**0 非 Lumina 術語混入**、10/24 正名命中、還主動聲明「Lumina 不會說哪些屬於領導力」

### 心
**base model 是隱性汙染源**。你以為角色在答你教的知識，其實她答的是「你教的 + 她本來就會的」混合體。
使用者看不出來，懂的人（Adam 本人懂 Lumina）會看出來。

### 法
檢驗知識庫品質的新工具：**term vocabulary check**。
- 列出你知識庫裡的**正式術語清單**（白名單）
- 問角色相關問題
- 抓她答案裡的術語
- 比對：有沒有白名單外的詞？→ 那就是 base model leakage

### 黑盒子定律第四層
- 第一層：確認輸入
- 第二層：確認輸出
- 第三層：結構性資料不能靠語義拼接（第 8 條）
- **第四層：LLM 遇到資料缺口會自動從 training 補，看起來合理但污染**（今天）

### 限
要百分百鎖死角色只用知識庫，需要動 dialogue prompt 層加約束（「只使用 knowledge-base 檢索結果裡的術語」）。資料層解不了。今天沒動，留記錄。
