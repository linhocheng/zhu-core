# LESSONS 2026-06-17 · ailivex v10.0.3 — 判別式 gate + 別名系統

## L1：LLM 的「交棒」是事件視角，不是接收者視角
- 現象：「聖嚴法師怎麼看？」→ 聖嚴 agent 和達賴 agent 的 gate 都回 handoff=true
- 根因：LLM prompt 問「這是不是交棒？」→ 兩個 agent 都看到交棒事件，都答 yes。LLM 不知道「我是誰」vs「我是被交棒的那個人」
- 下次：gate prompt 的核心問題要是「這句話期待你開口嗎？」不是「這是不是交棒？」；明確點名的部分根本不用問 LLM
- 對應 feedback：天條「確定性的工作用程式，不要丟 LLM」

## L2：deterministic pre-check 比修 prompt 穩一個數量級
- 現象：修 gate prompt 仍有機率出錯（LLM 是機率引擎）；判別式是 100%
- 根因：「我的別名在這句話裡嗎？」是字串比對，不是語意理解，本不該丟 LLM
- 下次：所有 gate/resolver 先問「這題確定性高嗎？」確定的先程式處理，模糊的才丟 LLM
- 對應 feedback：天條「確定性的工作用程式，不要丟 LLM」

## L3：別名是角色的「被叫到偵測器」，要跟著角色建立時一起存
- 現象：原本沒有 aliases 欄位，gate 只靠 LLM 語意辨識名字，靠不住
- 根因：schema 缺欄位 → 判別式沒有原料 → 只能退化成 LLM
- 下次：建新角色時就讓 LLM 建議 aliases；繁簡互轉程式自動補；conditional alias（法師限單法師時）是下一步
- 對應 feedback：（新洞察，可能升 feedback memory）

## L4：「handoff」這個概念本身會讓 LLM 停在旁觀者視角
- 現象：Adam 一針見血：LLM 很容易停在「這是一個交棒事件」，不往下問「交棒給誰，那個人是我嗎？」
- 根因：術語本身有歧義——從旁觀者看是 handoff event，從接收者看是 addressed to me
- 下次：gate schema 改成 target_type(self/other/group/none/unclear) + local_action(speak/yield/silent/wait) + reason；這樣 LLM 被強迫站在自己視角
- 對應 feedback：（Adam 的架構設計，v10.0.4+ 可以升級）

# LESSONS 2026-06-17（第二 session）· ailivex v0.2.0 版本指派 + v12.0 讀網址工作臺

## L5：沙推≠驗證，這次做對了——動碼前翻套件源碼驗 API 存在
- 現象：Req 2「即時語音通話中讀網址」依賴「LiveKit Agents 1.5.1 能 mid-call 暫停聽 + 改 context + 收前端 data」。
- 根因：這是可量測前提。6/16 embedding 那次沙推翻車（cos=1.0）的教訓這次內化了——沒假設「應該能暫停」，直接 grep 裝好的 `livekit/agents` 套件源碼，四個原語全部找到：`session.input.set_audio_enabled()`、`session.interrupt()`、`agent.update_instructions()/update_chat_ctx()`、`register_rpc_method`/`data_received`。計畫從此有根。
- 下次：凡計畫依賴某框架/API「會這樣反應」→ 先翻它的安裝源碼或最小腳本驗，再寫計畫。
- 對應 feedback：`feedback_sandtable_not_validation`（這次是正面執行，不是踩雷）

## L6：CLI 無瀏覽器時，用自簽 session cookie 做線上 API 端到端自測
- 現象：Req 1 改了 token route 的版本決策，但我在 Code CLI 沒瀏覽器點不了 UI。
- 根因：認證鏈（cookie→middleware→route）+ 派工結果（agentName 藏在 LiveKit JWT）都能在無 UI 下驗——只要有 SESSION_SECRET 自己照 auth-session.ts 簽一個 admin/user cookie，curl 打線上端點，再 base64 解 JWT 的 roomConfig 撈出實際派的 agentName 對賬。11/11 全過。
- 下次：改了 auth-gated route 又沒瀏覽器 → 自簽 cookie + 打線上 + 解回傳憑證裡的真相，比「等 Adam 點」快也更實。誠實標注「UI 點擊未驗，但 UI 就是呼叫這些已驗端點」。
- 對應 feedback：`skill_local_replay_over_remote_wait`（同精神：能本機/離線驗就不等遠端）

## L7：新功能寄生在「剛上線的生產預設版」→ 開新版不改 live，用版本指派當 rollout gate
- 現象：Req 1 把全域預設語音設成 v3（這 session 剛上線）；Req 2 要在 v3 上加讀網址，但 source_intake 是未驗證新行為，import 出錯會讓生產預設**開機就掛**。
- 根因：ailivex 鐵律「實驗不碰 live 版」。v3 一變預設就等於 live。
- 下次：開 v12（=v3+source intake），live v3 不動；測試用 Req 1 的 per-user 指派把測試帳號指到 v12，驗過再翻全域預設 v3→v12。Req 1 的指派機制正好天然就是 Req 2 的安全 rollout gate——兩需求互相成全。
- 對應 feedback：`feedback_flagged_risk_must_be_verified` + ailivex append-only 紀律
