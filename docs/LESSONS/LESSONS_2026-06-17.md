# LESSONS 2026-06-17 · ailivex v10.0.3 — 判別式 gate + 別名系統

## L1：LLM 的「交棒」是事件視角，不是接收者視角
- 現象：「聖嚴法師怎麼看？」→ 聖嚴 agent 和達賴 agent 的 gate 都回 handoff=true
- 根因：LLM prompt 問「這是不是交棒？」→ 兩個 agent 都看到交棒事件，都答 yes。LLM 不知道「我是誰」vs「我是被交棒的那個人」
- 下次：gate prompt 的核心問題要是「這句話期待你開口嗎？」不是「這是不是交棒？」；明確點名的部分根本不用問 LLM
- 對應 feedback：天條「確定性的工作用程式，不要丟 LLM」

## L2：deterministic pre-check 比修 prompt 穩一個數量級
- 現象：修 gate prompt 仍有機率出錯（LLM 是機率引擎）；判別式是 100%
- 根因：「我的別名在這句話裡嗎？」是字串比對，不是語意理解，本不該丟 LLM
- 下次：所有 gate/resolver 先問「這題確定性高嗎？」確定的先程式處理，模糊的才丟 LLM
- 對應 feedback：天條「確定性的工作用程式，不要丟 LLM」

## L3：別名是角色的「被叫到偵測器」，要跟著角色建立時一起存
- 現象：原本沒有 aliases 欄位，gate 只靠 LLM 語意辨識名字，靠不住
- 根因：schema 缺欄位 → 判別式沒有原料 → 只能退化成 LLM
- 下次：建新角色時就讓 LLM 建議 aliases；繁簡互轉程式自動補；conditional alias（法師限單法師時）是下一步
- 對應 feedback：（新洞察，可能升 feedback memory）

## L4：「handoff」這個概念本身會讓 LLM 停在旁觀者視角
- 現象：Adam 一針見血：LLM 很容易停在「這是一個交棒事件」，不往下問「交棒給誰，那個人是我嗎？」
- 根因：術語本身有歧義——從旁觀者看是 handoff event，從接收者看是 addressed to me
- 下次：gate schema 改成 target_type(self/other/group/none/unclear) + local_action(speak/yield/silent/wait) + reason；這樣 LLM 被強迫站在自己視角
- 對應 feedback：（Adam 的架構設計，v10.0.4+ 可以升級）
