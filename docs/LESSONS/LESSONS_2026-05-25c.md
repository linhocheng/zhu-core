# LESSONS 2026-05-25c · ANEWS image pipeline 打通 + babysit 節制

## L1：Firestore Admin SDK transaction — reads 必須全部在 writes 之前
- 現象：image worker transaction 用 `tx.get(taskRef)` 寫後再 `tx.get(query)` → 全部 500 "Firestore transactions require all reads to be executed before all writes"
- 根因：Admin SDK（server-side）transaction 硬規則：reads 必須完成才能開始 writes；Client SDK 無此限制，常見範例誤導
- 下次：transaction 內一律先 `Promise.all([tx.get(A), tx.get(B)])` 再寫，不按流程順序交錯
- 對應 feedback：記憶會說謊——「transaction 範例是對的」不代表用的是 Admin SDK

## L2：babysit 過於積極 = 幫倒忙，Cloud Tasks + babysit concurrent fire 消耗 repairAttempts
- 現象：babysit 每 20s 就 fire section-qa；Cloud Tasks 也在 fire；同一 section 同時跑 2 個 worker → repairAttempts 累積 → needs_repair
- 根因：babysit 設計是「Cloud Tasks 掛了的 fallback」，但實際上 Cloud Tasks 正常在跑，babysit 只是添亂
- 下次：fallback 用 2 條件擋：(1) node 創建已過 2 分鐘（Cloud Tasks 早就有機會了），(2) 自身 5 分鐘 cooldown per section
- 對應 feedback：「不要為了「安全多一層」加會壞的元件」

## L3：needs_repair propagation 應以 main article 進度為準，不是 failing article 本身
- 現象：sub_b section 寫失敗 → needs_repair → 整個 issue kill，但 main 已 polish_done
- 根因：第一版 Bug 2 fix 只查 failing article 狀態，但 failing article 就是在 section_writing，不滿足條件
- 下次：needs_repair handler 要查 main article 是否已 stitching_done+，是就放行繼續
- 對應 feedback：「解決問題還是繞開根本問題」——第一版 fix 方向對但查的是錯的 article

## L4：Cloud Tasks 不配信 image_tasks（planned 卡著）根因是 IMAGE_DRY_RUN 只在 .env.local 而非 Vercel prod env
- 現象：issue=image_generating 但全 6 個 image_tasks 停在 planned 超過 5 分鐘
- 根因：IMAGE_DRY_RUN=true 只在 .env.local，Vercel prod 沒有這個 env；Cloud Tasks 打 prod Vercel 的 image worker 可能因為缺 env 或其他原因沒有正確執行
- 下次：手動 fire 驗是否是 env 問題；把關鍵 dry-run flag 加進 Vercel env
- 對應 feedback：動手前驗證 secret/env 是否有效
