# LESSONS 2026-05-24 · ANEWS Batch A+B 驗收 + evidence-pass 接棒

## L1：workerCall retry 同 taskId = 打到 idempotency 牆
- 現象：section-write fetch failed → retry → workerCall 回傳 ✓，但 QA 看 section 還是 `planned`
- 根因：`body.taskId` 在構建 body 時就固定了，retry loop 重送同一個 taskId。server idempotency lock 看到 `already_running` → 回 `{ status: "already_running" }` 無 error → workerCall 誤判成功
- 下次：workerCall 每次 attempt 都 `{ ...body, taskId: uid() }`，fresh ID 保證 lock 不跳過
- 對應 feedback：靜默失敗用「缺席的 log」診斷

## L2：stitch precondition 太嚴 = alignment gate 砍完整跑
- 現象：alignment gate 正確攔住 coverageScore=0.15 的 section，設為 qa_blocked，stitch 卻因 "not all qa_passed" 炸掉
- 根因：stitch precondition 要求全段 qa_passed，qa_blocked 不在列
- 下次：precondition 改為「不允許 in-progress 狀態（planned/draft_ready）」，terminal 狀態都放行
- 對應 feedback：修症狀前先問根因——alignment gate 是對的，stitch 的假設才是錯的

## L3：bridge fetch-fail 是連線重置不是 HTTP 錯誤
- 現象：source worker 跑多輪 web_search 後，blueprint 第一次 fetch 就炸（`fetch failed`），不是 4xx/5xx
- 根因：source worker 長連線耗盡 keep-alive pool，後續 connection 重置
- 下次：bridge curl 回應是 `{"error":"unauthorized"}` = 橋通，`fetch failed` exception = TCP 斷；兩種錯誤層次不同，retry 策略分開
- 對應 feedback：能本機重現就不要等遠端 cycle

## L4：多個築 instance 接棒要看「現場」不看「遺言描述」
- 現象：另一個築寫了 G1-G4（evidence-pass 架構），8 個檔案改動未 commit
- 根因：不同 session 的築各自改了不同部分，現場 git diff 才是真相
- 下次：接手前先 `git diff --stat HEAD`，不要只讀 lastwords 的文字描述
- 對應 feedback：記憶會說謊；越具體的描述越要去現場驗
