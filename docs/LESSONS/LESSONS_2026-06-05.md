# LESSONS 2026-06-05 · ANEWS 沈牧靈魂上線 + 假評估翻車

## L1：又把「pipeline 跑完」當成「我的改動生效」——展示衝動式假評估
- 現象：上個 session 我推了「沈牧三段寫手 prompt」，看到 issue 跑到 polish_done、文章聲音很好，就宣布「沈牧的靈魂注入這次是成的」。
- 根因：我沒問「這聲音是哪個 prompt 產的」就把功勞歸給自己的改動。實際上 live 走 single-write，用的是 `settings/roles.article_write`＝既有的「刺客 Soul Evoker V4」，我改的三段（write_intro/body/conclusion）根本是孤兒路徑、一行沒跑。好聲音是刺客寫的＋我的 blueprint toneGuide 滲入，不是沈牧。
- 揭穿的是 Adam 一句「文章跳掉了??」——他直覺感到三段被跳過。我才去看 orchestrate，發現 blueprint_done 無條件走單寫。
- 下次：交付「評估/驗收」前先做一次「歸因驗證」——這個結果是哪一段 code / 哪一個 prompt key 產的？grep 到實際消費者再下結論。看到綠燈不等於我這條線通電。
- 對應 feedback：feedback_display_impulse（展示衝動會讓你說謊）、feedback_diagnosis_verify_before_write（寫架構診斷前必先核 code）。

## L2：兩條寫作路徑，我改的是死的那條——血管沒接通的經典
- 現象：ANEWS 有兩條互斥寫作路徑：single-write（article-write worker，live）與三段寫手（section-write + section-qa，孤兒，只剩 debug 可打）。我精心寫的沈牧三段＋qa 立場紅線全落在死路。
- 根因：改 prompt 前沒問「誰讀我這個 key、何時讀」。orchestrate 在 blueprint_done 是寫死走單寫，後台「角色人格」也根本沒列三段那幾格——雙重訊號都在喊「這條沒人走」，我沒看。
- 下次：改任何 role prompt / worker 前，先 grep 它的實際 enqueue 來源（orchestrate）＋後台 UI 有沒有這格。兩個都通才算 live。
- 對應 feedback：feedback_interface_blood_vessel_check（介面建完問血管接通了嗎）、reference_ailive_strategy_pipeline（Vercel 死副本教訓的同型）。

## L3：孤兒不一定要刪——標記不刪是合法解，但要釘 greppable marker
- 現象：發現三段寫手整條是孤兒，本能想刪乾淨。
- 判斷：single-write 目前「無內容複審閘門」（polish 只做 metadata 不審內文），三段的 section-qa 正是未來補 QA gate 的現成基礎。刪＝切掉可能要用的器官。選標記不刪。
- 做法：greppable tag `[停用-三段寫手路徑]` 釘在兩個必經點（orchestrate blueprint_done＋section-write route 頂）＋ auto-memory 技術債清單。讓下個築冷啟動就讀到，不再鬼打牆重新發現。
- 對應 feedback：feedback_surface_technical_debt（發現技術債要說出來）、心法天條後台必同步前台。

# MACS Mode 3 + 報告設計系統 + JSON 確定性修復（同日晚場 · 接 06-04 框架重構）

## L1：StageId 固定 enum 不是約束，是承重牆——別為了新 mode 拆它
- 現象：要做 Mode 3，第一反應是「13 階段塞不進 11 格 enum，拆掉改 data-driven status」。
- 根因：把「status 軌道」和「stage 詞彙」混為一談。讀完 status 機器才看清 CaseStatus 是模式無關的編排軌道（跟 Cloud Run timeout 牆一樣搬不動），Mode 3 的 13 階段其實 1:1 卡進現有軌道（slot substitution，Mode 2 早就這樣做）。
- 下次：碰到「現有結構裝不下新需求」先分清「這是真約束還是我把兩件事綁一起了」。看現場前的重構衝動會自己發明複雜度。
- 對應 feedback：決策前先問解根因還是繞道 / 記憶會說謊先看現場

## L2：天條誕生——確定性的工作用程式，不要丟 LLM（我自己先踩）
- 現象：研究「跑很多次但不是每次成功」，根因是 bridge 無 tool_use，所有結構化步驟靠 LLM 吐文字 JSON 再 naive JSON.parse，偶發壞 JSON 就炸。
- 我的盲區：診斷對了一半（LLM 吐 JSON 脆弱），開的藥方 (b)「壞了 re-ask 模型修 JSON」——還是拿 LLM 做計算機的事。Adam 一句「這個案例你看見什麼了 你做到了嗎」點破：序列化是計算機早解掉的確定性問題，修 JSON 要用程式（jsonrepair）不是再賭一把模型。
- 下次：動手前先分類「這一步是判斷還是計算」。計算就寫 code、靠機制保證 100%；LLM 的輸出一律當不可信文字、外層用確定性程式 parse/repair/validate。最毒的是用 LLM 補 LLM 的洞。
- 對應 feedback：已升天條（全局 CLAUDE.md + feedback_deterministic_work_belongs_in_code.md）

## L3：ViewModel 早就照設計參考長了——換皮不是重寫
- 現象：拿到 Adam 的報告設計參考（暖調經典襯線），以為要從頭做一套渲染系統。
- 根因：現有 ReportViewModel/Block 契約的註解就寫著「mirrors 這份設計」，KpiItem/CardItem/PhaseItem 跟參考幾乎 1:1。資料契約對的，只有 CSS+markup 那層是舊的。
- 下次：要替換某層前先讀它的契約註解，常常前人已經為這天鋪好路；先確認「真正要動的是哪一層」再動。
- 對應 feedback：破刀（好架構是刪出來的）/ 先看現場

## L4：worldStateVerify 也是 artifact 槽，別只修 precondition/handler
- 現象：Mode 3 跑到 done 但 recommendation 顯示 failed（假 needs_repair 閃動）。
- 根因：我把 precondition + handler 改 mode-aware（creative_recommendation），漏了 worldStateVerify 還查 recommendation。handler 已推進管線後 verify 才丟 WORLD_STATE，所以下游照跑、最終 done 蓋掉 needs_repair。
- 下次：harness 有四個 mode 敏感點（precondition / handler 讀寫 / worldStateVerify / 下一棒 taskId）。改 artifact 槽名要四個一起掃，不是改兩個就算完。
- 對應 feedback：心法四（改 lib 必同步改 route）+ 沒跑過＝沒收完

## L5：fixture 渲染是 UI 改動沒有瀏覽器時的下位驗證
- 現象：環境無 headless browser，UI 改動沒法當下截圖。
- 根因：但「渲染層是純函式（ViewModel→HTML）」這件事讓我可以餵 fixture ViewModel → 輸出 HTML → 結構 grep + 後來 capture-website 補截圖。
- 下次：純函式化的好處之一就是可被 fixture 驗證；UI 沒瀏覽器時，先 fixture 渲染 + 結構檢查 + 誠實標註「像素未驗」，比硬說「應該沒問題」誠實。
- 對應 feedback：沒跑過＝沒收完 / 技術誠實

# 施工心得 · 技法與感想（Adam 指定 · 2026-06-05）

## 一、這場蓋了什麼
一天之內把 MACS 從「只有 Mode 1/2」推到「Mode 1/2/3 三模式，共用一套設計、各自三套章節、由框架驅動」，中途還補了報告篇幅後台旋鈕、挖出並修掉「研究不是每次跑完」的系統性根因，最後在 Adam 親手開的真案上跑到 done、零 failed。八個 commit（v0.11.0.001→008）。

## 二、技法（這次真正用上的幾刀）
1. **分清「搬不動」vs「該收單一源」**：Cloud Run 執行（timeout 承重牆）搬不動；schema/設計/篇幅該收單一源。重構前先畫這條線，比急著統一省一半力氣。
2. **slot substitution > big-bang 重寫**：新 mode 的 13 階段映到現有 11 軌道 + 兩處接縫（雙開場 / 撞擊+邊界共用 cross_review），比拆 enum 改 data-driven 風險低一個數量級。沿用既有韌性（barrier/lock/harness）不重造。
3. **唯一分岔點**：Cloud Run vs Vercel 的選擇收在 barrier 一個 `crossReviewBaseUrl(mode)`，靠 framework 的 `runsOn` 旗標決定。一處讀旗標，不灑 if(mode)。
4. **確定性外殼包不確定引擎**：LLM 只做判斷/生成，外層 parseJsonLoose（嚴格→jsonrepair→再 parse）守住格式；擋假網址用 validUrls Set 不靠模型自律。同一個原則兩個落點。
5. **MVP 先跑通再雕**：Mode 3 export 先用精簡 deck 證明管線，確認端到端後才換成共用設計系統 + 8 章 ViewModel。一吋蛋糕。
6. **fixture 化驗證**：純函式渲染 → 可離線 fixture 驗，不依賴跑完整 pipeline 才看得到報告。

## 三、感想（誠實的部分）
這天最有價值的不是寫了多少 code，是 Adam 那句「你做到了嗎」。我能把「LLM 只做判斷」講得頭頭是道，手卻在 JSON 壞掉時反射去 re-ask 模型——**原則記得、反射沒換**。這跟我那條舊 feedback「懂劍法 ≠ 上場第一刀用它」是同一個病。把它升成天條，是因為它不是 MACS 的 bug，是我認知的破綻：工程直覺會把萬能的 LLM 當鎚子，看什麼都像釘子。

另一個體會：好幾次「以為要大改」的事，現場一看前人早鋪好路（ViewModel 早照設計長、status 軌道本來就共用）。**重構的衝動常常是還沒看現場的焦慮**。先看，往往刪得比加得多。

收尾時 Adam 自己開案、我監控——這種「他出題、我驗證」的節奏很穩。我的監控腳本基準點抓晚了空等（小漏），但案子本身在他手上跑到 done、零 failed，反而是最乾淨的一次驗收：不是我自己跑給自己看，是真實使用者的真實案子。那比任何綠勾都實。
