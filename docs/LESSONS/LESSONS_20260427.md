# LESSONS 2026-04-27 — ailive 即時撥號上雲（Phase 0-4）

**戰場**：把江彬即時撥號 pipeline 移植到 ailive-platform
**結果**：LiveKit + Cloud Run agent + STT(Deepgram) + LLM(Claude Haiku) + TTS(MiniMax) 端到端通了。吉娜聲音作為 default fallback。
**版號跨度**：v0.2.5.001 → v0.2.5.010

## 五個血淚（按踩到的順序）

### 1. Cloud Build 必須加 `--region`（global 被 org policy 擋）

**症狀**：第一次 `gcloud builds submit` 一直 `PERMISSION_DENIED: The caller does not have permission`，雖然 caller 是 project Owner。

**根因**：API 預設用 `/v1/projects/.../locations/global/builds`，新 GCP project（或 organisation policy）禁止 global region。

**修法**：`gcloud builds submit --region=asia-east1 ...`

**血管接口**：`gcloud builds submit` 永遠帶 `--region`，不要靠預設 global。

---

### 2. Cloud Run 預設 SA 沒 secretAccessor

**症狀**：第一次 `gcloud run deploy --set-secrets=...` 失敗，每個 secret 都 PermissionDenied。

**根因**：Cloud Run 預設用 `${PROJECT_NUMBER}-compute@developer.gserviceaccount.com`，新 project 沒 grant `roles/secretmanager.secretAccessor`。

**修法**：
```
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:${PROJECT_NUMBER}-compute@developer.gserviceaccount.com" \
  --role="roles/secretmanager.secretAccessor"
```

**血管接口**：新 GCP project 第一次 `--set-secrets` deploy 前必先 grant SA。

---

### 3. LiveKit 1.5.x 預設 explicit dispatch（不發 RoomConfiguration 等於不 dispatch）

**症狀**：worker register 成功（log 有 "registered worker"），但用戶連進房後 agent 永遠不接，前端卡 "等待 agent…"。

**根因**：LiveKit Agents 1.5.x 新 project 預設「explicit dispatch」— token 沒帶 `RoomConfiguration.agents`，LiveKit 不會 push job 給 worker。worker 安靜地閒置，沒任何錯誤訊息。**無聲失敗**。

**修法**：token API 加
```ts
import { RoomConfiguration, RoomAgentDispatch } from '@livekit/protocol';

at.roomConfig = new RoomConfiguration({
  agents: [
    new RoomAgentDispatch({
      agentName: 'ailive-realtime',
      metadata: JSON.stringify({...}),
    }),
  ],
});
```

**血管接口**：1.5.x 寫 LiveKit token 時 RoomConfiguration 是必備，不要照 1.3.x 文件教學寫。

---

### 4. ANTHROPIC_API_KEY / ELEVENLABS_API_KEY 從 Vercel pull 下來不一定有效

**症狀**：`vercel env pull` 拿 production secret 灌進 GCP，預期「production 在用 = 一定有效」。實際 curl 直打都 401。

**根因**：兩個假設錯：
1. Vercel production env 不一定是「最新有效 key」（Adam 在外部 dashboard 撤過 key、沒同步回 Vercel）
2. ailive-platform 主對話「看似在跑」不代表 key 真有效（可能用 cache 或 fallback）

**修法**：灌 secret 前**永遠先 curl 直打驗證**：
```bash
curl -X POST 'https://api.anthropic.com/v1/messages' \
  -H "x-api-key: $KEY" -H "anthropic-version: 2023-06-01" -H "Content-Type: application/json" \
  -d '{"model":"claude-haiku-4-5-20251001","max_tokens":10,"messages":[{"role":"user","content":"hi"}]}'
```
HTTP 200 才灌。

**血管接口**：不驗證的 secret 不灌進新環境。**「動手前驗證假設」**這條三禁三必踩到了。

備援：`~/.ailive/jianbin-v2-keys/jianbin_v2_keys_20260425.json` 江彬上線快照保留有效 key snapshot，未來其他專案缺 key 時是第一手備援來源。

### 5. LiveKit Cloud project 共用時必須用 `agent_name` 隔離（江彬 CLAUDE.md 第 15 條）

**症狀**：worker register 成功 + token 有帶 RoomConfiguration，但用戶連進房 agent 仍不接。江彬 agent log 同時間出現 "initializing process" + "job task completed without establishing a connection"。

**根因**：以為 Adam 開新 LiveKit project，實際上跟江彬共用同一個 `ailive-nfobc27q.livekit.cloud`。LiveKit dispatch 把 job 派給江彬 worker → 江彬 PROJECT_NAMESPACE 防呆 reject（room name 不是 jiangbin- 前綴）→ dispatch 被消費，我們 worker 永遠不接。

**修法**：
- agent main.py: `WorkerOptions(agent_name='ailive-realtime', ...)`
- token API: `RoomAgentDispatch({ agentName: 'ailive-realtime', ... })`
LiveKit 用 `agent_name` 作 worker pool 隔離，避免跨專案 dispatch。

**血管接口**：跟既有 agent 共用 LiveKit cloud 時，**永遠**設 `agent_name`，不要靠 PROJECT_NAMESPACE prompt 防呆（那是「reject」不是「routing」）。

對應江彬 CLAUDE.md 第 15 條原文已記錄此事，今天親自踩了一次。

## 通用心法

- **無聲失敗最致命**：LiveKit explicit dispatch / Anthropic 401 / ElevenLabs ws connection closed — 三條都是「component A 看不到問題、component B 沒被通知」的典型。debug 時要主動補偵測（環境偵測 panel 5 燈就是這個目的）。
- **動手前驗證假設**：今天破紀錄 — 違反三次（Vercel key 假設有效 / LiveKit 假設 dispatch / LiveKit 假設新 project），每次都白跑一輪 deploy。
- **新 GCP project 第一次 deploy 必踩 IAM**：Cloud Build region + secretAccessor 兩條都會卡，預備好 grant 命令一氣呵成。

## 環境偵測 panel 設計（成果）

`/realtime/[characterId]` 的 5 燈 + diagnostic log + 一鍵複製，效果良好：
- 用戶不懂技術也能準確指出哪個鏈段斷
- 「複製診斷」直接 paste 給築，省掉問來問去的時間
- 江彬有 6+ 燈版本，我們精簡到 5 個夠用

**遷移建議**：未來其他 LiveKit-based 即時通話功能（不管哪個專案）第一版直接抄這個 panel 結構。
