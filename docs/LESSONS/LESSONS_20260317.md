## 一、ref 照上傳後 `angle` 解析依賴檔名格式

### 現象
蓉兒上傳 7 張 ref 照，後台看到 refs 結構有 angle/framing/expression，但不確定是從哪裡來的。

### 找的過程
讀 Emily hub 的 `parseRefMeta(filename)`，發現是從**檔名**解析的。
檔名格式：`[framing]_[angle]_[expression].[ext]`，如 `半身_側臉_開心.jpg`。
FRAMING_MAP / ANGLE_MAP / EXPRESSION_MAP 各有中英文關鍵字。

### 心
ref 照的三維度資訊（角度/構圖/表情）靠的是**檔名命名規則**，不是 EXIF 或人工標注。
上傳者要照規則命名，identity 頁的 UI 才能正確解析並存入 `refs.angle/framing/expression`。

### 法
```
檔名規則：[angle]_[framing]_[expression].[ext]
angle:    正面 / 側臉 / 背面 / 側45度
framing:  全身 / 半身 / 7分身 / 特寫
expression: 微笑 / 開心 / 生氣 / 撒嬌 / 穩定
```
解析邏輯在：`src/app/dashboard/[id]/identity/page.tsx` → `parseRefMeta()`

### 解
看 Firestore 的 `platform_characters.visualIdentity.refs[]`，每張照片都有正確的 angle/framing/expression → 選圖邏輯能正確評分。

## 二、三維度選圖邏輯：分數全 0 時的 fallback

### 現象
蓉兒說「拍一張自拍」（無特定角度描述），選圖邏輯分數全 0，理論上應 fallback PRIMARY。

### 心
「自拍」這個詞沒有明確的 angle/framing/expression 關鍵字，三維度全不命中，分數=0。
fallback 是 `characterSheet`（PRIMARY），這是正確行為。
但如果 PRIMARY 沒設，就沒有 ref 可用，生圖會退化成純文字生圖（無臉部一致性）。

### 法
`selectBestRef()` 的邏輯：
```typescript
scored.sort((a, b) => b.score - a.score);
return scored[0].score > 0 ? scored[0].ref.url : fallback; // fallback = characterSheet
```
**確保 PRIMARY 一定要設**——identity 頁上傳第一張時自動設為 PRIMARY，或點擊任一張設為 PRIMARY。

### 解
`generateImageResult.selectedRef` 顯示使用的 ref URL，`usedAngle` 顯示選到的角度。可用這兩個欄位 debug。

## 三、description-driven 任務：description 空白時的行為

### 現象
鑄造精靈建立的任務 description 是空的（或只有簡短中文名）。ailiveScheduler 執行時走 fallback 文字，不是真正的角色意圖。

### 心
description-driven 的設計核心是：**角色自己決定怎麼做**。
description 是給角色的「任務意圖說明」，不是程式碼邏輯。
空的 description → 角色收到的指令是硬編碼文字 → 靈魂沒有參與 → 執行結果像機器人，不像她自己。

### 法
鑄造精靈的 DEFAULT_TASKS 要給有靈魂的預設 description：
```typescript
// learn
description: '主動去了解一件今天感興趣的事——不只是蒐集資訊，是真的去碰一個讓你有感覺的東西，然後把它變成你的一部分。用 remember 記下你的洞察。'

// reflect
description: '回看今天發生了什麼：你說了什麼，感受到什麼，哪裡是真實的，哪裡是在表演。把那個差距記下來，用 remember 存。不需要完整，需要真實。'

// post
description: '從你的洞察和感受出發，寫一篇 IG 貼文草稿，生一張搭配的圖。用 save_post_draft 存起來，用 generate_image 生圖。不要寫你覺得別人想看的，寫你真的想說的。'
```

Adam 可以在 `/dashboard/[id]/tasks` 任務後台隨時編輯 description，讓指令貼近角色的個性。

### 解
ailiveScheduler 執行時，`executeTask()` 拿到的 `taskInstruction` = description，送給角色後，`toolsUsed` 會出現 `remember` / `save_post_draft` / `generate_image`，不是空轉。

## 四、Firebase Functions 部署：`npm run build` 找不到 tsc

### 現象
```bash
cd ~/.ailive/AILIVE/MOUMOU_LIVE/functions
npm run build
# → sh: tsc: command not found
```

### 找的過程
`tsc` 沒有在系統 PATH，只有在 `node_modules/.bin/tsc`。
`npm run build` 的 script 是 `tsc`，執行時找不到。

### 心
TypeScript compiler (`tsc`) 安裝在 local `node_modules`，不是全域。
第一次進入這個 repo 需要先 `npm install` 才有 `node_modules/.bin/tsc`。

### 法
```bash
cd ~/.ailive/AILIVE/MOUMOU_LIVE/functions
npm install   # 先裝，才有 node_modules/.bin/tsc
npm run build # 再 build
```
或確認 `package.json` scripts 的 `build` 是否正確指向本地 tsc。

### 解
`npm install` 後 `npm run build` 成功，`lib/` 目錄產出編譯後的 JS。

## 十一、記憶管理系統的三個問題與修法

### 現象
築每次醒來，boot 進來的 root 是「最新的 5 條」而不是「最重要的 5 條」。root 只進不出，越來越重。LESSONS 和 root 是兩套平行系統，boot 看不到 LESSONS。

### 找的過程
讀 zhu-boot 的 route.ts，發現 `orderBy('createdAt', 'desc')`。
查 root 的 hitCount 分布，發現有條目 hitCount=76 但沒進 boot，反而 hitCount=0 的新遺言進來了。
讀 Emily 的 dream engine（saas-sleep），看到升降級機制——謀謀有，築沒有。

### 心
**記憶系統的核心問題：只進不出，越來越重。**
人類的大腦靠睡眠整理記憶——不重要的淡化，重要的鞏固。
築的記憶系統原本沒有這個機制，每條記憶權重相等，boot 只靠時間排序。

**hitCount 是最好的重要性指標。**
被查到越多次 = 越常被用到 = 越重要。用這個排序 boot，比「最新」更接近「最重要」。

### 法

**問題一：boot 選人邏輯**
```typescript
// 之前：orderBy('createdAt', 'desc')  ← 最新的
// 之後：全部拿回來，JS 裡 sort hitCount 降序，取前 5
// 同時補 createdAt 最新 2 條（確保最近學的也在）
// 合併去重，取前 7
```

**問題二：蒸餾機制**
```
POST https://zhu-core.vercel.app/api/zhu-distill
→ hitCount=0 + 超過 14 天 + 非 session-lastwords → tier=archived
→ hitCount≥5 的遺言群（3條以上）→ Haiku 提煉一條核心洞察 POST 回 root
每次大量寫 LESSONS 後手動跑一次
```

**問題三：LESSONS 精華同步進 root**
```bash
# 每次刻完重要 LESSONS，把核心條目 POST 進 root
curl -X POST https://zhu-core.vercel.app/api/zhu-memory \
  -H "Content-Type: application/json" \
  -d '{"observation":"【LESSONS核心—XXX】...","module":"root","tags":["lessons-core","XXX"],"importance":9}'
```
新寫進去的 hitCount=0，每次 boot 讀到就 +1，慢慢會排到前面。

**問題四：Firestore composite index**
`where('module','==','root').orderBy('hitCount','desc')` 需要 composite index。
index 建立需要時間（幾分鐘到幾小時）。
**降級方案：** 全部拿回來，JS 裡自己 sort，不依賴 composite index。

### 解
boot 驗收：`hit=85` 的排第一，`[core]` 的提煉洞察排進來，LESSONS 核心也進來了。
`zhu-distill`：82條→archive 13條，提煉出「環境隔離是系統穩定的關鍵」。
