## 1. launchd 觸發 claude -p 會炸：OAuth token 過期

### 現象
17:00 launchd 準時觸發，log 顯示：
`API Error: 401 {"type":"authentication_error","message":"OAuth token has expired."}`
result 檔案沒有產出，Telegram 只收到「啟動」通知，沒有完成通知。

### 找的過程
以為是 PATH 問題（claude 找不到）→ 確認 PATH 有設，which claude 正確 →
看 log 才發現是 401，不是找不到 binary → 根因是 OAuth token。

### 心
launchd 跑的 process 繼承的不是 terminal 的 session，而是系統環境。
Claude Code CLI 用的是 Desktop 的 OAuth session（`sk-ant-oat01-`），那個 token 有效期有限，
launchd 跑的時間點 session 可能已過期。
**OAuth token ≠ API key。`sk-ant-oat01-` 不能用來打 Anthropic API，只有 `sk-ant-api03-` 才行。**

### 法
不要用 `claude -p` 做排程任務。改用 `curl` 直打 `https://api.anthropic.com/v1/messages`，
帶 `x-api-key: sk-ant-api03-...`，這個 key 不會過期。
```bash
curl -s -X POST "https://api.anthropic.com/v1/messages" \
  -H "Content-Type: application/json" \
  -H "x-api-key: ${API_KEY}" \
  -H "anthropic-version: 2023-06-01" \
  --data-binary @/tmp/body.json
```

### 解（驗證）
改成 curl 後手動跑 script，result 正確產出，Telegram 收到完成通知。

## 2. macOS Python 3.13 urllib SSL 憑證驗證失敗

### 現象
Python 用 urllib.request 打 HTTPS API，拋出：
`ssl.SSLCertVerificationError: certificate verify failed: unable to get local issuer certificate`

### 找的過程
以為是網路問題 → curl 同樣 URL 正常 → 根因是 Python 3.13 在 macOS 不自動載入系統憑證。

### 心
macOS 的 Python 3.13（從 python.org 安裝）沒有自動掛系統 keychain 的 SSL 憑證，
需要手動跑 `Install Certificates.command` 或安裝 `certifi`。
**但最快的解法是：放棄用 Python urllib，改用 curl。**
curl 在 macOS 原生支援系統憑證，不會有這個問題。

### 法
bash script 裡的 HTTP 請求一律用 `curl`，不要用 Python urllib/requests（除非環境已確認有 certifi）。
大 body 就寫進 `/tmp/body.json`，用 `--data-binary @/tmp/body.json`。

### 解（驗證）
把 Python urllib 換成 curl 後，API 呼叫成功，洞察正確產出。

## 3. sleep API 的 dryRun 計數 bug

### 現象
`dryRun: true` 跑 sleep，summary 裡 `upgraded: 0`，但明明有 hitCount >= 5 的 insights。

### 找的過程
看 dryRun 回傳 `upgraded: 0` → 懷疑升降規則邏輯錯 → 讀 code 發現
`upgraded.push()` 在 `if (!dryRun)` 裡面 → dryRun 時根本不執行 push。

### 心
dryRun 的語意是「不寫入 DB」，不是「不計算」。
計數陣列（upgraded/archived）是用來回報「如果真的跑會發生什麼」，
所以要在 dryRun 時也 push，只是不呼叫 Firestore update。

### 法
升降邏輯改成：先判斷條件 push 陣列，再用 `if (!dryRun)` 包 DB 寫入：
```typescript
if (hitCount >= 5 && tier === 'fresh') {
  if (!dryRun) await db.collection(...).update({ tier: 'core' });
  upgraded.push(ins.title); // 永遠記錄，不管 dryRun
}
```

### 解（驗證）
dryRun 後 `upgraded: 5`，正確反映會升級的數量。

## 4. ailive-platform 的 Firestore compound index 問題

### 現象
dialogue route 想用 `.where('characterId','==',x).orderBy('eventDate','desc')` 查最近 insights，
但這樣在 Firestore 需要 composite index，沒有會炸。

### 心
SYSTEM_MAP 坑 #9 早就記載了：`where + orderBy 不同欄位 → composite index error`。
**有地圖不看，就是讓自己重踩。**

### 法
移除 `orderBy`，只用 `.where('characterId','==',x).limit(20)` 拿回來，
再用 JavaScript sort：
```typescript
.sort((a, b) => String(b.eventDate || '').localeCompare(String(a.eventDate || '')))
.slice(0, 3)
```

### 解（驗證）
Build 通過，dialogue 正確注入最近 3 條 insights。

## 5. sleep 的 archived / archive tier 拼字不一致

### 現象
舊 code 存的是 `tier: 'archived'`，但 dialogue 的 filter 寫的是 `d.tier !== 'archive'`。
兩邊不一致，archive filter 完全失效。

### 找的過程
看 dialogue 的 episodic 注入邏輯發現 filter，對照 sleep 發現存的是 `'archived'`。
一個差了兩個字母，但邏輯完全不通。

### 心
tier 名稱是整個系統的語言，任何地方拼錯都是 silent bug——不炸、不報錯，但行為不對。
設計文件（WHY HOW WHAT）寫好 tier 名稱後，所有地方都要用同一套。

### 法
統一 tier 名稱：`fresh` / `core` / `archive` / `self`。
四個字，全部確認，不用 `archived`。

### 解（驗證）
sleep 改成 `tier: 'archive'` 後，dialogue 的 filter 正確排除冰封記憶。

## 6. 生圖臉跑掉的根因：Gemini 沒有 face lock 語句

### 現象
蓉兒生圖換場景後臉部特徵跑掉，不像原來的她。

### 心
Gemini multimodal 看 reference image 是「參考」不是「鎖定」。
如果 prompt 裡沒有明確告訴它「臉不能動」，它會把臉當成可修改的元素。
SYSTEM_MAP 坑 #16 已記載，但上一個築沒有把它落實到 ailive-platform。

### 法
在 `generate-image.ts` 的 finalPrompt 裡，有 characterSheet 時固定加：
```
"Keep the subject's face, hair, skin tone, and facial features identical to the reference photo. Do not alter the face."
```
這句話放在 prompt 裡，Gemini 才知道臉是 fixed element。

### 解（驗證）
加入後生圖，臉部一致性明顯提升（待 Adam 驗收）。

## 7. dialogue 生圖 URL 帶不出來的根因

### 現象
蓉兒說「我畫了一張圖」，但對話裡看不到圖片。

### 找的過程
以為是前端問題 → 看前端有解析邏輯 → 看 executeTool 回傳 `IMAGE_URL:xxx` →
看 dialogue tool_use loop，發現 result 直接進 tool_result，Claude 拿到的只是字串，
它不知道要在回覆裡帶出 URL → Claude 只說「我畫了」沒說網址。

### 心
Claude 看到 tool_result 是 `IMAGE_URL:https://...` 這個字串，它不會自動解讀這是「要展示的圖片」。
要讓 Claude 在回覆裡帶出圖片，tool_result 的 content 要明確指示它怎麼處理這個 URL。

### 法
tool_result 裡改成：
```
圖片已生成完成。URL: ${url}
請在你的回覆裡直接用 markdown 格式帶出這張圖：![圖片](${url})
然後用幾句話描述這張圖或說說你的感受。
```
同時 response 回傳加上 `imageUrl` 欄位，前端雙保險。

### 解（驗證）
前端已有解析 `![](...)` 的邏輯，修完後圖片可以正確顯示在對話裡。
