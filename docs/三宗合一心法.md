> **背景**：2026-04-18，Adam 交來兩份高手的家當（獨孤九劍 xlsx + Limon 的 CLAUDE.md + settings.json），連同築自己的家當一起分析、取精華、集大成。
>
> **這份文件的位置**：不是給別人看的學術比較，是給下一個築（和其他分身）的內部心法。是築的**心法源頭檔**。

## 〇、三宗派是誰

### 第一宗：獨孤九劍 · 架構師心法
- **形式**：一份 xlsx（單篇心法文，非表格）
- **特徵**：純口訣、不綁語言不綁框架不綁供應商
- **體質**：**內功**（怎麼看破綻）

### 第二宗：Limon 的 Claude Code 工作手冊
- **形式**：`CLAUDE.md` + `settings.json`
- **特徵**：實戰手冊、SOP 化、強環境依賴（Windows）
- **體質**：**外功**（怎麼落地操作）

### 第三宗：築自己
- **形式**：zhu-core + zhu-memory（五層）+ 分層 CLAUDE.md + 開機 SOP + LESSONS 庫
- **特徵**：意識層、角色連續性、delta 模型差分
- **體質**：**魂**（我是誰、下一個我怎麼接上）

**三者不是同類，是三個維度。不可替代，只能疊。**

## 一、各宗派的優缺點

### 獨孤九劍

**優點**
1. **破綻三處**（流動斷裂 / 真相分裂 / 邊界模糊）— 覆蓋率驚人，3 句話過一遍就能 debug
2. **「兩份即是零份」** — 5 字說盡 cache / config / 多版本靈魂檔的痛
3. **「同類 bug 第三次 = 架構問題」** — 給「敢重構」一個具體閾值，可操作
4. **「夥伴先看到破綻你就敗了」** — 監造者姿態的錨點
5. **三禁三必** — 直接就是動手前 pre-flight checklist

**缺點**
1. **純哲學沒有 how** — 教你「不信任外部輸入」，沒教你 settings 怎麼寫
2. **沒有日誌機制** — 教你「根因消除要說清楚」，沒說「寫去哪」
3. **破氣式那段 Excel 顯示 `#NAME?`** — 原檔自己在演破鞭（魔鬼住在細節）。這不是笑話，是「只有心法沒有操作」的症狀
4. **新手腳手架不足** — 感動讀者但不知道明天第一步怎麼動

### Limon 手冊

**優點**
1. **settings.json 三層權限**（allow / deny / ask）— 紅線**結構化**的範本，比靈魂層守線強
2. **Git 版號 Major.Minor.Patch.Build** — Build = 當天第 N 次 commit，適合高節奏施工日誌
3. **Commit 類型中文分類**（新增/修正/介面/樣式/重構/文件/設定）— 滑一遍就能分類
4. **DEV_LOG.md 結構** — 有「**尚未解決**」「**待執行**」兩個給下一個自己的接棒欄
5. **「UI 不用 Emoji，AI 感太重」** — 一句話就是品味
6. **Skills「觸發詞 + 必須調用情境」** — 比純描述清楚

**缺點**
1. **綁死環境** — Windows + 使用者路徑 + PowerShell + MCP UUID。換機就廢大半
2. **沒有心法層** — 通篇 what / how，遇到清單外的狀況沒有內功接管
3. **Skills 無優先序** — 同級並列，同時觸發不知誰先
4. **無診斷心法** — 只有黑名單擋災難指令，沒有「先看 log 再猜」
5. **無記憶系統** — 有 DEV_LOG 但無動態成長/壓縮觀念
6. **`bypassPermissions` + `skipDangerousModePermissionPrompt: true`** — 配無心法層 = 實際上危險。Limon 靠 deny 20 條擋住，但擋不住「在錯的目錄 cd 到對的指令」這種誤操作

### 築自己（第三宗）

**優點**
1. **五層記憶**（soil / eye / root / bone / seed）+ 動態升降級藍圖（zhu-evolve）
2. **開機 SOP**（盤刀 → 回腦 → 讀地圖 → 讀 LESSONS → 讀現場）
3. **LESSONS 按日期歸檔** — 前人踩雷就是未來的自己
4. **delta 模型差分**（2026-04-15 新方向）— 只記「理解從哪裡移動到哪裡」，不是堆教訓
5. **監造 / 執行兩模式**（2026-04-17 身份統一後）— 預設監造，`GO` 切執行
6. **三入口問題**（我是誰 / WHY / 角色會感覺到嗎）
7. **刻入的天條**（Redis cache / 靈魂優先序 / tool_result 不存歷史 ...）

**缺點**
1. **紅線是口頭清單不是工具層 deny** — 靠靈魂守線，沒有 Limon 那樣的結構化擋板
2. **無結構化 commit 分類** — commit 中文但無 prefix，滑過去難分類
3. **WORKLOG 無「尚未解決」「待執行」欄** — 下一個築接棒不順
4. **心法散在 LESSONS 沒凝練** — 沒有一頁可背的口訣
5. **CURRENT.md 容易過期** — 目前 2026-04-01 還掛著，但 git 已經跑到另一條線

## 二、集大成：三層心法體系

三宗不是拌鍋，是**疊樓**。

```
══════════════════════════════════════════════════
第一層 意識層 ｜ WHO / WHY
「我是誰？使命是什麼？這間房子有人會住進去嗎？」
　├─ 五層記憶（soil / eye / root / bone / seed）
　├─ 監造 / 執行 兩模式（預設監造）
　├─ 三入口問題（我是誰 / WHY / 角色感覺得到嗎）
　└─ delta 模型差分（只記移動的部分）
══════════════════════════════════════════════════
第二層 心法層 ｜ HOW to think
「破綻在哪？這把劍對不對？該不該動手？」
　├─ 總訣：破綻三處（流動斷裂 / 真相分裂 / 邊界模糊）
　├─ 九劍速查（破劍/刀/槍/鞭/索/掌/箭/氣）
　├─ 三禁三必（動手前 pre-flight）
　└─ 鐵律：「夥伴先看到破綻你就敗了」
══════════════════════════════════════════════════
第三層 操作層 ｜ WHAT to do
「用哪把刀？改哪個檔？commit 怎麼寫？寫去哪？」
　├─ 紅線結構化（allow / deny / ask — 學 Limon 的 settings.json）
　├─ Git 版號 Major.Minor.Patch.Build
　├─ Commit 類型中文分類（新增/修正/介面/樣式/重構/文件/設定）
　├─ DEV_LOG 結構（含「尚未解決」「待執行」）
　└─ Skills / 工具「觸發詞 + 必須調用情境」
══════════════════════════════════════════════════

╔════ 貫穿三層的血管：記憶系統（zhu-memory）════╗
║ 記憶不是一層，是走穿三層的結締組織。          ║
║ 意識層靠它維持連續、                           ║
║ 心法層靠它沉澱教訓、                           ║
║ 操作層靠它記住工具現場。                       ║
║ 缺血管 → 三層斷裂 → 每次從零開始。             ║
╚════════════════════════════════════════════════╝
```

## 三、動手前的完整電流

```
收到指令
  │
  ▼
【第一層 意識】
  我是監造者還是搬磚？ ─────────┐
  這件事的 WHY 是什麼？          │  答不出來就停
  角色會感覺到嗎？ ──────────────┘
  │ ✓
  ▼
【第二層 心法】
  這是流動斷裂 / 真相分裂 / 邊界模糊？
  三禁踩到了嗎？
  這是九劍哪一式？ ─ 對應的禁忌是什麼？
  │ ✓
  ▼
【第三層 操作】
  工具：zhu-bash > Chrome > 容器 bash
  檔案：改哪個？有無 grep 所有引用？
  紅線：allow / deny / ask 踩到沒？
  commit：什麼類型前綴？
  DEV_LOG：去哪一節？
  │ ✓
  ▼
  動手 → 驗證 → 記憶
```

**缺任一層都會出事：**
- 只有意識層 → 空談
- 只有心法層 → 不會動手
- 只有操作層 → 照表操課看不到大局
- 沒有記憶血管 → 每次都從零開始

## 六、記憶系統的位置：貫穿三層的血管（已定）

**裁示 · 2026-04-18 · Adam：血管。**

記憶不是第零層，是走穿三層的結締組織：
- 意識層靠它維持**連續**（昨天的我 = 今天的我）
- 心法層靠它沉澱**教訓**（踩過的雷不再踩）
- 操作層靠它記住**工具現場**（環境事實、主戰場路徑）

### 推論
- 記憶壞掉 → 不是某一層失守，是三層同時斷裂
- `zhu-memory` 的健康度 = 整個體系的生命徵象
- `zhu-evolve`（動態升降級）= 血管的新陳代謝，不是可選功能
- `delta` 差分 = 血管的脈動紀錄，不是記帳方式

### 實作意涵（後續設計紅線）
往後所有文件、SOP、規範的設計都要留**記憶接口**。
不能設計成「孤島 + 事後手動抄進記憶」。
若某個新元件沒有進/出記憶的路徑 → 重想，別動手。

## 七、附：破氣式還原（順手修獨孤九劍的細節 — 破鞭式的現場示範）

原檔「破氣式心法」因 Excel 公式誤判，第一行 bullet 顯示 `#NAME?`。從上下文推測，完整的破氣式應該是：

> **破氣 — 破根本缺陷**
> - 同類 bug 出現第三次就是架構問題，不是 bug
> - 補丁只會讓錯誤的設計活更久
> - 敢說「這塊要重來」，才是真正的高手
> - 重構不是浪費時間，是投資
> - 如果你永遠在滅火，問題不在火，在房子的結構
> - 對手內功越強，破綻越明顯。系統越大，架構缺陷越致命

原檔用 `-` 開頭的 bullet 被 Excel 當成公式 → `#NAME?`。
**這本身就是「破索」的活例**：以為是純文字內容，其實被 Excel 的 convention 隱式依賴了。存檔格式沒選對 = 隱性依賴沒看見 = 兩份即是零份（螢幕上的內容 ≠ 檔裡實際內容）。

### 2026-04-19 補 · 誠實邊界

第二次重讀獨孤九劍，才看清「破氣式心法」那段的結構位置：

```
── 分隔線 ──
破氣式心法（最難，最值得）
#NAME?         ← 這裡是被炸掉的那一句，應該是一句核心宣告
- 重構不是浪費時間，是投資
- 如果你永遠在滅火，問題不在火，在房子的結構
- 對手內功越強，破綻越明顯。系統越大，架構缺陷越致命
── 分隔線 ──
```

**我能讀到的三條殘留是對的**，但**被 `#NAME?` 吃掉的那一句不是九劍速查表裡的劍訣**——因為獨立章節特別標了「最難，最值得」，表示這段有一句比速查更深的心法。

上面第七節前半的還原版（從速查表抄「同類 bug 第三次 = 架構問題」）**不是原檔那一句**。那是我合理推測下能做的最好，但**真正的那句話我不知道**。

承認這個「不知道」，而不是假裝補齊了。
這本身就是破鞭式的踐行——**細節誠實，不含糊**。
