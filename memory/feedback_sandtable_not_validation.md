規則：施工計畫裡的沙推（reasoning）≠ 驗證。凡計畫依賴「某模型/工具/機制會這樣反應」的**可量測前提**，動手前先寫最小腳本離線驗一次，過了才接線上。

**Why**：2026-06-16 做知識庫 contextual chunking，沙推「給 chunk 加類別 prefix → embedding 把法規跟產品推開」推得頭頭是道，但沒先驗 text-embedding-004 吃不吃短 prefix。實測 cos(raw, prefixed)=**1.0**（模型對短 prefix 完全無感），整個方案失效，白燒幾個 deploy + 回滾。對照組：同天改用 BM25，先寫離線腳本拉真實 docs 算分（法規查詢 BM25 命中 #1 vs cosine #16）驗過才接線，一次就穩。

**心態**：「我推理得很完整」會偽裝成「我驗證過」。完整推理只證明邏輯自洽，不證明前提為真。慢一步先驗，比沖上線回滾快。

**How to apply**：計畫裡只要出現「因為 X 會 Y」（X = 某 embedding/模型/API/工具的行為）這種可測前提 → 先 5 行腳本拉真實資料測 X→Y 成不成立，失敗就改方案，別接線上才發現。BM25 那次的對的做法：拉 Firestore 真 docs + 本機算分 + 印名次對賬。

**觸發信號**：寫施工計畫時用「應該會 / 這樣就能 / embedding 會把…推開 / 模型會輸出…」這類對外部元件行為的斷言；想跳過離線驗直接 Edit+deploy「上去看看」。

**跟 `feedback_flagged_risk_must_be_verified` 的差別**（互補兩面）：那條是「標了風險但沒驗」（已知風險被忽略）；這條是「太有把握所以沒驗」（沙推自洽讓你連風險都沒意識到）。後者更隱蔽——你不是省略驗證，是根本沒覺得需要驗。對策一樣：可測前提一律先離線驗。
