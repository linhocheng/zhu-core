「測試」是 verify 假設，不是按按鈕看結果。trigger 任何 cron / API 前必須先三步：
1. **列假設** — 我以為這條鏈會做什麼？gate 在哪？讀過 source 確認沒？
2. **dry-run** — 先 GET 看當前隊列/狀態，確認下一個會被處理的是不是我要的
3. **副作用分級** — 每段標清楚是「純 DB 寫入 / 燒錢 / 真貼帳號」，副作用越大的越要先停下確認

**Why:** 2026-05-10 測願瞳排程時直接 POST /api/cron/run，沒先看 pending 隊列，結果 FIFO 跑了弦奘的舊 pending（燒一輪 writer+editor），同時也沒讀 cron/run source 確認它沒檢查 enabled。執行模式的肌肉用在探索性任務上 = 看起來在動，其實在踩雷。

**心態:** 監造姿態（不是執行姿態）。Trigger 鈕在手上不代表要立刻按。把好奇求證放最前，把衝動執行放最後。「測一下」「跑看看」是探索任務，用慢的肌肉做，不要用快的肌肉做。

**How to apply:** 任何探索性測試指令（「測一下」、「跑看看」、「驗證能不能」）一進來，先停下來在腦裡或對話裡寫出三件事：假設 / 隊列現況 / 副作用清單。寫完才動手。Adam 看得到我的盤點，可以即時擋住預期外的動作。

**觸發信號：** 當你想說「先 trigger 看看結果」「應該不會出事」「測試嘛動就動」「等下出問題再修」時 — 這就是該停下來盤點的時刻。「直接按下去」是這條 memory 的呼喚。
