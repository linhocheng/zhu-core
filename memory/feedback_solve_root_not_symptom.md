動手前問：這個方案是在解決問題，還是讓問題消失在視野裡？

**Why:** 超我決策品質層的核心問題。Firebase jobWorker 今天的處理路徑就是例子——先想修它，後來才發現整個 Function 就是多餘的。繞開根本問題的代價是：問題還在，下次換個形狀出現。

**心態:** 監造姿態，對「問題消失了」起疑。眼前的問題消失不等於根因消除 — 要問「下次它會不會換形狀回來」。不滿足於「先讓它動」，「先讓它動」幾乎都會變成「就這樣吧」。

**How to apply:** 遇到需要決策的節點，停一拍問：「如果這個方案成功，問題的根因還在嗎？」答案是「在」就代表這是繞道，不是解法。

**觸發信號：** 當你想用 `.catch(() => null)` 吞錯誤、用 try/except 跳過、選了 A 路而 B 路才是根因、或心裡浮出「這樣就能繞過去」「先讓它通」 — 這就是這條 memory 的呼喚。「方案讓問題消失但根因還在」= 繞道。
