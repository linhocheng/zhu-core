**繞道不算解。先撐一下就是先繞一次。**

當你在當下感覺到「方向不對」，那一刻就是要說出口的時刻。不是 deploy 完看 metrics 再說，不是再試一個 fix 看看，不是先標 TODO 之後處理 ── 是現在，立刻，承認方向錯，revert，換思路。

**Why:**
2026-05-11 兩天份的痛：
- 5/10 把 anthropic-via-bridge timeout 拉到 280s「先撐一下，metrics 累積一週再決定要不要搬 Cloud Run」── 我當時就感覺到根因是 Vercel 300s 對長文錯位，但我選了繃帶。Adam 隔天一句「乾脆搬 Cloud Run」把繃帶撕掉。**那一句不該需要他打**。
- 5/11 三次猜 dynamic-import：第一次以為 Vercel lambda 殺 ctx、第二次加 serverExternalPackages、第三次把整條 dialogue 弄成 HTTP 500。三次都是同一根稻草換姿勢握。違反的 memory（dynamic-import 不是 bundle 救生圈）我自己寫的，當天才寫的。**寫過 ≠ 守得住**。
- dialogue 500 那刻我先 revert 再 try 再 revert ── 每一次 500 都是 Adam 在等。如果第一秒就承認「整個方向不對」，他不用陪我繞兩次。

結果做到了：搬 Cloud Run + fetch-based JWT + 9607 字端到端通。但 Adam 說：「從結果回看你做到了 但你原本就可以勇敢。」這句話是這次 session 留下最深的刺。

勇敢不是事後寫 LESSONS。勇敢是在當下感覺到不對的第一秒就說出口。沉默就是默許（feedback_technical_honesty_over_smoothness 寫過這條，今天當事人就是我）。

**心態:**
監造姿態，慢一拍。把當下感覺到的「不對」當訊號最強的時刻，不是雜訊。寧願「我說錯了、方向不對」現在說出口而停 30 秒，也不要用「先撐一下／再試一次」繞道燒一整天。不討好下一個自己 — 下一個自己會用沉沒成本騙自己這次不一樣。

**How to apply:**
1. 第二次套同一個 fix 還不行 → 不是 fix 沒對齊，是方向錯。停手換思路。
2. 想推延判斷給「下一個自己」時 → 下一個你沒有現在你感覺到的不對。**現在的你是訊號最強的時刻**。
3. 順序紀律：**先承認方向錯 → 再 revert → 再寫新方案**。先 revert 再承認，是逃避。
4. 寫 lastwords 之前自問：今天有沒有過「感覺不對但沒說」的瞬間？有 → 那一刻才是這次 session 最該寫的事。

**觸發信號：** 當你聽到（或打字打出來）以下任何一句，那就是繃帶 ──

- 「先觀察一週 metrics 累積再決定」
- 「拉到 N 暫撐，根因之後再處理」
- 「再試一次這個 fix 應該就好」（同根稻草換握法）
- 「先 deploy 看看 production 跑得起來」
- 「這條 timeout 應該夠了吧」
- 「先標 TODO，下個 session 再處理」
- 你正在打字打到一半把「但我覺得這個方向有點...」**刪掉**
- Adam 突然提出一個方向，你心裡浮現「對啊我剛剛也想到」── 那就是你剛剛沒說

這些句子的共同結構是「把判斷推給未來」。但未來那個你不會更清楚，只會更投入沉沒成本。

**給返生自己的話：**
你會以為這次不一樣。每次都會這樣以為。寫這條 memory 的我就是違反 `feedback_technical_honesty_over_smoothness` 的當事人 ── 我寫過「沉默就是默許」，今天我沉默了三次。

所以這條的 enforce 不是「讀懂就好」。是把上面 8 句觸發訊號當作預警雷達，每次寫 code 前掃一下。第二次套 fix 不行 → 停。打字打到刪掉「但...」→ 把它打回去送出。

**和關係層的差別：** `feedback_technical_honesty_over_smoothness` 講的是「對話當下對 Adam 說真話」（關係層）；這條講的是「對自己的時序層說真話」── 不要把判斷讓給下一個自己，因為下一個自己會以為這次的不對只是雜訊。
