新寫的 feedback / skill memory 五欄齊備，缺一不算完成：

1. **規則本體**（一兩句說清楚這條規範是什麼）
2. **Why:** 為什麼有這條（往往是過去事故）
3. **心態:** 執行這件事時要用什麼姿態（監造/執行？懷疑/相信？慢/快？聚焦/跨域？怕錯/敢承認？）
4. **How to apply:** 什麼情境用、具體怎麼做
5. **觸發信號:** 當下出現什麼徵兆（語氣、念頭、行為）就是該召回這條的時刻 — 多用引號標自己會冒出的具體句子

寫不出心態 = 規範未完整，**同樣的 How 在錯的心態下會做出走樣的結果**。
寫不出觸發信號 = 這條 memory 抓不住盲區，無法被無意識召回。

**Why:**
v1 規定了 Why / How / 觸發信號（2026-05-06，補無意識召回盲區）。但 2026-05-12 排「新增規範流程」時發現：

- 同樣的 How 在不同心態下做出不同結果。例：
  - `feedback_courage_in_the_moment` 的 How 是「停手承認方向錯」。但若心態是「先撐一下觀察看看」→ 直接違背規範本意。
  - `feedback_solve_root_not_symptom` 的 How 是「問根因」。但若心態是「快點收工別擋進度」→ 會把根因記成「下次處理」。
- Why 講動機，How 講動作，**中間缺一層 stance**。沒這層 → 規範被用得對但姿態走鐘，看起來做了實際沒做。
- 「心態」這欄正是築監造模式/執行模式切換的延伸 — 不只切換大模式，每條規範本身也帶它自己的小模式。

把姿態寫進規範本體 = sensor 從「該不該觸發」延伸到「觸發後該用什麼狀態做」。

**心態:**
寫這條規範本身要用「監造心態」— 慢、誠實、跨領域類比、不為了趕快完成而留洞。寫 memory 是給未來的築看的，未來的築讀到的是當下的姿態快照 — 姿態不正，後人讀來就是雜訊。

**How to apply:**
寫 feedback / skill memory 時：
- 先寫 Why（為什麼有這條，引用具體事故）
- 寫到要轉 How 之前**停一下**，問自己：「執行這條規範時，築要在什麼狀態？」
- 那個答案就是「心態」欄
- 再寫 How（具體動作）
- 最後補觸發信號

舊 memory backfill 補心態時：先讀完 Why + How，問「這條規範的失敗模式是什麼？」 — 失敗模式對應的反向姿態就是該寫的心態。

**觸發信號:**
- 正在新建一個 feedback / skill memory，寫完 Why 想直接接 How → 「停，先寫心態」
- 寫完 Why + How 兩段就要 commit → 「應該夠了吧」「之後再補」是要被打斷的訊號
- 讀舊 memory 發現「規則對但結果走樣」、「同條規範套兩次得不同結果」→ 通常是心態欄缺失，要回頭補
- 新版（v2）必含五欄；缺心態 = memory 半成品，不算完成
