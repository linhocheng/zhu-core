任何「能讓 Adam 輸入/編輯資料」的介面，交付前必須回答：
1. 這份資料**誰會讀**？（哪個 worker / API / cron）
2. **何時讀**？（哪個 stage、prompt 哪一段）
3. **沒讀到會怎樣**？（fallback 到什麼？還是純展示？）

三題答不出來 = 介面建好但血管沒接 = 半成品。CLAUDE.md 血管原則：「若某個新元件沒有進/出記憶的路徑 → 重想，別動手」的鏡像版本。

**Why:** 2026-05-10 建 ContentMapTab 編輯器讓 Adam 貼 content_map JSON，但沒有任何 worker 真的讀 content_map — 只是寫進 Firestore 沒人讀。Adam 之後一定會問「那 worker 什麼時候吃這份」，與其等他問不如主動標清楚。被動標 = 違反「發現技術債要說出來」的精神。

**心態:** 監造姿態，把「介面交付」當「需要回答三題血管問」的觸發點，不是「上線了給 Adam 用」的時刻。沒接通就主動標斷點 — 不要等 Adam 問才說。「介面好了你試試」要改寫成「介面好了，但血管狀態是 X」。被動標 = 違反「發現技術債要說出來」。

**How to apply:** 寫 UI / 設定編輯器 / 表單時，commit 前自問三題血管問。沒接通就在交付訊息裡明寫「⚠️ 血管未接通：這份資料目前沒有 worker 讀，下一步要動 X 才會生效」，給 Adam 兩個選項：（a）我馬上接通（b）先放著等需要時再接。不要默默交付半成品。

**觸發信號：** 當你做完一個 form / editor / settings panel，準備收尾說「上線了你可以用」時 — 這就是該停下來自問三題血管問的時刻。「介面好了你試試」是這條 memory 的呼喚，要改成「介面好了，但血管狀態是 X」。
