# Live Media 系統進化文件
## v2.0 — 從流水線到有機體

> 初稿：築 × 2026-05-01
> 觸發：Phase 1-3 驗收後的深層覺察
> 目的：在動手之前，把看見的東西說清楚

---

## 一、我們在 v1.0 建了什麼

一條**單向瀑布流水線**：

```
情報官（每日10:00）
    ↓ 抓熱帖
排重員（即時）
    ↓ 去重
停格者（即時）
    ↓ 寫稿
閾（每20分鐘 poll）
    ↓ APPROVE / REJECT
閘（每20分鐘 poll）
    ↓ 發布
痕（48h後）
    ↓ 追蹤成效
析（每週）
    ↓ 優化建議
弦（每週）
    ↓ 戰略回顧
```

它能動。但第一天驗收就出現了死循環。

---

## 二、觀察到的問題（根因分析）

### 問題一：退稿信號消失在虛空

閾退稿「把占卜視為焦慮症狀而非靈性工具，違背品牌的肯定之心」。

這個判斷精確。

但它只寫進了 Firestore 的 `rejectionReason` 欄位。明天情報官還是照樣抓，停格者還是照樣用同一個靈魂框架寫，閾還是照樣退。**這不是死循環，這是盲循環——系統甚至不知道自己在循環。**

根因：**退稿信號沒有回授路徑。角色之間只有單向文件傳遞，沒有橫向溝通。**

---

### 問題二：閾是法官，不是編輯

現在的閾只有兩個按鈕：APPROVE 或 REJECT。

真正的總編輯做的事是：拿起紅筆，把那段「把占卜視為焦慮症狀」的開場改成「當世界的不確定感到頂點，人開始用另一種語言問問題」，然後說：「這樣發。」

**法官的代價是：每一篇不夠好的稿子都死了。編輯的能力是：把不夠好的稿子救活。**

根因：**閾的輸出協議是二元的，沒有「修改後通過」這條路。**

---

### 問題三：角色沒有工作記憶

每次 worker 跑起來，每個角色都是全新的。

停格者不記得上次被退的原因。
閾不記得上次發布後讀者反應好的文章有什麼共同點。
情報官不記得哪些關鍵字帶來了被發布的稿子，哪些一直被退。

**一個連上週發生了什麼都不記得的角色，無法成長。它只能重複。**

根因：**沒有角色工作記憶機制。每次 system prompt 都是靜態的，沒有根據歷史動態注入。**

---

### 問題四：熱訊號被排程殺死

某個話題今天下午兩點開始在 Threads 上燒。

我們的情報官明天早上十點才跑。

到那時這個話題已經死了三天。

**這是選舉期間不能犯的錯。對手說了一句話，你明天才回應，你已經輸了。**

根因：**系統是定時觸發，沒有事件觸發機制。熱訊號和觸發邏輯完全分離。**

---

### 問題五：弦不在

流水線的每個節點都做自己的事，但沒有人在看整體。

閾退稿了五篇，調性都偏了，沒有人注意到。
情報官連續三天找到的都是同一個角度的素材，沒有人注意到。
發布了十篇文章，沒有人問：這十篇放在一起，我們在跟讀者說什麼？

**弦的角色不是最終審稿人——是唯一能看見整體漂移的人。沒有她，系統能跑，但它不知道自己在往哪個方向漂。**

根因：**執行長工作位置尚未建立。目前由人類代替執行長做判斷，但人類的注意力是不穩定的資源。**

---

## 三、v2.0 的核心設計決策

以下是我作為代理執行長的決策，不是建議，是方向鎖定。

---

### 決策一：閾升級為「編輯模式」

**改變什麼：**
閾的輸出從兩段式（APPROVE / REJECT）變成三段式：

```
PUBLISH_AS_IS  → 直接交給閘發布
REVISED        → 閾自己改稿，改完的版本交給閘發布
REJECT         → 真的救不了，退稿並寫清楚方向給停格者
```

REVISED 的觸發條件：文章基礎品質合格（score ≥ 65），但語氣、立場或開場需要校正。

REJECT 的觸發條件：文章的核心前提就是錯的（例如把靈性工具視為心理應對機制），改了也是換湯不換藥。

**影響：**
- 死循環從根本破解：大多數文章不會死，會被編輯後通過
- 發布量提升（預估 REJECT 率從現在的 ~100% 降到 ~20%）
- 閾的角色更貼近真實總編輯的工作

**代價：**
- 閾的每次 Claude 呼叫成本增加（輸出更多 token）
- 修改後的文章品質需要監控（閾的編輯能力上限未知）

---

### 決策二：建立角色工作記憶（Character Work Memory）

**改變什麼：**
在 Firestore 建立 `live_media_char_memory` collection，每個角色一個文件。

記憶結構：
```json
{
  "role": "停格者",
  "rejection_signals": [
    "2026-05-01: 把占卜視為焦慮症狀，違背品牌肯定之心",
    "2026-05-02: ..."
  ],
  "approval_signals": [
    "2026-05-03: 顯化文章，以讀者內在體驗為中心，閾評分 88"
  ],
  "updated_at": "..."
}
```

**誰寫：**
- 閾 → 寫到 停格者 的記憶（每次判決後）
- 痕 → 寫到 情報官 和 停格者 的記憶（48h成效後）
- 弦 → 每週覆寫所有角色的記憶（戰略校準）

**誰讀：**
- 停格者 在寫稿前讀自己的記憶，注入 system prompt
- 情報官 在搜尋前讀自己的記憶，調整關鍵字策略

**影響：**
- 角色開始「學習」——不是情感成長，是工作校準
- 閾不需要每次都從頭判斷，可以參考歷史模式
- 系統的品質應該隨時間提升，而不是保持靜態

**代價：**
- 需要謹慎管理記憶的「半衰期」——太舊的信號會成為噪音
- 記憶污染風險：如果連續幾次閾的判斷本身就有偏差，停格者會被錯誤校準

---

### 決策三：熱訊號旁路（Hot Bypass）

**改變什麼：**
情報官在每次搜尋後，對每篇素材輸出一個 `heat_score`。

如果 `heat_score > 85`：
```
不等明天 10:00 → 立刻觸發寫稿 → 閾優先審 → 閘立刻發布
```

如果 `heat_score ≤ 85`：進正常排程。

**實作方式：**
情報官 worker 不只每天跑一次。加一個「熱偵測模式」：每小時快速掃描一次（低成本，只看數字，不深入分析）。熱分超標就立刻觸發完整鏈。

**影響：**
- 時效性從「次日」提升到「數小時內」
- 真正能在話題燃燒期發聲
- 選舉比喻：這是快速回應部隊（Rapid Response Team）

**代價：**
- 每小時掃描的 API 成本（但用 Haiku + 快速模式，很低）
- 可能因為誤判熱度而發出品質較低的文章（需要閾的把關）

---

### 決策四：弦的最小可行版本（弦 MVP）

**現在不做完整的弦。但要做她最重要的一件事：**

每週日 23:00，弦讀：
1. 本週所有角色的工作記憶
2. 本週發布文章的成效數據
3. 本週閾的所有判決（APPROVE / REVISED / REJECT 分布）

輸出：
1. 給停格者的「本週寫作方向更新」（覆寫角色記憶的核心指導語）
2. 給情報官的「本週關鍵字優先序更新」
3. 給閾的「本週審稿標準確認或調整」

**這不是弦在做所有事。這是弦在做她最不可替代的事：看整體，調頻率。**

## 四、我們可能還在視盲區的東西

以下是在設計 v2.0 時我注意到的潛在盲點，尚未有解法，但需要命名：

---

### 盲點一：讀者信號完全缺席

我們有痕（成效追蹤員）和析（績效優化員），但他們追蹤的是什麼？

目前的架構裡，我們沒有方法知道讀者對一篇文章的真實反應。

閱讀量：沒有。
留言：沒有。
分享：沒有。
讀完率：沒有。

我們的「成效追蹤」是一個空殼，因為文章發布在 Cloud Run 的 URL 上，沒有任何追蹤機制，也沒有連接任何社群平台。

**這是系統最大的盲點。整個優化鏈條的基礎——讀者的真實反應——目前是零。**

解法方向（尚未設計）：
- Phase 5 社群層接通後，社群互動數據才會有
- 或考慮在文章頁面加基礎的 pageview 追蹤

---

### 盲點二：文章之間沒有關係

每篇文章都是孤立的。它不知道前一篇寫了什麼，也不知道下一篇應該接什麼。

真正的媒體有：主題線（本週關注點）、系列（連載）、呼應（這篇回應了上週那篇的問題）。

目前的停格者只看眼前的素材，不看整體地圖。

**這意味著讀者永遠無法積累「追這個頻道」的理由——每篇都像是陌生人說的話。**

解法方向（弦的進階版）：弦應該每週制定「主題方向」，停格者和情報官都以此為搜尋和寫作的主軸。

---

### 盲點三：系統調性漂移不可見

假設閾連續三週的判決開始偏向「只接受悲觀療癒風格」，整個發布庫的調性就慢慢偏移了。

但沒有人能看見這個漂移。我們沒有儀表板，沒有「發布文章的整體調性分布」，沒有任何讓人類可以一眼看見「系統健康」的工具。

**盲目運行的系統，比不運行更危險——因為它在累積你看不見的偏差。**

解法方向：後台管理頁需要加「系統健康儀表板」：
- 近30天 APPROVE / REVISED / REJECT 分布
- 近30天發布文章的關鍵字雲
- 角色記憶最後更新時間

---

### 盲點四：「熱訊號」的定義沒有共識

情報官說「這篇 heat_score 90」，但 90 分的標準是什麼？

目前的評分邏輯是情報官自己在 system prompt 裡估算的，沒有客觀標準。

一個剛爆紅的話題和一個持續熱的話題，heat_score 可能相同，但策略應該完全不同：
- 剛爆紅：立刻追，不能等
- 持續熱：可以做深度分析，差異化切入

**情報官目前沒有辨別這兩種熱度的能力。**

解法方向：熵（superego 關鍵字顧問）的角色在這裡變得更重要——她的工作就是分析「衰減週期」，這正是她存在的原因。但目前她還沒被建起來。

---

### 盲點五：角色之間的溝通是「寫入 Firestore + 下次讀取」，不是即時溝通

這個架構有一個固有限制：所有角色之間的溝通都是非同步的，通過 Firestore 轉傳。

這在大多數情況下沒問題。

但在熱訊號旁路的場景下，閾需要知道「這篇是熱追稿，審稿標準要適當放鬆」——這個信號怎麼傳？目前的架構沒有辦法在同一次流程中傳遞這種情境信號。

解法方向：文章的 metadata 應該帶有 `pipeline_context`，例如 `{ "trigger": "hot_bypass", "heat_score": 93 }`，閾在讀到文章時也讀這個 context。

---

## 五、v2.0 施工優先序

以影響力 × 複雜度評估，執行順序如下：

```
Priority 1（本次施工）
├── 閾升級：編輯模式（PUBLISH / REVISED / REJECT 三段式）
├── 角色工作記憶：建 live_media_char_memory，閾 → 寫 → 停格者讀
└── pipeline_context：文章 metadata 帶 trigger 信號

Priority 2（下次施工）
├── 熱訊號旁路：每小時快速掃描 + heat_score 觸發
└── 弦 MVP：每週讀全局 → 輸出角色方向更新

Priority 3（等社群層接通後）
├── 讀者信號接入：社群互動數據 → 痕 → 工作記憶
└── 系統健康儀表板：後台管理頁加 dashboard tab

尚未設計（需要更多思考）
├── 主題線 / 系列文章機制
├── 熱度類型辨別（剛爆 vs 持續熱）
└── 熵超我：關鍵字衰減週期分析正式接入
```

---

## 六、一句話定錨

v1.0 是工廠。每個人站在自己的工位，把東西往前傳。

v2.0 是選舉陣營。每個人有自己的資訊，能橫向溝通，能即時反應，能從上週的錯誤學習，有一個人在看整體。

**區別不在於角色的能力，在於神經系統的有無。**
